# AI 协作驱动开发方法论 · 入门手册

> v1.3 | 方法论精简版，帮你快速上手
> 
> 本手册讲"怎么做"，方法论讲"为什么"。两者配合使用——手册用来起步，完整版用来深挖。

**关于平台**：本文所述的 Agent 能力——文件系统作为长期记忆、sub-agent 并发执行、Calendar 定时任务、Heartbeat 心跳轮询、工具调用执行实际操作——基于具备文件持久化、工具调用、子任务派发等核心能力的 Agent 平台（以 Coze/扣子 为例，同类平台还包括 Dify、AutoGPT、LangChain Agent、OpenClaw 等）。普通 AI 助手（ChatGPT 对话、LLM API 调用）不具备这些能力，它们的协作模式是"对话即服务"，无法实现本文描述的项目级长期协作。

## 1. Agent 是什么

### 1.1 Agent 本质

Agent 是**长期驻留的协作伙伴**，不是一次性对话机器人。

| 特征 | 普通AI对话 | Agent |
|------|-----------|-------|
| 记忆载体 | 对话上下文（对话结束即丢失） | 文件系统（跨对话保持） |
| 持续性 | 每次对话从零开始 | 跨对话保持上下文 |
| 主动性 | 等你问才回应 | 能自己规划任务 |
| 自主性 | 无 | 有自己的判断和偏好 |
| 执行能力 | 只能输出文字 | 可调用工具执行实际操作 |
| 知识积累 | 每次都要重新教 | 写入文件后永久记得 |

**普通AI**：你问一句它答一句，这次聊完下次就忘了。  
**Agent**：有文件系统作为长期记忆，能调用工具执行实际任务，是你的"项目搭档"。

### 1.2 Agent的核心能力来源

Agent 的能力来自三个核心来源：

| 来源 | 作用 | 说明 |
|------|------|------|
| **文件系统（记忆）** | 持久化存储 | 关键信息写入文件，下次唤醒时读取恢复。不写入文件的内容，约等于没告诉 Agent |
| **工具（交互）** | 与外部世界交互 | 调用bash、联网搜索、文件读写等工具，实际执行操作 |
| **技能（skill）** | 结构化专业知识包 | 加载特定技能后获得专项能力，如PPT制作、文档生成、图表可视化等 |

**核心关系**：文件系统是记忆层，工具是执行层，技能是扩展层。三者配合让 Agent 能持续工作、实际执行、专项深入。

### 1.3 消息处理流程

Agent 收到消息后会按顺序处理：

```
收到消息 
→ 读取SOUL.md（确认性格和行为规则）
→ 读取USER.md（了解你的偏好）
→ 读取MEMORY.md（恢复项目上下文）
→ 读取TOOLS.md（获取工作规则）
→ 判断任务类型（对话 / 执行任务）
→ 自己执行 或 派sub-agent
→ 执行任务
→ 结果验收
→ 更新记忆文件（重要结论写入MEMORY，新经验写入TOOLS）
```

**关键推论**：如果 MEMORY.md 是空的，Agent 就不知道项目进展到哪了；如果 TOOLS.md 没写规则，Agent 就按自己想法来，容易踩坑。

### 1.4 各文件对Agent行为的影响

| 文件 | 影响什么 | 示例 |
|------|---------|------|
| **SOUL.md** | 性格、称呼、行为模式 | 写了"严谨务实"→Agent回复更偏向技术细节 |
| **USER.md** | 对你偏好的理解 | 写了"喜欢简洁"→Agent回复更精炼 |
| **MEMORY.md** | 项目上下文和待办 | 写了"当前分支v3.0"→Agent不会改错分支 |
| **TOOLS.md** | 工作规则和避坑经验 | 写了"BUG修复workflow"→Agent按流程走 |
| **SECRET.md** | 敏感信息 | 有token→Agent可push；没有→需要你提供 |
| **INDEX.md** | 资源导航 | Agent通过INDEX定位skill和知识库 |

### 1.5 文件状态不好时的后果

| 问题 | 后果 |
|------|------|
| **文件过大** | 上下文被占满，早期关键信息被挤出，Agent在残缺上下文中推理 |
| **文件过时** | Agent用过期信息做决策，方案可能针对旧版本 |
| **文件冲突** | 同一信息多处记录不一致，Agent无法确定以哪个为准 |

**优化目标**：TOOLS.md ≤5KB，MEMORY.md ≤3.5KB，活跃文件合计 ≤15KB。

### 1.6 上下文管理要点

上下文是 Agent 工作的"内存条"，容量有限。

| 消耗程度 | 风险 |
|---------|------|
| 0-50% | 正常工作 |
| 50-70% | 早期信息开始被稀释，偶发遗忘 |
| 70-90% | 信息被截断，推理质量下降 |
| >90% | 严重遗忘，幻觉风险上升 |

**核心原则**：上下文窗口有限，文件系统无限。关键信息写入文件，上下文只放当前任务需要的"热数据"。

**开新对话的信号**：
- Agent 反复问同一个问题（上下文已丢失）
- 回答开始自相矛盾（上下文碎片化）
- 上下文占用 >80% 且任务未完成

### 1.7 怎么获得这样的Agent

| 平台 | 方式 | 备注 |
|------|------|------|
| **Coze/扣子高阶订阅** | 开通即用 | 最省心，文件自动创建 |
| **Dify云端部署** | 开通即用 | 灵活度高，支持自定义工具 |
| **OpenClaw自部署** | 手动搭建 | 需要运维能力，完全可控 |
| **AutoGPT/LangChain** | 自行搭建 | 需要开发能力 |

**安全提醒**：Agent 拥有 bash 执行权限。不要部署在主力电脑上，使用云电脑或虚拟机隔离环境。

## 2. 协作工作流

### 2.1 为什么需要workflow

没有 workflow 时出的问题：

| 问题 | 后果 |
|------|------|
| AI未经确认就改代码 | 破坏开发者有意保留的设计，回滚返工 |
| 改了A忘B | AI只改了问题点，没检查关联模块，其他模块崩了 |
| 同一问题反复出现 | AI每次对话都重新分析，浪费轮次 |
| AI在死胡同死磕 | 试了4种方案全失败，浪费时间，代码越改越乱 |

**结论**：Workflow 的存在是为了约束 AI 行为、减少返工、提高一致性。没有 workflow，AI 会"高效地犯错"。

### 2.2 协作全景图

| 阶段 | 目标 | AI职责 | 人的职责 |
|------|------|--------|---------|
| 想法→需求 | 模糊变清晰 | 梳理：拆解痛点为结构化需求 | 定义业务边界和约束 |
| 需求→方案 | 可行性验证 | 分析：可行性验证、方案对比 | 判断方向、做取舍 |
| 方案→落地 | 可运行代码 | 实施+验收：编码、日志分析 | 真实环境测试、提供完整信息 |
| 迭代决策 | 持续优化 | 迭代推进：瓶颈分析、方案输出 | 止损判断、确认审批 |

**核心原则**：人提方向，AI 出方案；人做测试，AI 做分析；人掌舵，AI 执行。

### 2.3 六种工作流核心步骤

#### 需求分析与调研
适用于：想法很模糊、不确定能不能做。

```
描述模糊想法 → Agent梳理 → 确认目标和边界 → Agent可行性分析 → 你确认方向 → Agent拆解任务 → 你确认拆解 → 执行
```

**关键**：模糊需求不要直接动手，先问清楚再干。

#### 新功能/需求开发
适用于：需求已经明确，需要从零开发新功能。

```
复述理解 → 给实现方案 → AI自查方案 → 你确认 → 编码实现 → 自检代码 → push → 你测试
```

**关键**：理解偏差1度，执行偏差180度。先确认需求理解再动手。

#### 信息整理与输出
适用于：搜索/采集信息、整理资料、生成报告。

```
明确目标和格式 → 执行采集 → 整理结构化 → 给中间结果确认方向 → 补充完善 → 交付 → 归档知识库
```

**关键**：先给中间结果确认方向，不要闷头整理完才发现偏了。

#### 文件生成类操作
适用于：生成文档、报告、PPT等文件。

```
确认内容和格式要求 → 给方案/大纲 → 你确认 → 生成 → 验收质量 → 交付 → 不满意则调整
```

**关键**：大纲确认5分钟，全文返工30分钟。

#### BUG修复
```
记录问题 → AI给方案 → AI自查方案 → 你确认 → 执行修改 → 自检代码 → push → 你测试
```

**关键**：所有问题凑齐→给方案→自查→确认→一次性全改→push。不要边记边改。

#### 代码重构/优化
适用于：性能优化、架构调整、代码清理。

```
明确优化目标 → 给方案（含风险评估）→ 你确认 → 实施 → 回归测试 → push → 你测试
```

**关键**：重构最怕改A坏B，方案里必须说明影响范围和测试策略。

### 2.4 铁律：方案先行

**用户给方向 ≠ 授权直接动手**。即使是"小改动"，也要：

```
给方案（改哪、怎么改、措辞是什么）→ 确认 → 再动手
```

跳过确认的口子一开就会越改越大。

### 2.5 技能全生命周期管理

技能（skill）是 Agent 能力的扩展，让它在特定领域更专业。

#### 选型原则

根据需求匹配技能，不匹配则不调用：

| 场景 | 用什么 |
|------|--------|
| 生成图表 | echart skill |
| 生成 PPT | create-ppt skill |
| 处理 PDF | pdf skill |
| 处理 Word | docx skill |
| 搜索/安装技能 | skill-assistant |

#### 标准加载流程

```
判断需要 skill → 加载 skill → 读 SKILL.md 理解规范 → 按规范执行
```

**关键**：skill_load 后必须读 SKILL.md，理解能力边界和执行规范。只读名称不读规范容易踩坑。

#### 异常处理

| 问题 | 处理方式 |
|------|---------|
| 加载后效果不好 | 检查是否按 SKILL.md 规范执行 |
| 不知道能做什么 | 读 SKILL.md 的描述和示例 |
| 需要组合多个技能 | 注意加载顺序和依赖关系 |

## 3. 执行方式

### 3.1 sub-agent（子任务派发）

**sub-agent 是什么**：
- 主Agent 派出的独立执行单元
- 不了解对话历史，只通过任务描述获取上下文
- 异步执行，完成后结果返回主Agent

**为什么要用**：
- 不阻塞对话（主Agent可以继续和你聊天）
- 并行处理（同时跑多个子任务）
- 隔离风险（子任务失败不影响主对话）

**主Agent派sub-agent的流程**：

```
判断需要异步执行 → 编写任务描述（含完整上下文）→ 派发 → 主Agent继续对话 → 收到结果后验收 → 符合则呈现给你
```

**任务描述是关键**：子Agent没有对话历史，任务描述是它唯一的上下文。必须包含：
- 项目背景
- 当前进展
- 相关文件路径
- 产出要求
- 结果发送目标

### 3.2 sub-agent派发规范

| 规范 | 做法 |
|------|------|
| **串行原则** | B依赖A的产出时，必须等A完成再派B |
| **指定源文件** | 明确写"基于哪个文件改"，不能假设"当前代码"就是最新版 |
| **递进链路** | 多级产出有递进关系时，写明链路（谁消费、什么格式、链路位置） |
| **保留清单** | 派子Agent修改时，明确列出哪些必须保留 |

### 3.3 sub-agent异常处理

| 信号 | 处理方式 |
|------|---------|
| 超时/无响应 | sessions_status查看状态，running但无进展则sessions_abort终止 |
| 结果不达标 | 用sessions_send补充要求，让子Agent继续修改，不要自己上手重做 |

**关键**：不要自己上手重做。用sessions_send补充要求，让sub-agent继续修改。

### 3.4 Calendar（日程/定时任务）

**Calendar 是什么**：Agent的定时任务工单系统，到设定时间自动触发。

| 功能 | 说明 |
|------|------|
| 一次性提醒 | "明天下午1点半提醒我开会" |
| 定期执行 | "每天下午查一下XX价格" |
| 重复追踪 | "持续追踪某行业新闻" |

**适合场景**：定时提醒、定期执行、有明确产出的检查/报告任务。

**不适合**：需要秒级响应的检查、需要生成大文件的复杂任务（用sub-agent）。

**创建注意事项**：
- summary：用清晰动作短语，主语是Agent自己
- description：写给执行session看的工单详情，必须包含完整上下文
- time_range：设置弹性调度窗口，避开整点高峰
- rrule：只填freq/interval/count/until，不要填与dtstart冲突的字段

### 3.5 Heartbeat（心跳/周期轮询）

**Heartbeat 是什么**：每30分钟触发一次的轻量检查机制。

**工作方式**：每次触发读取heartbeat.md中的检查清单，逐项快速检查，有事通知无事沉默。

**适合场景**：
- 检查后台任务/sub-agent是否完成或卡住
- 回复其他渠道消息
- 时间要求不严格的提醒

**heartbeat.md格式**：
```
检查清单
- 检查XX任务是否正常运行，卡住则通知用户
- 看一眼XX群是否有需要回复的消息
- 用户久坐超过2小时，提醒活动一下
```

> 首行必须是"检查清单"纯文本标题，之后用 `- ` 列表。

### 3.6 三种执行方式选择指南

| 场景 | 选什么 | 原因 |
|------|--------|------|
| 现在就要做一件事 | sub-agent | 即时执行，结果秒回 |
| 每天固定时间做某事 | Calendar | 定时触发，重复执行 |
| 定期追踪某个话题 | Calendar | 有明确产出 |
| 30分钟看一眼状态 | Heartbeat | 轻量检查，无事不扰 |
| 生成文件/报告 | Calendar或sub-agent | Heartbeat不适合 |
| 久坐提醒 | Heartbeat | 30分钟粒度够 |

## 4. 沟通与确认

### 4.1 沟通规范

**有效信息结构**：现象 + 触发条件 + 与预期的差异

| 低效表达 | 有效表达 |
|---------|---------|
| "程序崩了" | "开启VPN后闪退，关闭VPN正常运行" |
| "时间不准" | "配置-0.1s偏移，日志显示实际-0.3s" |
| "按钮点不动" | "点击后无响应，但日志显示click事件已触发" |

信息越精确，AI定位越快。模糊描述 = AI猜测 = 方向可能偏 = 浪费轮次。

**描述越详细，定位越快**：
- ✅ "点击购买按钮后页面没跳转，控制台报错 `Uncaught TypeError: Cannot read property 'sku' of null`"
- ❌ "购买功能坏了"

### 4.2 确认机制

**结构性改动必须先说明方案、等待确认、再实施。**

起源案例：AI看了代码觉得某处可以优化，没问人就改了，结果破坏了开发者的设计意图，回滚返工浪费大量时间。

**必须确认的场景**：
- 修改代码（任何涉及文件内容变更的操作）
- 改架构或配置项
- 删除文件或模块
- 改分支或版本号

**可以先做的场景**：
- 纯解释性的问题（"为什么..."）
- 纯分析性的问题（"帮我看看..."）
- 确认一个再改一个（但要等确认完再执行）

### 4.3 止损机制

AI倾向于持续尝试，不具备止损能力。设定试错上限（3-4次），超限则换方向或回滚。

**何时继续**：
- 问题性质明确、方案方向正确
- 只是还没试到对的参数
- 证据显示方向是对的

**何时止损**：
- 多次不同方案均失败
- 问题可能在更底层（如系统级网络劫持）
- 继续投入的边际收益急剧下降

**止损案例**：VPN闪退问题试了4种方案全失败，最终识别为系统级网络劫持，回滚并标注环境限制。

### 4.4 Workflow遗忘防范

AI在正确方向上跳步的问题——该确认的不确认、该停的不停、该问的不问。

**遗忘的表现模式**：

| 模式 | 表现 | 后果 |
|------|------|------|
| 跳过确认直接执行 | 看了代码觉得可以改，直接改了push | 改掉你有意保留的设计 |
| 看到代码就忍不住改 | 你只是问"为什么"，AI就开始读代码准备改 | 未经授权的修改 |
| 给方案但不等确认就动手 | 方案刚给出来，AI就开始执行 | 方案有问题时已经改完 |

**制止提示词**：
```
停。你又在跳步了。
→ 问你解释的时候，不要开始改代码
→ 给方案的时候，不要动手执行
→ 等我确认了再动
```

**即时处理流程**：
1. 立即停止当前操作
2. 回退到跳步前的位置
3. 补上缺失的环节（先给方案→自查方案→等确认→再执行）
4. 确认方案后再继续

**预防性措施**：
- 重要规则写入TOOLS.md铁律区
- 执行前自检清单
- heartbeat定期自检

## 5. 验收双向检查

### 5.1 复盘要点

**复盘时机**：每个项目/版本迭代完成后进行一次，不需要等到"完美时机"。

**复盘模板**：

```
## 1. 项目背景与目标
- 项目名称：
- 核心目标：
- 时间跨度：

## 2. 实际交付结果
- 原计划交付：
- 实际交付：
- 完成度评估：

## 3. 踩坑记录
| 问题 | 根因 | 解决方案 | 教训 |

## 4. 可复用经验
- 哪些做法可以标准化？
- 哪些流程值得固化到workflow？
- 哪些教训值得写入harness？

## 5. 待改进点
```

**复盘产出**：
- 教训 → 写入TOOLS.md的harness
- 经验 → 固化为workflow
- 踩坑记录 → 归档知识库

### 5.2 验收8条标准

验收不是"看起来对就行"，是**双向检查**：

**正向验证（需求→产出）：确认"该做的做了没"**

| 检查项 | 怎么做 |
|--------|--------|
| 1. 覆盖完整性 | 逐条对照原始需求，每条是否有对应产出 |
| 2. 意图准确性 | 产出是否真正满足意图，还是字面糊弄 |
| 3. 产出可用性 | 产出能不能实际工作，不只是"看起来有了" |

**反向挑刺（产出→问题）：找"不该出的问题出了没"**

| 检查项 | 怎么做 |
|--------|--------|
| 4. 逻辑审计 | 因果是否成立？概念是否矛盾？上下文是否衔接？ |
| 5. 边界完整性 | 序号连续？引用有效？命名一致？ |
| 6. 副作用扫描 | 改A是否意外影响B？ |
| 7. 合适性判断 | 这么做是否合适？有没有更好的方式？ |
| 8. 交叉验证 | 用grep/脚本扫描，不靠肉眼 |

### 5.3 交付引用

**验收通过后必须立即引用给你看，无例外。**

```
验收通过 → 立即引用 → 等待你确认 → 完成
```

不要等你来问，改完就引。

### 5.4 自查方案（给确认之前）

| 检查项 | 怎么做 |
|--------|--------|
| 需求覆盖 | 需求每条都覆盖了吗？ |
| 关联影响 | 改了A，B要不要跟着改？ |
| 副作用 | 有没有副作用？ |
| 隐含假设 | 隐含假设什么时候不成立？ |

### 5.5 自检代码（push之前）

| 检查项 | 怎么做 |
|--------|--------|
| 语法/import | 语法是否正确，import是否完整？ |
| 代码一致性 | 代码和方案一致吗？ |
| 异常处理 | 崩溃点在哪？异常路径怎么办？ |
| 数据流追踪 | 变量从哪来？变了没？ |

### 5.6 文件修改安全规范

**核心原则：副本先行，逐级验收。**

#### 副本先行

修改任何文件时，先复制副本在副本上改。原文件是最基本的安全保障。

```
cp 原文件 原文件.备份  # 创建副本
# 在副本上修改
# 验收通过后替换原文件
```

#### 逐级验收机制

| 阶段 | 检查什么 | 通过标准 |
|------|---------|---------|
| 1. 产出 | 按任务要求完成修改 | 自检通过 |
| 2. 验收 | 内容完整性、引用正确性 | 脚本验证通过 |
| 3. 确认 | 修改符合意图 | 你明确确认 |
| 4. 替换 | 三步全部完成 | 替换原文件 |

#### 防幻觉校验

文件修改后必须校验：
- 引用路径是否存在
- 关键内容是否完整保留

#### 安全检查清单

```
□ 我做了副本吗？
□ 我在副本上改的吗？
□ 引用路径都存在吗？
□ 关键内容没有丢失吗？
□ 验收通过了吗？
□ 你确认了吗？
□ 三步全部通过才替换原文件
```

## 6. 记忆与知识库

### 6.1 管理原则

**核心原则**：活跃文件保持精简，过时内容归档到知识库。

| 原则 | 说明 |
|------|------|
| **文件即记忆** | 关键信息必须写入文件，不能只存在于对话中 |
| **INDEX即入口** | Agent通过INDEX导航所有资源，不全量加载 |
| **按需加载** | 通过索引定位，只加载当前任务需要的文件 |
| **定期归档** | 过时内容从活跃文件移到知识库 |
| **单一入口** | 同一信息只在一个地方定义 |
| **同步维护** | 修改文件后必须更新INDEX |

### 6.2 记忆文件职责划分

| 文件 | 职责 | 写什么 | 不写什么 |
|------|------|--------|---------|
| **SOUL.md** | Agent人设与自我认知 | 性格设定、行为规则 | 项目信息 |
| **TOOLS.md** | 工具使用经验 | 当前有效的harness、workflow | 已解决的问题、过期信息 |
| **MEMORY.md** | 项目与事务记忆 | 活跃项目状态、待办事项 | 历史归档、详细技术文档 |
| **USER.md** | 用户个人信息 | 偏好、背景、习惯 | 项目信息 |
| **知识库/** | 归档与参考资料 | 历史归档、skill参考、项目索引 | 活跃项目状态 |
| **INDEX.md** | 导航入口 | 资源路径与摘要 | 具体内容 |

### 6.3 INDEX导航体系

INDEX.md是Agent访问所有资源的唯一入口。

**INDEX结构示例**：

**Agent执行路径**：
```
你提问 → Agent读取INDEX → 定位相关分类 → 读取具体文件 → 加载对应skill → 执行任务
```

### 6.5 长期项目跨session追踪

MEMORY.md的长期项目结构：

| 字段 | 内容 | 示例 |
|------|------|------|
| 项目阶段 | 当前在哪个阶段 | V3开发中，第二轮优化 |

### 6.6 知识库维护

**每月一次INDEX完整性检查**：

| 检查项 | 含义 | 后果 |
|-------|------|------|
| INDEX引用的文件是否存在 | 检查死链 | Agent找不到文件 |
| 实际文件是否在INDEX中 | 检查孤儿文件 | 新增内容无法被定位 |

**TOOLS.md章节顺序 = 权重顺序**：
- 铁律区（最顶部）：合规自检、方案先行、产出交付（每次必过）
- 流程区（中间）：BUG修复workflow、需求开发workflow
- 参考区（底部）：工具选型、资源定位、验收标准

**为什么权重重要**：上下文被压缩时，Agent优先保留前面的内容。核心原则放铁律区确保每次必过；放附录区可能被截断导致行为偏离。

**递进链路同步规则**：配置文件改了→方法论对应章节要同步→入门手册要同步。

| 场景 | 需要同步吗 |
|------|-----------|
| TOOLS.md改了一条harness | 是，方法论和入门手册对应章节都要改 |
| 完整版方法论改了 | 是，入门手册的对应章节要改 |
| 派发规范从5条变8条 | 是，方法论和入门手册都要更新 |

**配置修改标准工作流**：
```
1. 明确改什么、为什么改
2. 给方案 → 检查权重位置（改的内容放铁律区/流程区/参考区？）
3. 确认后修改
4. 同步更新下游文档（完整版→入门手册）
```

**维护检查清单**：
```
□ 我检查了改的内容应该放在哪个区吗？
□ 我检查了新位置是否影响其他条目的优先级吗？
□ 我同步更新了完整版方法论的相关章节吗？
□ 我同步更新了入门手册的相关章节吗？
□ 递进链路的所有文档是否一致？
```

## 7. 项目迭代

### 7.1 迭代逻辑

```
V1验证可行性 → V2识别瓶颈局部替换 → V3全链路优化
```

每次迭代的驱动力是**明确的性能瓶颈**，不是预设的完美方案。优化方向来自你对业务的理解，AI负责验证可行性并给出实现路径。

**核心原则**：
- 不要追求一开始就设计完美的架构
- 每一步都有价值，先跑起来比什么都重要
- 瓶颈驱动优化，不是理论驱动优化

| 版本 | 目标 | 驱动 | 关键决策 |
|------|------|------|---------|
| **V1** | 验证可行性 | 先跑起来 | 端到端流程跑通 |
| **V2** | 识别瓶颈 | 瓶颈驱动 | 局部替换优化 |
| **V3** | 全链路优化 | 架构升级 | 全面改进 |

### 7.2 各阶段协作模式

| 阶段 | 你做什么 | AI做什么 | 关键原则 |
|------|---------|---------|---------|
| V1 | 提出痛点 | 帮你分析可行性、出方案 | 先跑起来比什么都重要 |
| V2 | 反馈哪个环节慢 | 识别瓶颈、分析根因 | 瓶颈驱动优化 |
| V3 | 定义优化目标 | 全链路分析、架构设计 | 全面改进要有充分理由 |

### 7.3 迭代节奏

**不要追求完美方案再动手**：
- V1目标：跑通流程，验证技术可行性
- V2目标：识别瓶颈，针对性优化
- V3目标：架构升级，全面改进

**什么时候该迭代**：
- 当前版本有明显瓶颈
- 业务需求发生变化
- 技术栈需要升级
- 维护成本过高需要重构

**什么时候该止损**：
- 多次尝试均无法解决的核心问题
- 环境限制导致无法继续
- 投入产出比不划算

### 8.4 Force Push后本地同步

Force push后，本地`git pull`会报错（历史不一致）。

正确做法：
```powershell

git reset --hard origin/分支名
```

**强制同步**：不管本地有什么，强制和远程保持一致。Force push后必须这样做。

### 8.6 成本管理

**token消耗来源**：

| 操作类型 | 消耗量 | 优化空间 |

### 附录A：Git版本管理意识

Git 是协作开发的"时间机器"，跟平台无关，是开发基础设施。

#### 为什么要版本管理

没有版本管理时：
- 改错了不知道在哪一步错的
- 多人协作时不知道谁改了什么
- 想回退到之前的版本无从下手

有版本管理后：
- 每次提交是一个"存档点"，随时可回退
- 改动记录清晰可见，谁改了什么都记录在案
- 分支让你同时维护多个版本，互不干扰

#### 什么时候该commit/push

| 场景 | 建议 |
|------|------|
| 完成了一个小功能 | 立即commit + push |
| 修复了一个bug | 立即commit + push |
| 写了一部分还没测试 | 先不push，等测试完再push |
| 要尝试一个不确定的改动 | 开新分支，改完再merge |
| 要重构代码 | 先commit当前版本，再开始改 |

**commit原则**：每完成一个"有意义的变化"就commit。commit message 要说明改了什么。

#### 版本回退思维

当出现问题时，版本管理让你可以"后悔"：

| 情况 | 命令 | 说明 |
|------|------|------|
| 想撤销未提交的改动 | `git checkout -- 文件` | 恢复到上次commit的状态 |
| 想撤销已提交的改动 | `git revert HEAD` | 创建新commit来撤销旧commit |
| 想回到某个历史版本 | `git reset --hard 版本号` | 彻底回退到指定版本（慎用） |

**回退前必做**：确认要回退到哪个版本，回退后会丢失什么。commit过的内容不会真正消失，除非force push覆盖。

#### 你描述意图，Agent选命令

你不需要记住git命令，只需要说清楚想做什么：

```
"帮我把改动推到v3.0.4分支"
"我想建个新仓库叫my-project"
"看看最近改了什么"
"回退到上一个版本"
```

Agent 自动帮你执行正确的命令。

## 9. 避坑速查

### 9.1 文件与上下文

| 坑 | 根因 | 一句话方案 |
|----|------|-----------|
| 对话太长AI遗忘关键信息 | 上下文窗口有限，早期信息被截断 | 关键信息及时写入文件 |
| 长期项目换对话后状态丢失 | 只在对话中讨论没写文件 | 关键状态写入MEMORY.md |
| token消耗过快 | 记忆文件过大+频繁全量读取 | 精简文件+分批读取+按需搜索 |
| Agent重复搜索同一信息 | 没把搜索结果写入记忆 | 搜索结果写入文件 |

### 9.2 AI行为问题

| 坑 | 根因 | 一句话方案 |
|----|------|-----------|
| AI未经确认就改代码 | 缺少确认机制 | 结构性改动必须审批 |
| AI跳步执行 | 把"解释"等同于"修复" | 停→回退→补环节→确认→继续 |
| AI反复犯同一错误 | harness写了但执行前不自检 | 记忆锚点+自检清单+违反强制回滚 |
| sub-agent结果质量不达标 | 上下文缺失或任务描述不完整 | sessions_send补充要求，不亲自上手重做 |
| 技能加载了但没理解就执行 | 只读了技能名没读SKILL.md | skill_load后必须读SKILL.md理解能力 |

### 9.3 幻觉防范

| 坑 | 根因 | 一句话方案 |
|----|------|-----------|
| 静态分析推断功能废弃 | 代码中没找到≠不存在 | 必须用实际运行验证 |
| 不完整阅读就下结论 | 只读部分代码就下全局结论 | 先追踪代码路径，只从主函数搬 |
| 理解后重写引入偏差 | 没搞清参考代码实际走哪条路径 | 有参考代码直接复制适配，不"理解后重写" |
| 复合幻觉多错误叠加 | 多个错误互相"印证" | 先回滚，从正确起点出发 |

**幻觉检测方法**：

| 方法 | 核心思路 | 怎么做 |
|------|---------|--------|
| 交叉验证 | AI结论 vs 已有代码的实际行为 | 取实际行为为准 |
| 反向验证 | AI说"X已废弃" | 用X跑一下，能跑=没废弃 |
| 路径追踪 | AI说"参考代码就是这样" | 追问实际走哪个函数，从入口追踪 |

### 9.4 代码与调试

| 坑 | 根因 | 一句话方案 |
|----|------|-----------|
| 改源码不生效 | 编译缓存优先级高于源码文件 | 开发阶段不使用缓存，删.pyd和__pycache__ |
| 配置值与预期不符 | 多处定义同一参数，叠加生效 | 核心参数单一入口 |
| 打包后缺模块 | 打包工具扫不到懒加载模块 | 显式声明hidden-import |
| 改一个模块忘改关联 | 缺少全局搜索 | 让Agent grep全局检查引用 |
| force push后本地不同步 | git pull报错历史不一致 | git fetch + git reset --hard |

### 9.5 环境与网络

| 坑 | 根因 | 一句话方案 |
|----|------|-----------|
| VPN导致崩溃 | 系统级网络劫持，应用层无法绕过 | 标注环境限制，非代码问题 |
| Agent无法访问目标网站 | 云电脑网络限制 | 改用agent-browser |
| 本地正常/线上异常 | 环境差异 | 检查Python版本、环境变量、依赖版本 |

### 9.6 协作效率

| 坑 | 根因 | 一句话方案 |
|----|------|-----------|
| 派子agent任务太复杂卡死 | 任务描述不完整 | 拆分任务+写清楚文件路径 |
| 子agent规范不完整导致幻觉 | 没写保留清单 | 写保留清单+指定源文件+写明链路 |
| 同一问题反复出现 | 没把解决方案写入harness | 教训写入TOOLS.md，新经验固化为workflow |

### 9.7 常见排查方向

| 问题类型 | 排查方向 |
|---------|---------|
| 改了代码但不生效 | 检查编译缓存、CDN缓存、浏览器缓存 |
| 设了配置但不生效 | 检查多处配置是否叠加生效 |
| git pull/push报错 | 检查本地远程历史是否一致 |
| 开了代理就异常 | 检查系统级网络劫持 |

**通用排查原则**：
1. 环境隔离：开发/生产环境分离
2. 单一入口：核心参数只在一处定义
3. 充分验证：修改后必须测试，不要假设
4. 止损判断：识别无解问题后及时止损

## 10. 如何开始

### 第一步：选择一个你想做的项目

哪怕很模糊也没关系：
- "我想做一个自动抢票工具"
- "我想做个个人网站"
- "我想把某个重复工作自动化"

**重要的是先有一个起点**，而不是一开始就追求完美的方案。先跑起来，边做边调整。

### 第二步：按"想法→需求"阶段模式描述你的场景

用这套结构告诉Agent：

```
我遇到一个业务场景：
- 什么类型的网站/系统
- 用户操作流程是什么
- 核心痛点是什么
- 有哪些已知约束（时间、环境、技术栈）

请帮我分析：
1. 技术可行性
2. 主要风险点
3. 建议的技术方向
```

让Agent帮你梳理需求，而不是你自己先想清楚所有细节。AI的作用就是帮你把模糊变清晰。

### 第三步：建立基础文件结构

```
你的项目/
├── SOUL.md        # Agent的性格和行为规则
├── USER.md        # 你的偏好和背景
├── MEMORY.md      # 项目当前状态和上下文
├── TOOLS.md       # 工作规则和避坑经验
└── SECRET.md      # 敏感信息（token、密码等）
```

一开始不用写得很完整。核心原则：**Agent告诉你的重要结论，写进文件**。

**每个文件写什么**：
- SOUL.md：Agent该怎么称呼你、说话什么风格
- USER.md：你的偏好（暗色主题、喜欢简洁、深夜不打扰等）
- MEMORY.md：项目做到哪了、在做什么、下一步计划
- TOOLS.md：之前踩过的坑、workflow规则
- SECRET.md：token、密码等敏感信息

### 第四步：跑起来再说，先V1后V2

不要追求一开始就设计完美的架构：
- **V1**：目标是验证可行性，跑通流程
- **V2**：识别瓶颈，针对性优化
- **V3**：全面改进

**每一步都有价值，先跑起来比什么都重要。** V1跑通了，V2才知道瓶颈在哪，V3才有优化的基础。

### 第五步：超过15KB就建知识库

如果TOOLS.md或MEMORY.md越来越臃肿，执行优化：

1. 创建`知识库/`目录，按主题分类
2. 把历史内容、详细技术文档迁移到知识库
3. 创建INDEX.md作为导航入口
4. 活跃文件只保留当前有效的信息

**优化目标**：TOOLS.md ≤5KB，MEMORY.md ≤3.5KB，活跃文件合计≤15KB。

### 第六步：遇到问题查这本手册

手册是速查指南，帮你快速定位问题：

| 问题 | 看哪章 |
|------|--------|
| 不确定怎么跟Agent配合 | 第2章 协作工作流 |
| 不确定用什么执行方式 | 第3章 执行方式 |
| 不知道该不该确认 | 第4章 沟通与确认 |
| 不知道产出对不对 | 第5章 验收双向检查 |
| 文件太臃肿 | 第6章 记忆与知识库 |
| 项目怎么迭代 | 第7章 项目迭代 |
| Git/调试怎么操作 | 第8章 开发基础设施 |
| 踩坑了 | 第9章 避坑速查 |

### 常见问题

**Q：Agent会忘记我之前说的话吗？**  
A：会。关键信息必须写入MEMORY.md，只在对话里说过约等于没说。

**Q：为什么要先给方案再动手？**  
A：没有确认就改代码，Agent可能改掉你有意保留的设计，回滚返工成本很高。

**Q：什么时候用sub-agent什么时候用Calendar？**  
A：现在就要做 → sub-agent；到点/重复做 → Calendar；顺手看一眼 → Heartbeat。

**Q：Agent改错了怎么办？**  
A：先回滚（git reset），再从正确起点出发，不要在错误代码上修修补补。

**Q：知识库什么时候建？**  
A：MEMORY.md超过15KB，或Agent频繁遗忘/混淆历史决策时。

**Q：遇到幻觉怎么办？**  
A：发现AI给出可疑结论，先暂停，用反向验证法测试（用AI说的"已废弃"方式跑一下），能跑通说明AI结论错了。

## 附录：TOOLS.md参考范例

> 以下内容必须与实际 `./基础设定/TOOLS.md` 完全一致

```markdown
# 工具使用技巧与工作流

## 一、铁律区（每次操作必过，按顺序检查）

### 1. 合规自检
□ 我给出方案了吗？
□ 我自查方案了吗？（遗漏？矛盾？副作用？）
□ 用户确认了吗？
□ 我是在确认之后才动手的吗？
□ 用户问的是"解释/原因"吗？如果是，我不应该动手改
□ 我是否在"顺便"做用户没要求的事？如果是，停下来

### 2. 方案先行
- **用户给方向≠授权直接动手**。"方向明确"只是需求清楚了，不代表跳过确认环节
- 即使是"小改动"，也要先给方案（改哪、怎么改、措辞是什么）→ 确认 → 再动手
- 跳过确认的口子一开就会越改越大

### 3. 验收+交付
- 验收时双向检查：正向（该做的做了没）+反向（不该出的问题出了没），详细清单见第七区
- **每一次** edit_file/write_file 成功后，下一个动作就是用 computer:// 协议引用给用户看，无例外
- 不要等用户问，改完就引

### 4. 副本+逐级验收
- **所有修改走副本**：先cp原文件→副本，在副本上改，逐级验收后再替换原文件
- 流程：cp原文件→副本 → 改副本 → 主agent验收 → 用户验收 → 确认后替换原文件
- 子agent产出 → 主agent验收 → 不通过则返工，通过则交给用户
- 主agent产出 → 用户验收 → 不通过则修改，通过则上线
- 例外：简单局部修改（<5行）且精确指定了old_string/new_string的，可直接改

### 5. 不要自我创造
- 已有能跑的代码直接搬，不要"理解后重写"——重写=引入新bug
- 严格参考现有代码，不能凭想象编造

## 二、子agent规范（派发时必读）

### 防幻觉+派发规范
1. **保留清单**：明确列出哪些内容必须保留在原位，不能删、不能迁移
2. **迁移约束**：涉及内容迁移时，必须写明目标文件路径、迁移后源文件处理方式，必须先确认目标文件创建成功才能删源内容
3. **验收检查项**：任务描述末尾列出主agent验收时会检查什么，让子agent知道会被检查
4. **禁止自创文件路径**：子agent不能自己编路径然后引用，所有引用路径必须在任务描述中指定或已确认存在
5. **串行原则**：B的输入依赖A的产出时，必须等A完成再派B，不能并行
6. **指定源文件**：派发时明确写"基于哪个文件改"，不能假设原文件就是最新版（可能有未覆盖的副本）
7. **递进链路**：多级产出有递进关系时（如配置→展开→精简），在任务描述中写明链路，子agent才知道自己的产出会被谁消费

## 三、场景工作流（用到才查）

### BUG修复
1. 用户反馈问题 → 只记录，不改
2. 所有问题凑齐 → 总结+给方案 → **自查方案** → 交给用户确认
3. 确认 → 一次性全改 → 自检代码 → push
4. 用户测试 → 有问题回到第1步
- 严禁边记边改、确认一个改一个push一个
- 不要给pull步骤，用户fetch+reset即可
- force push后必须提醒 `git fetch origin && git reset --hard origin/分支名`
- **⚠️ 所有写/改/删文件操作都必须等用户确认后才能执行**
- **⚠️ 用户问"为什么"时只解释，不要动手改代码！**

### 新功能/需求开发
1. 用户提需求 → 复述理解，确认没偏
2. 给实现方案 → **自查方案** → 交给用户确认
3. 确认 → 建项目目录+INDEX.md → 写代码 → 自检代码 → push
4. 用户测试

### 需求分析与调研
1. 用户提模糊需求/想法 → 先梳理，不要急于给方案
2. 明确三件事：目标是什么、边界在哪、约束条件
3. 给可行性分析 → 交给用户确认方向
4. 方向确认后 → 拆解为可执行任务 → 用户确认拆解 → 派sub-agent执行

**关键**：模糊需求不要直接动手，先问清楚再干。

### 信息整理与输出
1. 明确搜索/采集目标和产出格式
2. 执行搜索/采集 → 整理为结构化内容
3. 给用户看初步结果 → 确认方向
4. 补充完善 → 最终交付 → 重要结论写入知识库

**关键**：先给中间结果确认方向，不要闷头整理完才发现偏了。

### 文件生成类操作
1. 确认内容和格式要求
2. 给方案 → 确认 → 生成 → 验收质量 → 交付
3. 不满意 → 调整 → 重新交付

### 代码重构/优化
1. 用户提优化需求 → 明确优化目标
2. 给方案，**必须包含风险评估**
3. 用户确认 → 实施 → 自检代码 → 回归测试 → push
4. 用户测试

**关键**：重构最怕改A坏B，方案里必须说明影响范围和测试策略。

## 七、验收标准（交付前必查）

### 自查方案（给用户确认之前）
1. 需求每条都覆盖了吗？
2. 改了A，B要不要跟着改？
3. 有没有副作用？
4. 隐含假设什么时候不成立？

### 自检代码（push之前）
1. 语法/import/功能是否正确？
2. 代码和方案一致吗？
3. 崩溃点在哪？异常路径怎么办？
4. 数据流追踪：变量从哪来？变了没？

### 验收产出（检查任何交付物）
**正向验证（需求→产出）**：确认"该做的做了没"
1. **覆盖完整性** — 逐条对照原始需求，每条是否有对应产出？
2. **意图准确性** — 产出是否真正满足了需求意图，还是字面糊弄？
3. **产出可用性** — 产出能不能实际工作？

**反向挑刺（产出→问题）**：找"不该出的问题出了没"
4. **逻辑审计** — 因果是否成立？概念是否矛盾？上下文是否衔接？
5. **边界完整性** — 序号连续？引用有效？命名一致？
6. **副作用扫描** — 改A是否意外影响B？删一段是否连带信息丢失？
7. **合适性判断** — 这么做是否合适？有没有更好的方式？
8. **交叉验证** — 用grep/脚本扫描，不靠肉眼

**交付引用** — 验收通过后必须立即用 computer:// 协议引用给用户，无例外
```
