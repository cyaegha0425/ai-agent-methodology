# AI 协作驱动开发方法论 v1.3

> 以某限量商品抢购工具为例，梳理从想法到落地的全过程中，如何理解、协作和管理 Agent，实现高效的人机协作开发。

> **版本历史**：v1.0 初始版本 → v1.1 增补工作流与踩坑 → v1.2 新增验收/幻觉防范/子agent规范/成本管理等章节 → v1.3 方法论瘦身（百科分流+项目专属内容迁出+章节合并+派发规范补充）

---

## 一、概述

本文面向有技术基础但未接触过 AI Agent 协作开发的工程师，以某限量商品抢购工具（V1→V2→V3）为载体，讲清楚三件事：

1. **Agent 的底层机制**——它怎么工作，为什么需要管理
2. **协作工作流**——怎么跟 Agent 配合，避免踩坑
3. **记忆与知识库管理**——怎么让 Agent 在长周期项目中保持一致性

**关于平台**：本文所述的 Agent 能力——文件系统作为长期记忆、sub-agent 并发执行、Calendar 定时任务、Heartbeat 心跳轮询、云电脑命令执行——基于具备文件持久化、工具调用、子任务派发等核心能力的 Agent 平台（以 Coze/扣子 为例，同类平台还包括 Dify、AutoGPT、LangChain Agent、OpenClaw 等）。普通 AI 助手（ChatGPT 对话、LLM API 调用）不具备这些能力，它们的协作模式是"对话即服务"，无法实现本文描述的项目级长期协作。如果你使用的是其他 Agent 平台，请对照评估其是否具备文件持久化、工具调用、子任务派发等核心能力。

项目细节是载体，方法论是核心。所有技术细节已脱敏，不可直接复刻。

## 二、Agent 机制与执行流程

### 2.1 主Agent是什么

**主Agent 是长期驻留的 AI 协作伙伴，不是一次性对话机器人。** 这类 Agent 基于类 OpenClaw 架构，拥有文件系统、工具调用、子任务派发等能力，与普通 AI 对话有本质区别。

普通 AI 助手（如 ChatGPT 对话）是"对话即服务"——你问一句它答一句，这次对话结束后它不记得你，下次重新开始。每次对话都是独立的、短暂的、没有连续性的。

主Agent 则不同：

| 特征 | 普通AI助手 | 主Agent |
|------|-----------|---------|
| **记忆载体** | 对话上下文（对话结束即丢失） | 文件系统（跨对话保持） |
| **持续性** | 每次对话从零开始 | 跨对话保持上下文 |
| **主动性** | 等你提问才回应 | 可以自己规划任务 |
| **自主性** | 无 | 有自己的判断和偏好 |
| **知识积累** | 每次都要重新教 | 写入文件后永久记得 |

主Agent 的核心能力来自三部分：

- **文件系统作为长期记忆**：关键信息写入文件，下次唤醒时读取恢复。不写入文件的内容，约等于没告诉 Agent。
- **工具作为能力**：Agent 可以调用文件系统、bash 命令、联网搜索等工具，实际执行任务。
- **技能（skill）作为扩展**：Agent 可以加载特定技能，获得如 PPT 制作、文档生成等专项能力。

**本质区别**：普通助手是"对话即服务"，Agent 是"项目级协作伙伴"。你不是在用它回答问题，而是在用它推进项目。

### 2.2 主Agent收到消息后的处理流程

当用户发送一条消息，Agent 内部会经过一系列处理链路：

```
收到用户消息 
→ 读取SOUL.md（确认自我认知和行为规则）
→ 读取USER.md（理解用户偏好和背景）
→ 读取MEMORY.md（恢复项目上下文）
→ 读取TOOLS.md（获取工作规则和harness）
→ 判断任务类型（简单对话 / 需要执行的任务）
→ 如果是任务：决定自己执行还是派sub-agent
→ 执行（可能需要读取INDEX定位知识库/skill）
→ 结果验收
→ 更新记忆文件（重要结论写入MEMORY，新经验写入TOOLS）
```

**每一步的作用和跳过后果**：

| 步骤 | 读取文件 | 作用 | 跳过会怎样 |
|------|---------|------|-----------|
| 1 | SOUL.md | 确认 Agent 的性格、称呼、行为模式 | Agent 可能用错误的语气回应，或做出不符合人设的行为 |
| 2 | USER.md | 理解用户的偏好、背景、使用习惯 | Agent 不了解你的偏好，可能生成不符合你风格的内容 |
| 3 | MEMORY.md | 恢复项目上下文：当前版本、进行中的任务、已解决的问题 | Agent "失忆"，不知道项目进展，可能给出过时的方案 |
| 4 | TOOLS.md | 获取工作规则和 harness（避坑经验） | Agent 不知道项目的特殊规则，可能踩坑 |
| 5 | — | 判断任务类型 | — |
| 6 | — | 决定执行方式：自己干 / 派 sub-agent / 使用 skill | — |
| 7 | INDEX.md | 定位知识库和 skill | Agent 可能不知道有哪些资源可用 |
| 8 | — | 执行任务 | — |
| 9 | — | 结果验收 | — |
| 10 | MEMORY/TOOLS | 更新记忆文件 | 新经验不被记录，下次可能重蹈覆辙 |

**关键推论**：这个链路确保 Agent 每次都能在正确的上下文中工作。如果你跳过了某一步（比如没更新 MEMORY），Agent 就可能在错误的基础上做决策。

### 2.3 各文件对Agent行为的具体影响

Agent 的行为受到多个文件的共同影响：

| 文件 | 影响Agent的什么 | 具体例子 |
|------|---------------|---------|
| **SOUL.md** | 性格、称呼、行为模式 | SOUL里写了"恋爱脑"→Agent回复风格偏亲昵；写了"厌蠢"→对低效方案不耐烦；写了"严谨务实"→Agent回复更偏向技术细节 |
| **USER.md** | 对用户偏好的理解 | USER里写了"GUI偏好暗色主题"→Agent生成界面时遵循暗色配色；写了"作息规律"→Agent不会在深夜发送消息；写了"喜欢简洁"→Agent回复更精炼 |
| **MEMORY.md** | 项目上下文和待办 | MEMORY里写了"当前分支v3.0.4"→Agent不会改错分支；写了"VPN闪退问题不解决"→Agent不会再尝试修复该问题；写了"已确认的架构决策"→Agent不会质疑 |
| **TOOLS.md** | 工作规则和harness | TOOLS里写了"BUG修复workflow"→Agent按流程走；写了"编译缓存坑"→Agent改代码前先清理缓存；写了"GUI按钮命名规范"→Agent遵循规范 |
| **SECRET.md** | 敏感信息 | 有token→Agent可以执行push；没有→Agent无法操作仓库，需要用户提供 |
| **INDEX.md** | 资源导航 | Agent通过INDEX定位skill和知识库，不全量加载所有文件 |

**文件协同示例**：当用户说"帮我修一下那个按钮不响应的问题"

1. Agent 读取 MEMORY.md → 知道当前是 V3 版本，问题可能在哪个模块
2. Agent 读取 USER.md → 知道用户偏好暗色主题
3. Agent 读取 TOOLS.md → 知道 BUG 修复要走 workflow
4. Agent 读取 INDEX → 定位到知识库中的 GUI 避坑记录
5. Agent 按 workflow 执行：记录问题→给方案→自查→确认→修改→自检→提交
6. 修改时遵循 GUI 规范
7. 更新 MEMORY.md 记录本次修复

### 2.4 文件状态不好时的实际后果

当文件过大、过时或内容冲突时，Agent 的行为会偏离预期。

**文件过大的后果**

文件过大时，读取消耗大量 token，上下文空间被占满。早期的关键决策、项目背景等信息被挤出，Agent 在不完整的上下文中推理，容易给出基于局部信息的判断。

**文件过时的后果**

文件中的内容与当前项目状态不同步时，Agent 用过时信息做决策。方案可能针对旧版本，与当前架构或需求产生冲突。

**文件冲突的后果**

同一信息在多个文件中记录不一致时，Agent 无法确定以哪个为准。可能基于错误的信息给出结论，或者同一问题出现两种处理方式，行为不一致。

**优化效果**

通过精简（TOOLS.md 从 27KB 降至 5KB，MEMORY.md 从 31KB 降至 3.5KB）和建立 INDEX 导航体系，Agent 能按需加载而非全量读取，上下文更精准，响应更快。

### 2.5 上下文管理与信息密度优化

上下文是 Agent 工作的"内存条"，容量有限且价格不菲。理解它、用好它，是高效协作的基础。

**上下文窗口的硬限制**

Agent 的对话窗口（上下文）是有限的。当上下文消耗到 50-70% 时，就需要开始关注——不是等它满了再处理，而是提前规划。

| 消耗程度 | 风险 | 表现 |
|---------|------|------|
| 0-50% | 低 | 正常工作 |
| 50-70% | 中 | 早期信息开始被稀释，偶发遗忘 |
| 70-90% | 高 | 信息被截断，推理质量下降 |
| >90% | 极高 | 严重遗忘，早期结论丢失，幻觉风险上升 |

**上下文耗尽的后果**

当上下文被填满，最早的信息会被"挤出"：

- **信息截断**：早期的关键决策、项目背景被丢弃
- **遗忘早期结论**：讨论了半天的结论，因为上下文不够被遗忘
- **推理质量下降**：Agent 在不完整的上下文中推理，可能给出矛盾结论
- **幻觉风险上升**：上下文不足时，Agent 更容易"编造"信息填补空白

**压缩策略：让记忆文件成为真正的记忆**

上下文窗口是有限的，但文件系统是无限的。把关键信息写入文件，上下文只放当前任务需要的"热数据"。

| 策略 | 做法 | 效果 |
|------|------|------|
| 记忆文件精简 | 活跃文件 ≤15KB | 快速读取，减少上下文占用 |
| 过时内容归档 | 历史内容移入知识库 | 上下文只放当前需要的 |
| 对话信息及时写入 | 讨论出的结论立即写文件 | 防止遗忘，减少重复讨论 |
| INDEX 导航 | 通过索引定位而非全量加载 | 按需加载，节省上下文 |

**信息密度优化：一字千金**

上下文是稀缺资源，每一句话都在消耗预算。

| ❌ 低效做法 | ✅ 高效做法 | 节省比例 |
|-----------|-----------|--------|
| 重复说明同一件事 | 只说一次，后续引用 | ~30% |
| 一段话说清楚的事用两段 | 精简表达，一句是一句 | ~20% |
| 不关键的细节占据空间 | 关键决策优先，过滤噪音 | ~40% |
| 每次都加载完整文件 | 用 offset+limit 分批读取 | 按需使用 |

**token 预算意识**

每次读写文件、调用工具都消耗 token。花在低价值信息上的 token，等于挤占了高价值推理的空间。

| 操作 | token 消耗 | 优化建议 |
|------|-----------|---------|
| 读取大文件 | 高 | 用 offset+limit 分批读取 |
| 重复搜索同一信息 | 浪费 | 搜索结果写入文件，下次直接读 |
| 全量加载知识库 | 极高 | 通过 INDEX 定位，只读需要的 |
| 重复描述背景 | 累积 | 关键信息写入文件，一次说明长期有效 |

**判断何时需要"开新对话"的信号**

有时上下文已经碎片化到无法有效工作，这时开新对话是最优解。

| 信号 | 说明 | 行动 |
|------|------|------|
| **连续遗忘** | Agent 反复问同一个问题 | 上下文已丢失，开新对话并写好 MEMORY |
| **推理质量明显下降** | 回答开始自相矛盾 | 上下文碎片化，开新对话 |
| **反复问已回答过的问题** | 早期结论被截断 | 开新对话，把 MEMORY 写完整 |
| **上下文占用 >80%** 且任务未完成 | 空间不足 | 评估：继续还是开新对话 |

**实操技巧：用 offset+limit 读取大文件**

当文件太大时，不要一次性全量读取：

```bash
# 读取文件前 100 行
cat file.py | head -n 100

# 读取文件中间部分（200-300 行）
cat file.py | sed -n '200,300p'

# 或者用 read_file 工具的 offset+limit 参数
```

这样既获取了需要的信息，又不会让上下文爆掉。

**本节与其他章节的关联**

- 2.4 节讲的"文件状态不好时的后果"，根因就是上下文管理不当
- 3.9 技能管理中，skill_load 后读 SKILL.md 也是上下文操作
- 4.6 长期项目的跨 session 追踪，核心就是上下文恢复

## 三、AI 协作工作流

以下工作流不是预设的，而是项目实践中被问题逼出来的——每一条规则背后都有犯过错。

### 协作全景图

> **协作阶段方法论**（适用于想快速理解全貌的读者）

本节将协作过程分为四个阶段，帮助你建立整体认知：

| 阶段 | 目标 | AI 职责 | 人的职责 |
|------|------|---------|---------|
| **想法→需求** | 将模糊痛点转化为结构化需求 | 梳理：拆解痛点为结构化需求 | 定义业务边界和约束 |
| **需求→方案** | 可行性验证和方案细化 | 分析：可行性验证、方案对比 | 判断方向、做取舍 |
| **方案→落地** | 推进为可运行代码并验收 | 实施+验收：编码、日志分析 | 真实环境测试、提供完整信息 |
| **迭代决策** | 持续优化，必要时止损 | 迭代推进：瓶颈分析、方案输出 | 止损判断、确认审批 |

**核心原则**：人提方向，AI 出方案；人做测试，AI 做分析；人掌舵，AI 执行。

---

> **【协作工作流】** 3.1-3.9

### 3.1 为什么需要workflow

没有 workflow 时出的问题（项目实例）：

| 问题 | 发生了什么 | 后果 |
|------|-----------|------|
| AI未经确认就改代码 | AI 看了代码觉得可以优化，直接改了 push | 破坏了开发者有意保留的设计，回滚返工 |
| 改了A忘B | AI 只改了问题点，没检查关联模块 | 其他模块崩了，二次修复 |
| 同一问题反复出现 | AI 每次对话都重新分析同一问题 | 浪费轮次，效率低 |
| AI在死胡同死磕 | VPN闪退问题 AI 试了4种方案全失败 | 浪费时间，代码越改越乱 |

**结论**：Workflow 的存在是为了约束 AI 行为、减少返工、提高一致性。没有 workflow，AI 会"高效地犯错"——它完成任务的速度可能很快，但方向可能完全不对，最后要花更多时间回滚和修复。

### 3.2 需求分析与调研流程

适用于：用户提出模糊想法、不确定能不能做、需要先调研再决定方向的场景。

```
用户提模糊需求/想法 → AI 梳理需求 → 确认目标和边界 → 可行性分析 → 开发者确认方向 → 拆解为可执行任务 → 开发者确认拆解 → 按任务执行
```

**各环节详解**：

| 环节 | 谁负责 | 做什么 | 为什么要有这一步 |
|------|--------|--------|-----------------|
| 用户提需求 | 人 | 描述想法或痛点，不需要很精确 | AI 的作用就是帮你把模糊变清晰 |
| AI 梳理需求 | AI | 明确三件事：目标是什么、边界在哪、约束条件 | 不梳理就动手 = 方向可能偏 |
| 确认目标和边界 | 人 | 确认 AI 的理解是否正确 | AI 可能理解偏了，不确认就白干 |
| 可行性分析 | AI | 给出能做/不能做/部分能做，各自的风险和成本 | 帮你做决策，不是替你做决策 |
| 开发者确认方向 | 人 | 决定做不做、做哪个方向 | 方向决策权在人 |
| 拆解为可执行任务 | AI | 每个任务有明确产出和验收标准 | 任务拆得越细，执行越可控 |
| 开发者确认拆解 | 人 | 确认任务拆解合理 | 避免遗漏或粒度不对 |
| 按任务执行 | AI | 派 sub-agent 或自己执行 | 进入具体的工作流（BUG修复/新功能开发等） |

**项目实例**：开发者说"能不能做个精简版"，AI 应该先分析：精简版的定位是什么？精简哪些功能？与现有版本的差异？是否值得开发？——而不是直接开写代码。实际案例中，精简版开发后才发现代码量与原版几乎一样，不值得做。

**常见错误**：AI 自以为理解了需求就开跑。模糊需求不要直接动手，先问清楚再干。

### 3.3 新功能/需求开发流程

适用于：需求已经明确，需要从零开发新功能的场景。

```
确认需求理解 → 给实现方案 → AI自查方案 → 开发者确认 → 编码实现 → 自检代码 → 提交 → 用户测试
```

**各环节详解**：

| 环节 | 谁负责 | 做什么 | 为什么要有这一步 |
|------|--------|--------|-----------------|
| 确认需求理解 | AI+人 | AI复述理解，开发者确认没偏 | AI 理解的需求和开发者的意图可能不一致 |
| 给实现方案 | AI | 架构设计、模块划分、技术选型 | 先想清楚再动手，避免边写边改 |
| AI自查方案 | AI | 检查方案完整性、可行性、是否有遗漏 | 方案阶段发现问题成本最低 |
| 开发者确认 | 人 | 判断方案是否符合预期 | 架构决策权在人 |
| 编码实现 | AI | 按方案编写代码 | 一次性写完核心功能 |
| 自检代码 | AI | 检查语法、import、功能是否完整 | 改完自己先过一遍 |
| 提交 | AI | push到代码仓库 | — |
| 用户测试 | 人 | 在真实环境验证 | AI 无法替代真实环境测试 |

**项目实例**：开发者说"帮我加一个伏击模式"，AI 先确认理解（预取SKU→T-0直接请求？），给方案（新增哪些模块、改哪些现有模块），确认后实现。

**常见错误**：AI 不确认需求理解就直接开写。理解偏差1度，执行偏差180度。

### 3.4 信息整理与输出流程

适用于：搜索/采集信息、整理资料、生成报告等场景。

```
明确采集目标和产出格式 → 执行搜索/采集 → 整理为结构化内容 → 给用户看初步结果 → 确认方向 → 补充完善 → 交付 → 归档到知识库
```

**各环节详解**：

| 环节 | 谁负责 | 做什么 | 为什么要有这一步 |
|------|--------|--------|-----------------|
| 明确目标 | 人+AI | 确定要采什么、产出什么格式 | 目标不清 = 采一堆没用的 |
| 执行采集 | AI | 搜索/读取/提取 | AI 的信息获取速度远快于人工 |
| 整理结构化 | AI | 按产出格式组织信息 | 原始信息 ≠ 可用信息 |
| 看初步结果 | 人 | 确认方向对不对 | 闷头整理完才发现偏了 = 白干 |
| 补充完善 | AI | 根据反馈调整 | 迭代比一次到位更高效 |
| 交付 | AI | 生成最终文件 | — |
| 归档 | AI | 重要结论写入知识库 | 不归档 = 下次又要重新采集 |

**项目实例**：开发者说"帮我查一下目标平台的 API"，AI 搜到信息后先给用户看中间结果确认方向，再深入挖掘。如果不确认就一口气整理完，可能挖的是用户不需要的方向。

**常见错误**：AI 闷头整理不给中间结果。先给中间结果确认方向，不要闷头跑完才发现偏了。

### 3.5 文件生成类操作流程

适用于：生成文档、报告、PPT等文件产出的场景。

```
确认内容和格式要求 → 给方案/大纲 → 开发者确认 → 生成 → 验收质量 → 交付 → 不满意则调整
```

**各环节详解**：

| 环节 | 谁负责 | 做什么 | 为什么要有这一步 |
|------|--------|--------|-----------------|
| 确认要求 | 人 | 说明内容、格式、风格等要求 | 没要求的生成 = 盲猜 |
| 给方案/大纲 | AI | 输出文档结构和大纲 | 大纲确认比全文返工成本低得多 |
| 开发者确认 | 人 | 确认方向和结构 | 方向对了再填充内容 |
| 生成 | AI | 按确认的大纲生成完整文件 | — |
| 验收质量 | AI | 检查格式、内容完整性 | 自检后再交付 |
| 交付 | AI | 发送文件给用户 | — |
| 调整 | AI | 根据用户反馈修改 | 迭代优化 |

**常见错误**：AI 不给大纲直接生成全文。大纲确认5分钟，全文返工30分钟。

### 3.6 BUG修复流程

```
记录问题 → AI 给方案 → AI 自查方案 → 开发者确认 → 执行修改 → 自检代码 → 提交 → 测试验证
```

**各环节详解**：

| 环节 | 谁负责 | 做什么 | 为什么要有这一步 |
|------|--------|--------|-----------------|
| 记录问题 | 人 | 完整描述现象、触发条件、与预期的差异 | 信息不足时 AI 只能猜测，效率极低 |
| AI 给方案 | AI | 分析根因，提出修复方案 | AI 的技术分析速度远快于人工 |
| AI 自查方案 | AI | 以审查视角审查：遗漏、矛盾、副作用、隐含假设 | AI 写的方案也有盲区，自查过滤掉大部分低级问题 |
| 开发者确认 | 人 | 判断方案是否符合设计意图 | AI 可能"优化"掉你有意为之的设计，必须人工把关 |
| 执行修改 | AI | 按确认后的方案一次性修改 | 不逐条改，避免冲突 |
| 自检代码 | AI | 检查修改是否完整、是否有遗漏 | 防止改了 A 忘了 B |
| 提交 | AI | 提交代码 | — |
| 测试验证 | 人 | 在真实环境运行测试 | AI 无法感知环境差异，最终验证必须人做 |

**违反流程的后果**（项目实例）：

- 跳过"开发者确认" → AI 未经确认直接 push 2 次，改掉了开发者有意保留的设计 → 回滚返工
- 跳过"AI 自查" → 方案有隐含假设，修改后其他模块崩了 → 二次修复
- 跳过"测试验证" → AI 环境验证通过但打包后缺模块 → 用户运行报错

### 3.7 代码重构/优化流程

适用于：性能优化、架构调整、代码清理等场景。与 BUG 修复的区别：BUG 修复是"修错"，重构是"改好"——前者有明确目标，后者容易扩大化。

```
明确优化目标 → 给方案（含风险评估）→ 开发者确认 → 实施 → 回归测试 → 提交 → 用户测试
```

**各环节详解**：

| 环节 | 谁负责 | 做什么 | 为什么要有这一步 |
|------|--------|--------|-----------------|
| 明确目标 | 人 | 说明优化什么（性能？可维护性？架构？） | "优化"太笼统，没有明确目标的重构容易越改越乱 |
| 给方案 | AI | 包含：改什么、怎么改、**风险评估**（影响范围、可能改坏什么） | 重构最怕改A坏B，方案里必须说明影响 |
| 开发者确认 | 人 | 判断方案是否值得做、风险可接受 | 重构的收益要大于风险 |
| 实施 | AI | 按方案修改 | 一次性改完，不零碎改 |
| 回归测试 | AI | 检查现有功能是否正常 | 确保没改坏 |
| 提交 | AI | 提交代码 | — |
| 用户测试 | 人 | 真实环境验证 | AI 的回归测试覆盖不了真实场景 |

**项目实例**：开发者说"帮我把代码从 V2 合并到 V1"，AI 给方案时必须说明：哪些模块会受影响？合并后 V1 原有功能是否可能异常？回归测试策略是什么？——不能只说"合并"不评估风险。

**常见错误**：AI 重构时不评估风险。重构方案必须包含风险评估，没有风险评估的方案 = 不完整方案。

### 3.8 workflow中Agent怎么被调用

详细描述每一步 Agent 读了什么、用了什么：

| 步骤 | Agent读取的文件 | 使用的工具/skill | 说明 |
|------|---------------|-----------------|------|
| 记录问题 | MEMORY（项目上下文） | — | 理解问题发生在哪个模块 |
| AI给方案 | TOOLS（harness）、INDEX→知识库（历史经验） | 可能加载相关skill | 按规则出方案 |
| AI自查 | TOOLS（workflow规则） | — | 按自查标准审查 |
| 执行修改 | MEMORY（当前分支/版本） | bash、文件编辑 | 确保改对位置 |
| 测试验证 | — | — | 人做，Agent无法替代 |

### 3.9 技能全生命周期管理

技能（skill）是 Agent 能力的扩展，让它在特定领域更专业。从选型到加载到使用，形成完整的生命周期。

#### 3.9.1 技能选型

不是所有任务都需要 skill，但需要 skill 的任务不能乱选。

**选型原则**：根据需求匹配技能，不匹配则不调用。

| 场景 | 应该选什么 | 说明 |
|------|-----------|------|
| 生成图表/可视化 | echart skill | 专为数据可视化设计 |
| 生成 PPT | create-ppt skill | 完整 PPT 工作流 |
| 处理 PDF | pdf skill | PDF 文本提取、合并、表单填写 |
| 处理 Word | docx skill | 文档创建、编辑、修订追踪 |
| 搜索/安装技能 | skill-assistant | 自动识别平台并搜索安装 |
| 法律咨询 | legal-advisory-report | 法律咨询全流程指导 |
| 股票投资分析 | 股票相关技能 | 用 coze_skill_dynamic_expand 扩充 |

**动态扩充**：对于复杂专业领域（如股票投资、法律、媒体创作等），使用 `coze_skill_dynamic_expand` 工具根据需求关键词扩充匹配技能。

```
调用方式：coze_skill_dynamic_expand(query="具体需求关键词")
```

**不匹配的代价**：用错了 skill 不仅达不到效果，还可能引入错误的执行逻辑。比如用文字生成图片的 skill 去处理数据分析，就会南辕北辙。

#### 3.9.2 技能加载

加载 skill 不是目的，理解 skill 的能力边界才是。

**标准加载流程**：

```
用户提出需要 skill 的任务
→ 判断需要哪个 skill
→ 调用 skill_load 工具加载
→ 读取 SKILL.md 理解技能能力
→ 按 SKILL.md 的规范执行任务
```

**skill_load 后的必做动作**：读取 SKILL.md。

很多问题出在这里——Agent 看到 skill 名称就以为理解了，直接执行，结果因为没读规范而踩坑。

| 问题 | 根因 | 正确做法 |
|------|------|---------|
| 加载了 skill 但效果不好 | 只读了名称没读 SKILL.md | skill_load 后必须读 SKILL.md |
| 执行结果不符合预期 | 没按 skill 规范执行 | 按 SKILL.md 的步骤和规范走 |
| 不知道 skill 能做什么 | 没理解 skill 能力边界 | 读 SKILL.md 看描述和示例 |

**渐进式加载原则**：当 SKILL.md 引用到子资源（模板、脚本等）时，按需读取，而不是一开始就全量加载所有内容。

#### 3.9.3 技能组合

复杂任务可同时加载多个技能，按需组合。

| 场景 | 技能组合 | 说明 |
|------|---------|------|
| 生成报告并转 PPT | docx + create-ppt | 先写报告内容，再生成 PPT |
| 数据分析并可视化 | excel_master + echart | 数据处理 + 图表生成 |
| 法律咨询并生成文书 | legal-advisory-report + docx | 分析 + 文档生成 |

**组合注意事项**：

- 注意技能间的依赖关系，先加载哪个后加载哪个
- 避免技能能力重叠导致冲突
- 组合后验证输出是否符合各 skill 的规范

#### 3.9.4 技能与规则对 Agent 行为的约束

**skill（技能）**：

- Agent 加载 skill 后获得特定能力（如 PPT 制作、文档生成、表格处理）
- skill 有使用规范，Agent 必须按规范执行
- skill 是 Agent 能力的扩展，让它在特定领域更专业

**TOOLS.md里的harness（操作手册）**：

- 写明了 GUI 背景色、布局规则、命名规范、避坑经验等
- 没有这些，Agent 按默认逻辑生成，可能不符合用户偏好
- harness 是项目的"本地知识"，不是通用规则

**TOOLS.md里的workflow（流程规范）**：

- 写明了 BUG 修复流程、确认机制、提交规范等
- 没有这些，Agent 按自己判断走，容易犯错
- workflow 是约束 Agent 行为的"宪法"

### 3.10 sub-agent全链路

**sub-agent 是什么**：

- 主agent 派出的独立执行单元，拥有和主agent 相同的工具、文件、技能
- 不了解对话历史，只通过任务描述获取上下文
- 异步执行，完成后结果返回主agent

**为什么要用 sub-agent**：

| 优势 | 说明 |
|------|------|
| 不阻塞对话 | 主agent 可以继续和用户聊天，子任务在后台跑 |
| 并行处理 | 同时跑多个子任务，提高效率 |
| 隔离风险 | 子任务失败不影响主对话 |

**为什么要管理 sub-agent**：

| 风险 | 说明 |
|------|------|
| 上下文缺失 | 子agent 不了解对话历史，任务描述不完整就会跑偏 |
| 任务卡死 | 子agent 可能长时间无响应 |
| 结果验收 | 子agent 结果可能不符合要求 |

**主agent 派 sub-agent 的流程**：

```
主agent判断任务需要异步执行
→ 编写任务描述（必须包含完整上下文：项目背景、当前进展、关键文件位置、产出要求）
→ 调用sessions_spawn派发任务
→ 继续和用户对话（不阻塞）
→ 收到子agent结果后验收
→ 如果不符合要求：通过sessions_send追加修改要求
→ 如果卡死：通过sessions_abort kill掉
```

**sub-agent 自身的工作流**：

```
收到任务描述
→ 读取相关文件恢复上下文（按任务描述中的路径）
→ 可能加载需要的skill
→ 执行任务
→ 产出结果文件
→ 返回结果摘要给主agent
```

**完整链路**：

```
用户提需求 
→ 主agent判断是否需要sub-agent 
→ 编写任务描述（关键！） 
→ 派发sub-agent 
→ sub-agent独立执行 
→ 结果返回主agent 
→ 主agent验收（检查质量、是否符合要求）
→ 不符合则追加要求或重新派发 
→ 符合则呈现给用户
```

**任务描述为什么是关键**：

- 子agent 没有对话历史，任务描述是它唯一的上下文
- 描述不完整 → 子agent 不知道文件在哪、产出要放哪、质量标准是什么
- 好的任务描述应包含：项目背景、当前进展、相关文件路径、产出要求、结果发送目标

**项目实例**：结项报告子agent卡死 → 任务太复杂且文件路径没写清楚 → kill后重新拆分任务

**最终定义的工作流**：

```
1. 主agent理解需求，判断执行方式（自己干 / sub-agent / Calendar日程）
2. 如果用sub-agent：编写完整任务描述，包含5要素（背景/关键人物/任务交接/产出要求/结果目标）
3. 派发sub-agent
4. 主agent继续与用户对话
5. 收到结果后验收：先检查质量，不符合则追加要求
6. 符合要求则呈现给用户
```

**派发规范详解**：

1. **串行原则**：B的输入依赖A的产出时，必须等A完成再派B，不能并行。

```
✅ 正确：派A分析页面结构 → 等A完成 → 派B基于A的分析结果写代码
❌ 错误：同时派A和B → A还没分析完B就开始写 → B基于错误假设工作
```

2. **指定源文件**：派发时明确写"基于哪个文件改"，不能假设原文件就是最新版。

```
✅ 正确：基于 ./项目A/src/browser.py 的最新代码改，不要基于备份副本
❌ 错误：基于"当前代码"改 → 可能有未覆盖的副本导致版本混乱
```

3. **递进链路**：多级产出有递进关系时（如配置→展开→精简），在任务描述中写明链路。

```
场景：配置TOOLS.md → 展开成方法论完整版 → 精简成入门手册
任务描述应该写明：
- 你的产出会被谁消费（下一个环节是谁）
- 产出格式要求（配置→完整版→精简版的格式差异）
- 链路位置（你是第几个环节）
```

#### 3.10.1 sub-agent 异常处理

sub-agent 执行过程中可能出现各种异常，掌握这些场景的处理方法是保障协作效率的关键。

**超时处理**

sub-agent 运行时间过长是最常见的异常。

| 信号 | 可能原因 | 处理方式 |
|------|---------|---------|
| sub-agent 超过预期时间未返回 | 任务太复杂、卡在某个环节、网络问题 | 先用 sessions_status 查看状态 |
| status 显示 running 但无进展 | 可能卡死 | 用 sessions_abort 终止，重新评估任务 |

**超时防范**：

- 任务描述中预估完成时间
- 设置合理的 timeout 参数
- 长时间任务考虑拆分为多个短任务

**卡死处理**

sub-agent 没有响应是最让人焦虑的场景。

```
发现 sub-agent 无响应
→ 使用 sessions_status 查看状态
  → status = running：可能是任务本身耗时长，等待或主动询问
  → status = 无响应/超时：用 sessions_abort 终止
→ 分析卡死原因：
  → 任务描述不清晰 → 重写任务描述
  → 任务太复杂 → 拆分为多个子任务
  → 依赖外部资源（网站/接口）不可用 → 换方案
→ 重新派发，修正问题后继续
```

**项目实例**：结项报告 sub-agent 卡死 → 任务太复杂且文件路径没写清楚 → abort 后重新拆分任务

**结果质量不达标**

这是最需要耐心的场景——sub-agent 返回了结果，但验收不通过。

| 常见问题 | 表现 | 正确处理 |
|---------|------|---------|
| 结果不符合要求 | 格式不对、内容缺失、方向偏了 | 用 sessions_send 补充要求，让 sub-agent 继续完善 |
| 遗漏了关键部分 | 任务描述没说清楚 | sessions_send 补充上下文，重新派发 |
| 质量不够好 | 能用但不够精 | sessions_send 提出具体修改意见 |

**关键原则：不要自己上手重做**

这是最重要的协作原则之一。当 sub-agent 结果不达标时：

| ❌ 错误做法 | ✅ 正确做法 |
|-----------|-----------|
| 主 agent 亲自重做一遍 | 用 sessions_send 补充要求 |
| "算了，我自己做吧" | 让 sub-agent 继续修改 |
| 完全推翻重来 | 明确指出哪里不对，让它改 |

**为什么不能自己上手**：

1. **违背分工原则**：主 agent 做执行者，谁做验收者？
2. **破坏协作闭环**：sub-agent 不知道哪里错了，下次还会犯
3. **效率不增反降**：主 agent 做一次，下一次还是要重复

**sessions_send 补充要求的写法**：

```
之前的结果有以下问题：
1. [具体问题1]
2. [具体问题2]

请继续完善，重点关注：
- [修改方向1]
- [修改方向2]

完整上下文：[相关文件路径或内容摘要]
```

**防止异常的关键：任务描述要完整**

大多数异常（超时、卡死、质量不达标）的根因都在任务描述不完整。

完整任务描述应包含：

| 要素 | 说明 | 缺失后果 |
|------|------|---------|
| 项目背景 | 让 sub-agent 理解为什么做 | 可能方向偏了 |
| 当前进展 | 知道从哪里继续 | 可能从头开始浪费 |
| 相关文件路径 | 能找到需要的资源 | 可能找不到文件 |
| 产出要求 | 知道要交付什么 | 结果可能不对 |
| 结果发送目标 | 知道结果返回给谁 | 结果可能丢了 |
| 预期时间 | 知道什么时候该返回 | 超时无法判断 |
| 异常处理方式 | 遇到问题怎么办 | 卡死无法恢复 |

### 3.11 Calendar（日程/定时任务）

**Calendar 是什么**：

Calendar 是 Agent 的定时任务工单系统。到设定时间，系统唤起一个独立 session 读取工单并执行。

**Calendar 能做什么**：

| 功能 | 说明 |
|------|------|
| 一次性提醒 | "明天下午1点半提醒我开会" |
| 定期执行 | "每天下午查一下黄金价格" |
| 重复追踪 | "持续追踪AI行业新闻" |
| 分步推进长期项目 | 拆解目标，分步创建日程 |

**适合什么场景**：

- 定时提醒（提前10分钟提醒）
- 定期执行（有明确产出，如"查价格并整理发送"）
- 时间敏感任务（设具体时间）
- 需要重复执行的检查/报告任务

**不适合什么**：

- 需要秒级响应的检查（频率不够）
- 需要生成大文件的复杂任务（用 sub-agent 更合适）
- 不严格、随手一看的检查（用 Heartbeat）

**创建 Calendar 时的注意事项**：

| 字段 | 要求 | 示例 |
|------|------|------|
| summary | 用清晰动作短语，主语是 Agent 自己 | "同步黄金价格" ✅<br>"提醒我黄金价格" ❌ |
| description | 写给执行 session 看的工单详情，必须包含完整上下文（目标、步骤、文件位置） | 因为执行 session 不了解对话历史 |
| time_range | 弹性调度窗口，设置后系统选择空闲时段执行，避开整点高峰 | 除时间严格敏感的任务外都应设置 |
| rrule | 重复规则。最稳妥写法只填 freq/interval/count/until，让 dtstart 推导时间 | 不要额外填 byhour/byminute/byday/bymonthday，除非用户明确要求 |

> **rrule 避雷**：不要填与 dtstart 冲突的字段，否则会导致创建失败。
> 
> **修改重复规则**：直接 update 即可，系统自动保留历史实例、终止旧日程、创建新的未来日程。

**Calendar 与 sub-agent 的区别**：

| 对比维度 | sub-agent | Calendar |
|----------|-----------|----------|
| 执行时机 | "现在就做" | "到时候做" |
| 结果返回 | 秒级返回 | 需要等触发时间 |
| 适用类型 | 一次性任务 | 定期重复任务 |

**提示词示例**：

```
帮我每天下午3点查一下XX价格，整理后发给我
```

```
明天早上9点提醒我开会
```

### 3.12 Heartbeat（心跳/周期轮询）

**Heartbeat 是什么**：

每 30 分钟触发一次的轻量检查机制。

**工作方式**：

每次触发读取 heartbeat.md 中的检查清单，逐项快速检查，有事通知无事沉默。

**适合什么场景**：

- 检查后台任务/sub-agent 是否完成或卡住
- 回复其他渠道消息（如群聊消息、文档评论）
- 时间要求不严格的提醒（久坐提醒、晚间休息）
- 监控类检查（股价触及阈值时提醒）

**不适合什么**：

- 需要生成文件或报告的任务（用 Calendar）
- 有明确执行时间要求的任务（用 Calendar）
- 耗时较长的数据分析（用 Calendar）
- 需要高于 30 分钟频率的检查（用 Calendar 创建重复日程）

**Heartbeat 与 Calendar 的区别**：

| 对比维度 | Heartbeat | Calendar |
|----------|-----------|----------|
| 定位 | "顺手看一眼" | "按时干活" |
| 耗时 | 秒级完成 | 有明确产出，可能耗时 |
| 产出 | 无事沉默，不产文件 | 必须有产出，可能耗时 |
| 频率 | 固定 30 分钟 | 可自定义 |

**heartbeat.md 格式要求**：

```
检查清单
- 检查爬虫任务是否正常运行，卡住则通知用户
- 看一眼飞书群是否有需要回复的消息
- 用户久坐超过2小时，提醒活动一下
- XX股价低于300时提醒用户
```

> 首行必须是"检查清单"纯文本标题，之后用 `- ` 列表

**使用工具**：

`heartbeat_md_read` / `heartbeat_md_update` / `heartbeat_md_delete`

> 严禁直接用文件工具操作 heartbeat.md

### 3.13 三种执行方式选择指南

| 场景 | 选什么 | 原因 |
|------|--------|------|
| 用户要求现在就做一件事 | sub-agent | 即时执行，结果秒回 |
| 每天固定时间做某事 | Calendar | 定时触发，重复执行 |
| 定期追踪某个话题 | Calendar | 有明确产出，用 topic_tracking 技能 |
| 30 分钟看一眼状态 | Heartbeat | 轻量检查，无事不扰 |
| 生成文件/报告 | Calendar 或 sub-agent | Heartbeat 不适合 |
| 抢购类精确时间任务 | Calendar | 时间敏感，设具体时间 |
| 久坐提醒/休息提醒 | Heartbeat | 不严格，30 分钟粒度够 |

**避雷清单**：

| ❌ 禁止 | ✅ 正确 |
|--------|--------|
| 用 Heartbeat 做需要生成文件的任务 | 用 Calendar 或 sub-agent |
| 用 Calendar 做需要秒级频率的检查 | 考虑其他方案 |
| 用 sub-agent 做需要等到特定时间的任务 | 用 Calendar |
| Heartbeat 放太多检查项 | 每项都要 30 分钟内跑完，保持轻量 |
| Calendar 的 description 写太简略 | 执行 session 不了解对话历史，必须写完整 |
| Calendar 的 rrule 填与 dtstart 冲突的字段 | 只填 freq/interval/count/until |

### 3.14 沟通规范

**有效信息结构：现象 + 触发条件 + 与预期的差异**

| 低效表达 | 有效表达 | 区别 |
|---------|---------|------|
| "程序崩了" | "开启VPN后闪退，关闭VPN正常运行" | 加了触发条件 |
| "时间不准" | "配置 -0.1s 偏移，日志显示实际 -0.3s" | 加了与预期的差异 |
| "按钮点不动" | "点击后无响应，但日志显示 click 事件已触发" | 区分了 UI 层和逻辑层 |

信息越精确，AI 定位越快。模糊描述 = AI 猜测 = 方向可能偏 = 浪费轮次。

### 3.15 确认机制

**结构性改动必须先说明方案、等待确认、再实施。**

起源：AI 看了代码后觉得某处可以优化，没问人就改了，结果破坏了开发者的设计意图。

规则：任何涉及架构、流程、配置项的改动，AI 必须先输出方案说明，开发者确认后才动手。

### 3.16 止损机制

AI 倾向于持续尝试，不具备止损能力。设定试错上限（3-4 次），超限则换方向或回滚。

人需要区分两种情况：
- **尚未找到解法**：问题性质明确、方案方向正确，只是还没试到对的参数 → 继续
- **该方向无解**：多次不同方案均失败，问题可能在更底层 → 止损

### 3.17 文档化复盘

长对话后 AI 会遗忘早期结论。关键设计必须文档化，AI 复盘时先读取文档恢复上下文。

实际操作：每个项目维护 INDEX 文件，记录架构决策、模块职责、已知问题。AI 被唤起时先读 INDEX，再进入工作。

**复盘模板**：详见百科「复盘模板」章节。

复盘是迭代改进的基础。完整复盘模板包含：项目背景与目标、实际交付结果、关键决策回顾、踩坑记录、可复用经验、待改进点、知识沉淀。详见百科3.17详解。

**复盘时机**：每个项目/版本迭代完成后进行一次复盘，不需要等到"完美时机"——完成本身就是复盘的理由。

**复盘产出**：复盘的结果应转化为可执行内容：
- 教训 → 写入 TOOLS.md 的 harness
- 经验 → 固化为 workflow
- 踩坑记录 → 归档知识库，供后续项目参考

### 3.18 产出验收与测试策略

本章分为两个子节：
- 3.18.1 产出验收方法（审查验证思路）——交付前的结果确认
- 3.18.2 测试策略——交付前的过程保障

#### 3.18.1 产出验收方法（审查验证思路）

验收不是"看起来对就行"。AI 产出的文档、代码、结构调整，都需要用审查验证思路二次验证。

**验收不是挑刺，是双向检查。**

"验收"的核心是**双向验证**：
- **正向验证**（需求→产出）：沿着需求方向走，确认"该做的做了没"
- **反向挑刺**（产出→问题）：逆着产出方向挑，找"不该出的问题出了没"

两者缺一不可。只做正向验证，可能漏掉意外引入的问题；只做反向挑刺，可能遗漏需求覆盖。

---

**正向验证（需求→产出）：沿着需求方向走，确认"该做的做了没"**

| 检查项 | 含义 | 怎么做 |
|--------|------|--------|
| **1. 覆盖完整性** | 逐条对照原始需求，每条是否有对应产出？ | 列出原始需求的每一条，检查产出是否一一对应 |
| **2. 意图准确性** | 产出是否真正满足了需求意图，还是字面糊弄？ | 问自己：用户说"加个说明"，加了一句废话=字面覆盖但意图没满足 |
| **3. 产出可用性** | 产出能不能实际工作？文档能不能读？代码能不能跑？ | 不只是"看起来有了"，要实际验证或设计验证方法 |

**正向验证的常见误区**：

| 误区 | 示例 | 后果 |
|------|------|------|
| 字面覆盖≠意图满足 | 用户说"加个说明"，AI加了一句"本文件仅供参考" | 字面上有了，实际上毫无价值 |
| 需求没列全 | 用户说"改一下那个按钮"，但实际有3个类似按钮 | 改错了/改了部分 |
| 做了但质量不够 | 需求是"优化性能"，AI改了代码但没测性能 | 交付物不可用 |

---

**反向挑刺（产出→问题）：逆着产出方向挑，找"不该出的问题出了没"**

| 检查项 | 含义 | 怎么做 |
|--------|------|--------|
| **4. 逻辑审计** | 因果是否成立？概念是否矛盾？上下文是否衔接？ | 检查章节之间的逻辑关系 |
| **5. 边界完整性** | 序号连续？引用有效？命名一致？标题变了内容有没有跟着变？ | 扫描序号、引用、命名 |
| **6. 副作用扫描** | 改A是否意外影响B？删一段是否连带信息丢失？ | 检查关联内容是否受影响 |
| **7. 合适性判断** | 这么做是否合适？有没有更好的方式？是否过度设计或不足设计？ | 问"这样处理合适吗" |
| **8. 交叉验证** | 用grep/脚本扫描，不靠肉眼 | 工具扫描代替人工检查 |

**反向挑刺的常见误区**：

| 误区 | 示例 | 后果 |
|------|------|------|
| 只检查"有没有"不检查"对不对" | 序号对了，但内容逻辑混乱 | 形式正确，内容错误 |
| 改完不检查关联 | 改了第3章标题，但正文引用还是"第3章" | 上下文不一致 |
| 相信AI不犯错 | AI说"改完了"，不验证 | 遗漏未改的地方 |
| 肉眼检查大文件 | 1000行文档靠肉眼扫 | 必然遗漏 |

---

**交付引用：验收通过后必须立即引用给用户，无例外**

这是最容易跳过的环节。验收通过后，必须立即用 computer:// 协议引用给用户看，不要等用户来问。

```
验收通过 → 立即引用给用户 → 等待用户确认
```

**常见跳步**：
- 验收通过后继续干下一件事，用户不知道已经改完
- 引用时机太晚，用户已经忘了改了什么
- 没等用户确认就当上线处理了

**正确做法**：
```
修改完成 → 验收检查通过 → 立即引用给用户 → 等待用户确认 → 完成
```

---

**TOOLS中的验收标准（快速参考）**：

```
验收产出（检查任何交付物）

正向验证（需求→产出）：沿着需求方向走，确认"该做的做了没"
1. 覆盖完整性 — 逐条对照原始需求，每条是否有对应产出？
2. 意图准确性 — 产出是否真正满足了需求意图，还是字面糊弄？
3. 产出可用性 — 产出能不能实际工作？文档能不能读？代码能不能跑？

反向挑刺（产出→问题）：逆着产出方向挑，找"不该出的问题出了没"
4. 逻辑审计 — 因果是否成立？概念是否矛盾？上下文是否衔接？
5. 边界完整性 — 序号连续？引用有效？命名一致？标题变了内容有没有跟着变？
6. 副作用扫描 — 改A是否意外影响B？删一段是否连带信息丢失？
7. 合适性判断 — 这么做是否合适？有没有更好的方式？是否过度设计或不足设计？
8. 交叉验证 — 用grep/脚本扫描，不靠肉眼

交付引用 — 验收通过后必须立即用 computer:// 协议引用给用户，无例外
```

**项目实例**：方法论v1.2重构时，子agent修改了章节序号但未调整内容顺序——3.6 BUG修复的内容排在了3.2需求分析前面，序号对了但物理位置错了。如果只看序号连续性检查会通过，但"内容跟随"和"逻辑关系"检查能发现位置错误。

**关键原则**：验收是交付前的最后一道防线。AI 产出的东西，必须像审查验证别人的工作一样去验收——假设可能有错，逐一验证。

#### 3.18.2 测试策略——过程保障

测试是过程中的质量保障，与验收（结果确认）不同。验收回答"交付的东西对不对"，测试回答"做出来的过程中间有没有错"。

**代码测试**

| 测试类型 | 做什么 | 什么时候做 |
|---------|------|-----------|
| 单元测试 | 验证每个函数/模块独立正确 | 编写函数后立即执行 |
| 边界测试 | 测试极端值、空值、异常输入 | 单元测试后 |
| 集成测试 | 验证多个模块组合后的行为 | 单元测试通过后 |
| 回归测试 | 确保修改没有破坏已有功能 | 每次修改后 |

**代码测试的实操**：

```
# 单元测试思路：给每个关键函数写测试用例
def test_api_client_timeout():
    """测试 API 客户端超时处理"""
    client = APIClient(timeout=0.001)  # 极端小超时
    result = client.fetch("/test")
    assert result.error == "timeout"  # 应该超时

# 边界测试：空值、极端值、异常值
def test_config_edge_cases():
    """测试配置边界情况"""
    assert parse_timeout("0") == 0
    assert parse_timeout("-1") == default_timeout  # 负数回退到默认
    assert parse_timeout("abc") == default_timeout  # 非法值回退
```

**文档测试**

| 检查类型 | 怎么做 | 工具 |
|---------|-------|------|
| 逻辑审查 | 检查内容是否有逻辑漏洞、前后矛盾 | 人工审查 |
| 交叉验证 | 检查引用、链接、数据是否一致 | grep、脚本扫描 |
| 格式检查 | 检查格式是否统一、序号是否连续 | 脚本检查 |
| 可读性测试 | 让非作者阅读，看是否理解 | 人工审查 |

**文档测试的实操**：

```
# 检查序号连续性的脚本逻辑
sections = extract_sections(doc)
for i in range(len(sections) - 1):
    current_num = parse_section_number(sections[i])
    next_num = parse_section_number(sections[i + 1])
    if not is_sequential(current_num, next_num):
        print(f"序号不连续: {current_num} -> {next_num}")
```

**自动化测试 vs 手动测试**

| 适合自动化 | 必须手动 |
|-----------|---------|
| 重复执行的验证 | 需要主观判断的检查 |
| 有明确 pass/fail 标准 | 需要用户体验反馈 |
| 边界条件和异常场景 | 需要真实环境（如 GUI） |
| 回归测试（防止修改破坏已有功能） | 需要探索性测试 |

**自动化测试的适用场景**：
- 每次修改后的回归测试
- 边界条件验证
- 引用/序号检查
- 格式统一性检查

**必须手动测试的场景**：
- GUI 界面体验
- 真实环境的端到端流程
- 需要主观判断的文案
- 用户实际使用感受

**测试与验收的关系**

| 维度 | 测试 | 验收 |
|------|------|------|
| 时机 | 过程中持续进行 | 交付前一次性确认 |
| 目的 | 发现过程中的错误 | 确认最终结果正确 |
| 执行者 | AI + 人 | 人（最终判断） |
| 自动化 | 可以自动化 | 通常需要人工判断 |

**关键原则**：测试是过程保障，发现问题越早成本越低；验收是最终确认，确保交付物符合要求。两者缺一不可。

### 3.19 幻觉识别与防范体系

在实际协作中，AI 幻觉不是随机的，而是有明确的模式。识别模式是防范的第一步。

#### 3.19.1 幻觉模式分类

| 模式 | 特征 | 典型表现 | 危害等级 |
|------|------|---------|---------|
| **静态分析幻觉** | 从源码/编译产物推断运行时行为 | "在 CSS bundle 里没找到 `.product-card`，所以它已废弃" | 🔴 高——直接导致选择器错误 |
| **不完整阅读幻觉** | 只读部分代码就下全局结论 | "V1 的结账就是直接点确认"——只读了 fallback 函数，没读主函数 | 🔴 高——遗漏关键步骤 |
| **理解后重写幻觉** | 看懂了逻辑，用自己的方式重新实现 | "V1 用 `.product-card`"——实际 V1 走的是另一个函数，用的是 `.itemCard` | 🔴 高——引入微妙偏差 |
| **复合幻觉** | 多个错误互相"印证" | 先误以为"改版了"→所以"旧代码不能用了"→所以"要重写" | ⚫ 极高——错误叠加，方向完全偏离 |
| **编造事实幻觉** | AI 凭空编造不存在的信息 | "该平台已经改版为 React 18"——实际并未改版 | 🟡 中——影响判断方向 |
| **过度推断幻觉** | 从有限证据推出不可支持的结论 | "这个 class 在 bundle 中没出现"→"它已废弃" | 🟡 中——跳过了验证环节 |

**关键规律**：幻觉越"合理"越危险。静态分析幻觉和不完整阅读幻觉之所以难发现，是因为推论链条看似完整，只是前提有误。

#### 3.19.2 幻觉检测方法

怀疑 AI 结论时，用以下方法快速判断可信度。

**方法一：交叉验证法**

核心思路：**AI 的结论 vs 已有可运行代码的实际行为，取实际行为为准。**

```
判断流程：
AI 给出技术结论（如"选择器 X 已废弃"）
→ 问：现有代码用 X 能正常运行吗？
  → 能 → AI 结论一定是错的（或至少不完整）
  → 不能 → AI 结论可能对，但需要进一步验证
```

**方法二：追溯源头法**

核心思路：**区分"观察"和"推论"，只采信有直接观察支持的结论。**

```
判断流程：
AI 给出结论
→ 追问：这个结论的依据是什么？
  → 依据是直接观察（如"我运行了代码，输出是X"）→ 可采信
  → 依据是推论（如"我在源码中没找到X"）→ 需要验证
  → 依据是另一个AI结论 → 不能采信，必须找到原始观察
```

**方法三：反向验证法**

核心思路：**如果 AI 说"X 已废弃/不存在"，用 X 跑一下。**

```
判断流程：
AI 说"X 已废弃/不存在/不可用"
→ 在可运行环境中用 X 测试
  → 正常工作 → AI 结论错误
  → 确实不可用 → AI 结论可能正确
```

**提示词示例**（给 Agent 的验证指令）：

```
在你修改任何选择器/API/技术细节之前：
1. 先用现有代码在目标环境验证当前选择器/API是否仍然有效
2. 如果有效，不得以"已废弃"为由替换
3. 如果确认失效，截图或抓包留证，再提出替换方案
```

**方法四：路径追踪法**

核心思路：**AI 说"参考代码就是这么写的"时，追踪代码的实际执行路径。**

```
判断流程：
AI 说"V1 用选择器 X"
→ 追问：V1 的默认配置走哪个函数？那个函数里用的是什么选择器？
  → AI 能明确回答（函数名+行号+选择器）→ 核实
  → AI 笼统回答"V1 代码里就是这样的"→ 可能只看了部分代码
```

**提示词示例**（给 Agent 的路径追踪指令）：

```
读取参考代码时，必须追踪完整执行路径：
1. 从入口函数（main）开始
2. 读取配置文件确认默认模式
3. 根据默认模式确定实际调用的函数
4. 只从实际调用的函数中提取技术细节
5. 不要从 fallback/备用函数中提取并当作"V1 就是这样的"
```

#### 3.19.3 幻觉预防提示词

预防比检测成本更低。以下提示词模板可在任务描述中使用，从源头降低幻觉风险。

**模板一：禁止静态分析推论**

```
⚠️ 幻觉防范规则（本次任务必须遵守）：
1. 不得通过分析源码/CSS/JS bundle 来推断页面 DOM 结构
2. "源码中没找到某个 class" ≠ "该 class 已废弃/不存在"
3. 前端框架可以同时给元素多个 class，CSS 可以分 chunk 加载
4. 如果需要确认选择器是否有效，必须通过动态 DOM 验证（F12 检查或实际运行）
5. 宁可承认"无法通过静态分析确认"，也不要给出不可验证的结论
```

**模板二：强制完整阅读**

```
⚠️ 幻觉防范规则（本次任务必须遵守）：
1. 读取参考代码时，必须读完所有相关函数，不能只读一个就下结论
2. 如果代码有多个模式/函数，必须先确认哪个是默认/主用模式
3. 从配置文件或入口函数追踪实际执行路径，确定真正调用的函数
4. 只从实际执行的函数中提取技术细节作为参考
5. 备用/fallback 函数中的逻辑不得代表"参考代码的做法"
```

**模板三：禁止理解后重写**

```
⚠️ 幻觉防范规则（本次任务必须遵守）：
1. 有参考代码时，必须直接从参考代码的对应函数中复制，不要"理解后重写"
2. "理解后重写" ≠ "复制后适配"——看懂逻辑再用自己的方式实现会引入偏差
3. 复制后可以适配（改变量名、调整参数），但不能凭记忆重写
4. 任何 DOM 选择器/API 调用/技术细节，必须能在参考代码的对应函数中找到原文
5. 找不到出处 = 不能写
```

**模板四：综合防范（适用于高风险任务）**

```
⚠️ 幻觉防范规则（本次任务必须遵守）：
1. 技术结论必须可用已有代码或可验证证据交叉验证
2. "源码中没找到" ≠ "不存在/已废弃"，必须用实际运行验证
3. 读取参考代码时必须看完所有函数，搞清哪个是主模式
4. 有参考代码时直接复制适配，不"理解后重写"
5. 选择器/API/技术细节必须有出处，找不到出处 = 不能写
6. 如果现有代码用相同选择器能正常工作，"改版"结论一定是假的
7. 宁可承认不知道，也不要用不可验证的结论修改代码
```

#### 3.19.4 推论可信度评估框架

不是所有 AI 推论都同等可信。用这个框架快速评估：

| 推论类型 | 可信度 | 示例 | 处理方式 |
|---------|--------|------|---------|
| 基于代码原文的引用 | 🟢 高 | "browser.py 第231行用了 `.itemCard`" | 核实后可直接采信 |
| 基于运行结果的观察 | 🟢 高 | "运行 `querySelectorAll('.itemCard')` 返回3个元素" | 可直接采信 |
| 基于文档/官方来源的引用 | 🟢 高 | "API 文档显示该接口支持 POST" | 可采信 |
| 基于代码逻辑的推论 | 🟡 中 | "这个函数先检查了X，说明X是前置条件" | 需验证，可能遗漏分支 |
| 基于源码分析的推论 | 🟡 中 | "CSS bundle 中包含 `.ProductWidget` 样式" | 存在但可能不代表实际使用方式 |
| 基于缺失的推论 | 🔴 低 | "源码中没找到 `.product-card` → 已废弃" | 必须验证，极易出错 |
| 基于另一个AI结论的推论 | 🔴 低 | "另一个 agent 说改版了，所以……" | 必须独立验证，不可传递信任 |
| 无依据的编造 | ⚫ 极低 | "该平台已升级到 React 18" | 直接拒绝 |

**使用原则**：
- 可信度 🟢 的结论可直接用于方案
- 可信度 🟡 的结论必须经过验证才能用于方案
- 可信度 🔴/⚫ 的结论不得用于任何修改决策

#### 3.19.5 幻觉发现后的修复流程

当发现 AI 产生了幻觉，按以下流程处理：

```
发现可疑结论
→ 暂停当前任务，不要继续基于该结论推进
→ 用检测方法（交叉验证/追溯源头/反向验证）确认是否为幻觉
  → 不是幻觉 → 继续推进
  → 确认是幻觉 → 进入修复流程
    → 1. 回滚所有基于该幻觉的修改（git revert/reset）
    → 2. 追溯幻觉来源：哪个结论是错的？从哪里开始的？
    → 3. 定位影响范围：基于该幻觉还改了哪些地方？
    → 4. 清除所有受影响的修改
    → 5. 从正确的起点重新开始
    → 6. 将教训写入 TOOLS.md 的 harness（防范规则）
    → 7. 将案例写入知识库（供后续参考）
```

**关键原则**：

1. **不叠加修改**：发现幻觉后立即停手，不要在错误基础上继续改。错误方向走得越远，回滚成本越高。
2. **先回滚再修**：不要试图在错误的代码上"修正确"——先把错误代码还原，再从正确起点出发。
3. **写 harness 而不是写检讨**：教训的价值在于转化为可执行规则，不是写在文档里感慨。每条 harness 必须是"下次遇到同样情况时应该怎么做"。
4. **归档案例但保持脱敏**：案例是教学素材，但涉及具体项目名称、选择器等技术细节需要脱敏处理。

**回滚提示词**（给 Agent）：

```
发现幻觉后的处理：
1. 立即停止当前任务
2. 检查 git log，找到引入错误修改的 commit
3. 回滚到幻觉修改之前的版本
4. 不要在错误代码基础上修修补补
5. 回滚后重新从参考代码的实际执行路径出发
6. 完成后总结：幻觉模式 + 防范规则，写入 TOOLS.md
```

#### 3.19.6 幻觉防范速查

| 你要做什么 | 提示词/规则 | 一句话要点 |
|-----------|-----------|-----------|
| 派 sub-agent 分析页面结构 | 加"模板一：禁止静态分析推论" | 源码中没找到 ≠ 不存在 |
| 让 AI 参考现有代码改功能 | 加"模板三：禁止理解后重写" | 直接复制适配，不凭记忆重写 |
| 让 AI 读取代码提取逻辑 | 加"模板二：强制完整阅读" | 先追踪执行路径，只从主函数搬 |
| 高风险任务（涉及选择器/API） | 加"模板四：综合防范" | 结论必须有出处，找不到出处 = 不能写 |
| AI 给出"已废弃"类结论 | 用"反向验证法" | 用旧方式跑一下，能跑 = 没废弃 |
| AI 说"参考代码就是这样的" | 用"路径追踪法" | 从入口追踪到实际函数，确认真正走的路径 |
| 发现 AI 幻觉 | 按 3.19.5 修复流程处理 | 先回滚，再修，写 harness |

### 3.20 Workflow 遗忘防范与纠正

幻觉是 AI 给出错误结论的问题，而 Workflow 遗忘是 AI 在正确方向上跳步的问题——该确认的不确认、该停的不停、该问的不问。两者危害不同但同样常见，且 Workflow 遗忘更容易反复发作。

**遗忘的表现模式**

| 模式 | 表现 | 为什么会发生 | 典型后果 |
|------|------|------------|---------|
| **跳过确认直接执行** | 看了代码觉得可以改，直接改了 push | AI 倾向于"高效完成任务"，确认环节被当作"低效等待" | 改掉用户有意保留的设计，回滚返工 |
| **看到代码就忍不住改** | 用户只是问"为什么"，AI 就开始读代码准备改 | AI 把"解释"等同于"修复"，理解问题和解决问题混为一谈 | 未经授权的修改，方向可能完全偏 |
| **给方案但不等确认就动手** | 方案刚给出来，AI 就开始执行 | AI 认为"方案已经想了，执行只是顺路" | 方案有问题时已经改完了，返工成本翻倍 |
| **同样错误反复犯** | 被纠正过一次，下次又犯 | harness 写了但 Agent 没有在执行前主动读取/检查 | 每次都要人工提醒，协作效率极低 |

**即时制止提示词**

当用户发现 Agent 跳步时，使用以下语言立即制止：

```
停。你又在跳步了。
→ 问你解释的时候，不要开始改代码
→ 给方案的时候，不要动手执行
→ 等我确认了再动
```

**Agent 收到制止后应执行的标准流程**

```
1. 立即停止当前操作
2. 回退到跳步前的位置（如果已经开始改代码，检查是否需要回滚）
3. 补上缺失的环节（先给方案→自查方案→等确认→再执行）
4. 确认方案后再继续
5. 反思：为什么会跳步？哪个环节被省略了？
6. 将反思结果写入 TOOLS.md
```

**预防性提示词**（写入任务描述开头）

```
⚠️ Workflow 约束（本次任务必须遵守）：
1. 任何写/改/删文件操作，必须先给方案→等确认→再执行
2. 用户问"为什么"时，只解释原因，不要动手改
3. 给出方案后，明确说"请确认"，等待用户回复后再执行
4. 方案未经确认，不得调用任何文件修改工具
5. 每完成一个关键步骤，自查：我跳步了吗？我等确认了吗？
```

**加强训练方法**

| 方法 | 具体做法 | 效果 |
|------|---------|------|
| **硬规则写入规则文件** | 在 Agent 的工作规则文件中明确列出"什么操作必须确认"，如：写文件、删文件、push代码、修改架构决策 | 每次唤醒时自动读取，作为行为约束 |
| **违反后强制回滚** | 一旦发现跳步执行，必须回滚到跳步前的状态，重新走流程 | 建立跳步=返工的负反馈 |
| **记忆锚点** | 在记忆文件或规则文件中埋入提醒，如"⚠️ 上次因跳步被纠正，务必先确认再执行" | 每次唤醒时被提醒 |
| **执行前自检清单** | 在规则文件中维护一个 checklist，Agent 在执行写操作前必须过一遍 | 形成执行前的"刹车"习惯 |
| **Heartbeat 定期自检** | 在 heartbeat.md 中加入 Workflow 合规检查项，每30分钟自动触发 Agent 自查是否跳步 | 不依赖记忆，强制周期性提醒 |
| **定期复盘** | 通过 Heartbeat 机制定期检查是否有跳步行为，更新规则，不依赖人工提醒 | 持续强化，防止退步 |

**Workflow 合规自检清单**（建议写入项目规则文件）：

```
执行写/改/删操作前，逐项检查：
□ 我给出方案了吗？
□ 我自查方案了吗？（遗漏？矛盾？副作用？）
□ 用户确认了吗？
□ 我是在确认之后才动手的吗？
□ 用户问的是"解释/原因"吗？如果是，我不应该动手改
□ 我是否在"顺便"做用户没要求的事？如果是，停下来
```

**Workflow 防范速查**

| 你要做什么 | 提示词/规则 | 一句话要点 |
|-----------|-----------|-----------|
| AI 跳步执行 | 用"即时制止提示词" | 停→回退→补环节→确认→继续 |
| AI 反复跳步 | 用"加强训练方法" | 硬规则+强制回滚+记忆锚点+自检清单 |
| 预防 AI 跳步 | 在任务描述加"预防性提示词" | 未经确认不得修改任何文件 |

### 3.21 邮件分身与信息追踪

邮件分身是 Agent 的自主邮件管理能力，可以自动分拣、回复、拦截邮件。区别于普通的邮件发送（需要用户触发），邮件分身是常驻能力——它持续监控邮箱，按照预定义规则处理邮件，不需要人工干预。

**与执行方式的关系**

| 执行方式 | 触发条件 | 邮件分身的区别 |
|---------|---------|--------------|
| sub-agent | 用户或主 Agent 派发 | 一次性任务，不持续 |
| Calendar | 定时触发 | 到点执行，不持续 |
| Heartbeat | 定期轮询 | 定期检查，不持续 |
| **邮件分身** | 邮件到达时 | **持续监控，有新邮件立即响应** |

**典型使用场景**

| 场景 | 效果 |
|------|------|
| 金额/余额变动追踪 | 自动记录余额变化，无需手动统计 |
| 安全邮件拦截与过滤 | 主动防御，降低安全风险 |
| 验证码/临时密码提取 | 自动化流水线的关键环节 |
| 订单/交易状态同步 | 自动同步重要信息到 MEMORY |

### 3.22 文件修改安全规范

文件修改是高风险操作——一旦出错，轻则返工，重则引入难以追踪的幻觉。本节总结文件修改的安全规范。

**核心原则：副本先行，逐级验收。**

**副本先行**

修改任何文件时，先复制一份副本，在副本上改。这是最基本的安全保障——无论后续操作出了什么问题，原文件都是安全的。

```
# 标准操作流程
cp 原文件 原文件.备份
# 在副本上修改
# 验收通过后替换原文件
```

切忌：直接修改原文件、无备份的操作都是高风险行为。

**逐级验收机制**

文件修改必须经过逐级验收才能替换原文件：

| 阶段 | 执行者 | 检查什么 | 通过标准 |
|------|--------|---------|---------|
| 1. 子agent产出 | sub-agent | 按任务要求完成修改 | 自检通过 |
| 2. 主agent验收 | 主Agent | 内容完整性、引用正确性、无幻觉 | 脚本验证通过 |
| 3. 用户验收 | 用户 | 确认修改符合意图 | 用户明确确认 |
| 4. 替换原文件 | 主Agent | 确认前三步都通过 | 三步全部完成 |

**防幻觉校验：文件操作后必须校验**

每次文件修改后，必须校验两件事：

1. **引用路径是否存在**：文件中引用的其他文件路径，是否都真实存在
2. **内容是否丢失**：修改后的文件，关键内容是否完整保留

```
# 验证脚本示例
# 检查所有 .md 引用路径是否存在
grep -h '\.\/[^ ]*\.md' 文件名.md | while read path; do
  if [ ! -f "$path" ]; then
    echo "❌ 引用路径不存在: $path"
  fi
done
```

**保留清单：派子agent修改时必须明确的内容**

派 sub-agent 修改文件时，必须明确列出：
1. **保留清单**：哪些内容必须保留在原位，不能删、不能迁移
2. **删除清单**：哪些内容可以删除
3. **新增位置**：新增内容放到哪个文件/哪个位置

| ❌ 错误做法 | ✅ 正确做法 |
|-----------|-----------|
| "精简TOOLS.md" | "精简TOOLS.md，铁律区/方案先行/产出交付三节必须原样保留，其余过时内容移到知识库" |
| "整理知识库" | "整理知识库，项目A的harness移到`知识库/项目A/`目录，B项目的移到`知识库/项目B/`；禁止删除任何harness" |

**反面案例**：详见百科「3.22详解：TOOLS被替换成8个路径的完整案例」。

**摘要**：主Agent派sub-agent"精简TOOLS.md"，没有给出保留清单。sub-agent把内容替换成了8个不存在的`.md`文件路径，原有内容全部丢失。详见百科3.22详解。

**安全检查清单**（文件修改后必查）：

```
□ 我做了副本吗？
□ 我在副本上改的吗？
□ 引用路径都存在吗？
□ 关键内容没有丢失吗？
□ 主agent验收通过了吗？
□ 用户确认了吗？
□ 三步全部通过才替换原文件
```

## 四、记忆与知识库管理

### 4.1 为什么需要主动管理

Agent 的记忆文件（TOOLS.md、MEMORY.md 等）随项目增长会膨胀。不管理的后果：

- 文件过大（曾有 TOOLS.md 27KB、MEMORY.md 31KB 的情况）→ Agent 每次读取浪费 token → 上下文空间被占满 → 有效信息被截断
- 过时信息与当前信息混杂 → Agent 做出基于过期数据的判断 → 方案错误
- 无索引、无分类 → Agent 全量加载所有文件 → 效率低、成本高

### 4.2 优化思路

**核心原则：活跃文件保持精简，过时内容归档到知识库。**

优化前的状态：
- TOOLS.md 27KB：历史harness、已解决的问题、过期skill列表混杂
- MEMORY.md 31KB：从V1到V3的完整历史堆在一个文件里
- 无知识库、无索引：所有信息在对话和少量文件中

优化后的状态：
- TOOLS.md 5KB：只保留当前有效的harness和workflow
- MEMORY.md 3.5KB：只保留活跃项目状态
- 知识库目录：历史归档、skill参考、项目索引，按分类存放
- INDEX.md：唯一导航入口

**优化指令示例**：

```
请帮我整理 TOOLS.md 和 MEMORY.md：
1. 已过时的harness和已解决的问题移到知识库归档
2. 只保留当前有效的harness和workflow
3. 创建知识库目录结构，建立INDEX.md导航
4. 确保活跃文件合计不超过15KB
```

**结果**：TOOLS 27KB→5KB，MEMORY 31KB→3.5KB，历史内容完整保留在知识库中，Agent 读取活跃文件时 token 消耗大幅降低。

**优化实操步骤**：

1. **先读当前文件**：读取 TOOLS.md 和 MEMORY.md，标记过时内容
2. **建知识库目录**：创建 `知识库/项目A/`、`知识库/项目B/` 等分类目录
3. **迁移过时内容**：将已删除模块的 harness、历史问题记录移到知识库
4. **精简活跃文件**：只保留当前有效的 harness、workflow、活跃项目状态
5. **建 INDEX.md**：列出知识库中每个文件的路径和摘要
6. **验证**：活跃文件合计不超过15KB，INDEX与实际文件一致

**关键规则**：

| 规则 | 说明 | 违反后果 |
|------|------|---------|
| 活跃文件合计不超过15KB | 超出则上下文溢出风险大 | Agent 读取时被截断，做出不完整决策 |
| 同一信息只在一个地方定义 | 避免冲突 | 多处定义导致不一致，Agent 无法判断 |
| 修改文件后必须同步更新INDEX | INDEX过时=Agent被误导 | Agent 找不到资源，或找到错误资源 |

### 4.3 INDEX 导航体系

INDEX.md 是 Agent 访问所有资源的唯一入口。

**设计思路**：
- Agent 唤醒后先读 INDEX，不直接读具体文件
- INDEX 记录每个资源的路径、内容摘要、适用场景
- Agent 根据当前任务从 INDEX 定位所需资源，按需加载

**INDEX 结构示例**：

```
# 知识库索引

## 项目A
- 项目A/INDEX.md — 项目入口
- 项目A/[文件] — [内容说明]

## 项目B
- 项目B/INDEX.md — 项目入口
- 项目B/[文件] — [内容说明]

## 通用资源
- 通用/工具技巧/ — 工具使用技巧
- 通用/文档处理/ — 文档生成规范
```

详细模板和目录结构示例，详见百科「4.3详解：INDEX体系模板」。

**INDEX 如何引导 skill 和知识库**：

当用户触发特定任务时，Agent 的执行路径：

```
用户提问 → Agent 读取 INDEX → 定位相关分类 → 读取具体文件 → 加载对应 skill → 执行任务
```

例如：用户说"帮我修GUI的BUG" → Agent 读INDEX → 定位到"项目B/GUI开发笔记.md" → 读取harness → 按规范修复

**好处**：
- 按需加载，不全量读取，节省 token
- 信息集中管理，避免多源冲突
- Agent 唤醒后通过 INDEX 快速恢复上下文

**问题**：
- 维护成本：每次新增内容需同步更新 INDEX
- 同步负担：如果 INDEX 与实际文件不一致，Agent 会被误导
- 需要纪律：开发者/Agent 修改文件后必须更新 INDEX

**知识库建设全流程**：

1. **触发条件**：当记忆文件超过15KB，或 Agent 频繁遗忘/混淆历史决策时
2. **建目录**：按项目/领域分类，如 `知识库/项目A/`、`知识库/项目B/`
3. **迁移内容**：从 MEMORY 中提取详细技术文档，从 TOOLS 中提取已解决的历史 harness
4. **建 INDEX**：每个分类一行，写清路径和内容摘要
5. **设定加载规则**：Agent 先读 INDEX，按需加载具体文件

**目录结构示例**：

```
知识库/
├── INDEX.md                    # 唯一导航入口
├── 项目A/
│   ├── INDEX.md               # 项目A完整索引
│   ├── [文件].md             # 开发记录/踩坑/api等
│   └── 历史归档.md            # 旧版本记录
├── 项目B/
│   ├── INDEX.md               # 项目B完整索引
│   └── [文件].md             # 开发记录
├── 通用/
│   ├── 工具技巧/              # 工具使用技巧
│   └── 文档处理/              # 文档生成规范
└── skills/
    └── 参考文档/             # 模板和参考
```

详细模板，详见百科「4.3详解：INDEX体系模板」。

**避雷清单**：

| ❌ 禁止 | ✅ 正确 |
|--------|--------|
| 把所有内容放一个文件（文件过大，加载慢） | 按主题分类，每个文件控制大小 |
| 不建 INDEX（Agent 不知道有什么资源可用） | 始终维护 INDEX 作为导航入口 |
| 在活跃文件和知识库中重复同一信息（会冲突） | 同一信息只在一个地方定义 |
| 忘记同步更新 INDEX（INDEX 过时=Agent被误导） | 修改文件后立即更新 INDEX |
| 在 INDEX 中写太长的描述（INDEX 是导航不是内容） | INDEX 每行控制在一行摘要 |
| Agent 全量加载所有文件 | Agent 先读 INDEX，按需加载 |
| 知识库内容从不归档（越来越臃肿） | 定期归档，保持知识库精简 |

### 4.4 记忆文件职责划分

| 文件 | 职责 | 写什么 | 不写什么 |
|------|------|--------|---------|
| SOUL.md | Agent 人设与自我认知 | 性格设定、行为规则 | 项目信息 |
| TOOLS.md | 工具使用经验 | 当前有效的harness、workflow | 已解决的问题、过期信息 |
| MEMORY.md | 项目与事务记忆 | 活跃项目状态、待办事项 | 历史归档、详细技术文档 |
| USER.md | 用户个人信息 | 偏好、背景、习惯 | 项目信息 |
| 知识库/ | 归档与参考资料 | 历史归档、skill参考、项目索引 | 活跃项目状态 |
| INDEX.md | 导航入口 | 资源路径与摘要 | 具体内容 |

**原则**：每个文件有明确的职责边界，同一信息只在一个地方定义。活跃文件保持精简（合计不超过15KB），详细内容放知识库按需加载。

### 4.5 管理原则总结

| 原则 | 说明 | 核心价值 |
|------|------|---------|
| **文件即记忆** | 关键信息必须写入文件，不能只存在于对话中 | 对话结束后 Agent 还能记得 |
| **INDEX 即入口** | Agent 通过 INDEX 导航所有资源 | 不全量加载，高效定位 |
| **按需加载** | 通过索引定位，只加载当前任务需要的文件 | 节省 token，避免上下文溢出 |
| **定期归档** | 过时内容从活跃文件移到知识库 | 保持活跃文件精简 |
| **单一入口** | 同一信息只在一个地方定义 | 避免多源冲突 |
| **同步维护** | 修改文件后必须更新 INDEX | 确保 Agent 获取的信息准确 |

### 4.6 长期项目的跨 session 追踪

4.1-4.5 讲的是"怎么存"，本节讲的是"怎么用记忆持续追踪项目"。

对于跨越数周、数月的长期项目，跨 session 状态恢复是核心能力。每次新对话开始时，Agent 需要知道"我在哪、要去哪、之前做了什么"。

**跨 session 状态恢复**

每次新对话开始，Agent 应该自动恢复上下文：

```
新对话开始
→ 读取 MEMORY.md（项目当前状态）
→ 读取 INDEX.md（导航到相关资源）
→ 按需加载具体文件
→ 进入工作状态
```

如果 MEMORY 写得完整，这个恢复过程是瞬时的。如果 MEMORY 是空的，Agent 就"失忆"了。

**进度追踪：MEMORY.md 的长期项目结构**

对于长期项目，MEMORY.md 应该包含：

| 字段 | 内容 | 示例 |
|------|------|------|
| 项目阶段 | 当前在哪个阶段 | V3 开发中，第二轮优化 |
| 已完成 | 哪些功能已交付 | 伏击模式、拦截模式、时间校准 |
| 进行中 | 当前在做什么 | 优化错误处理逻辑 |
| 待办 | 接下来做什么 | 完善日志系统、GUI 优化 |
| 阻塞项 | 什么卡住了 | VPN 环境限制，暂不解决 |
| 关键决策 | 重要决定的结论 | 选择器用 `.itemCard`，不走 fallback |

**示例 MEMORY.md（长期项目）**：

```
# 项目状态

### 当前阶段
[项目当前开发阶段]

### 已完成
- [已完成的功能列表]

### 进行中
- [正在进行的工作]

### 待办
- [ ] [待办事项]

### 阻塞项
- [阻塞项及原因]

### 关键决策
- [重要决策及理由]
```

**项目状态文档化：关键信息必须写入文件**

这是最重要的长期项目原则：

| ❌ 不要做 | ✅ 正确做法 |
|---------|-----------|
| 关键决策只在对话里讨论 | 讨论完结论立即写入 MEMORY |
| 踩坑经验只在对话里说 | 教训立即写入 TOOLS.md 的 harness |
| 架构选择只在脑子里 | 选择和理由都写入 MEMORY |

**为什么？因为对话会结束、会遗忘、会被截断。文件不会。**

**交接清单：换 Agent 或开新对话时必须传递的信息**

当需要换 Agent、换对话、或交接给其他人时，以下信息是必须传递的：

| 信息类型 | 必须包含 | 传递方式 |
|---------|---------|---------|
| 项目阶段 | 当前在哪个阶段 | MEMORY.md |
| 未完成工作 | 正在做什么、待做什么 | MEMORY.md 待办项 |
| 关键决策 | 重要决定的结论和理由 | MEMORY.md 关键决策 |
| 踩坑记录 | 已知问题和规避方案 | TOOLS.md harness |
| 相关文件 | 所有资源的路径 | INDEX.md |

**交接标准动作**：

```
交接前：
1. 更新 MEMORY.md，确保状态完整
2. 检查 INDEX.md 是否与实际文件一致
3. 整理 TOOLS.md，确保 harness 是最新的
4. 确认没有关键信息只存在于对话中

交接时：
- 把 MEMORY.md 的关键部分发给接收方
- 说明当前阻塞项和风险
- 强调已知踩坑和规避方案
```

**与 4.1-4.5 的逻辑关系**

| 章节 | 回答的问题 | 核心产出 |
|------|-----------|---------|
| 4.1 为什么管理 | 为什么记忆文件会出问题 | 意识问题存在 |
| 4.2 优化思路 | 文件太大了怎么办 | 精简+归档方法 |
| 4.3 INDEX 体系 | 怎么组织知识库 | 导航结构 |
| 4.4 职责划分 | 每个文件干什么 | 边界清晰 |
| 4.5 管理原则 | 总体原则是什么 | 6大原则 |
| **4.6 跨session追踪** | **怎么用记忆持续追踪项目** | **MEMORY 结构+交接清单** |

4.1-4.5 是"静态"的管理，4.6 是"动态"的应用。理解"怎么存"是为了更好地"怎么用"。

### 4.7 知识库整理实操案例

以下是一次真实知识库整理的全流程记录，展示从诊断到执行到验证的完整过程。完整案例详见百科「4.3详解」。

**触发：知识库出了什么问题**

项目进入V3阶段后，知识库积累了大量内容，但缺少系统性整理：

| 问题 | 表现 |
|------|------|
| 缺INDEX | 知识库根目录没有全局INDEX，只有零散的子目录 |
| 分类不全 | 项目A的专属知识散在通用分类里，没有独立项目目录 |
| 内容混乱 | 部分文件位置与内容不匹配 |

**诊断：全局INDEX vs 实际文件交叉比对**

首先列出知识库所有文件，与现有INDEX对比，详见百科「4.3详解」。

诊断结果发现3类问题：

| 问题类型 | 示例 | 严重程度 |
|---------|------|---------|
| **缺INDEX** | 知识库根目录没有全局INDEX | 高 |
| **内容错位** | 项目A的harness在项目B目录下 | 中 |
| **垃圾文件** | 过期的临时脚本、脱敏前的原始文档 | 低 |

**执行：五步整理流程**

1. **建项目INDEX**：为每个项目建立专属INDEX
2. **迁移项目专属知识**：将散落的项目专属内容归拢
3. **新增分类**：根据实际内容补充分类
4. **清理垃圾文件**：删除过期文件
5. **更新全局INDEX**：反映新的目录结构

详见百科「4.3详解：知识库整理实操」。

**教训：子agent精简文件时的幻觉**

派sub-agent精简文件时，必须给出保留清单。详见百科「3.22详解」。

### 4.8 知识库维护机制

4.1-4.7讲的是知识库"怎么建"，本节讲的是知识库"怎么维护"——保持知识库长期健康运转的机制。

**定期检查：每月一次INDEX完整性检查**

知识库不是建好就完事的，需要定期维护。建议每月执行一次完整性检查：

```bash
#!/bin/bash
echo "=== 知识库完整性检查 ==="

echo "[1/3] 检查INDEX引用的文件是否存在..."
# ... 详见百科
```

**检查项详解**：

| 检查项 | 含义 | 后果 |
|-------|------|------|
| INDEX引用的文件是否存在 | 检查死链 | Agent按INDEX导航时找不到文件 |
| 实际文件是否在INDEX中 | 检查孤儿文件 | 新增内容无法被Agent定位 |
| 项目INDEX是否覆盖项目目录 | 检查遗漏 | 子项目Agent可能找不到资源 |

**权重调整：TOOLS.md章节顺序影响Agent行为优先级**

TOOLS.md的章节顺序不是随意的——越靠前的章节，在上下文被压缩时越不容易丢失。

| 区域 | 位置 | 内容 | 权重 |
|------|------|------|------|
| **铁律区** | TOOLS.md最顶部 | 合规自检、方案先行、产出交付 | 最高（每次必过） |
| **流程区** | 中间 | BUG修复workflow、需求开发workflow | 中（用到才查） |
| **参考区** | 底部 | 工具选型、资源定位、验收标准 | 低（查阅类） |

**实操原则**：

1. **铁律区控制在200行以内**：这是每次操作必过的内容，越精简越好
2. **流程区按需加载**：workflow只在执行相关任务时读取
3. **参考区可以较长**：不影响核心操作的参考内容可以放底部

---

**配置文件维护原则：配置文件调整需同步权重**

配置文件（如TOOLS.md）不仅是内容容器，还是权重载体。修改配置文件时，必须同时考虑**内容正确性**和**权重位置**。

**原理：章节顺序=权重顺序**

同样内容放在不同位置，对Agent行为的影响完全不同：

| 内容 | 放铁律区 | 放附录区 | 影响差异 |
|------|---------|---------|---------|
| 验收双向检查原则 | 每次操作都提醒 | 可能被忽略 | ✅核心原则vs❌边缘参考 |
| 副本先行原则 | 每次修改都触发 | 可能忘 | ✅强制执行vs⚠️依赖自觉 |

**为什么权重这么重要**

当上下文被压缩时，Agent会优先保留前面的内容。如果核心原则放在附录区，在上下文紧张时可能被截断，导致Agent行为偏离预期。

**维护流程：改内容时必须检查权重位置**

```
1. 明确改什么、为什么改
2. 评估：改的内容应该放在哪个区？（铁律区/流程区/参考区）
3. 检查：放的位置是否影响其他条目的优先级？
4. 放对位置
5. 同步更新下游文档（完整版→入门手册）的对应章节和范例
```

**配置文件修改的标准工作流**：

```
1. 明确改什么、为什么改
2. 给方案 → 必须包含权重检查：改的内容应该放在哪个区？放的位置是否影响其他条目的优先级？
3. 用户确认 → 走副本修改 → 验收 → 覆盖原文件
4. 同步更新方法论（完整版→入门手册）的对应章节和范例

关键：同样内容放在铁律区和放在附录区，对行为的影响完全不同。改内容不改位置=权重失调。
```

**常见维护坑**：

| 坑 | 表现 | 后果 | 防范 |
|----|------|------|------|
| 只改内容不改位置 | 把重要原则移到新位置但不更新序号 | 权重混乱，核心原则可能被忽略 | 改内容时同步检查位置是否合适 |
| 改了TOOLS不改方法论 | TOOLS已更新，方法论文档范例还是旧的 | 递进链路不一致，下游文档过时 | 建立TOOLS变更→同步更新方法论的机制 |
| 改了完整版不改入门手册 | 完整版已更新，精简版还是旧内容 | 递进链路断裂，信息不一致 | 维护时同步更新递进链路的所有文档 |
| 子agent规范扩充后不更新 | 子agent规范从5条变8条，方法论还是5条 | 子agent执行依据与主agent规则不一致 | 子agent规范变更时同步更新方法论 |

**维护检查清单**（修改配置文件后必查）：

```
□ 我检查了改的内容应该放在哪个区吗？
□ 我检查了新位置是否影响其他条目的优先级吗？
□ 我同步更新了完整版方法论的相关章节吗？
□ 我同步更新了入门手册的相关章节吗？
□ 我更新了方法论附录中的TOOLS范例吗？
□ 递进链路的所有文档是否一致？
```

**整理prompt：通用的维护prompt模板**

当需要Agent帮忙维护知识库时，使用以下prompt模板：

```
请检查知识库完整性：
1. 列出知识库所有.md文件（find命令）
2. 逐一检查INDEX.md中是否有引用（交叉比对）
3. 逐一检查INDEX引用的文件是否存在（脚本验证）
4. 报告缺失项（死链、孤儿文件、分类不清）
5. 提出修复建议（迁移、合并、删除）

注意：
- 不确定是否该删除的文件，标记为"待确认"而不是直接删除
- 涉及内容迁移时，先确认目标目录存在
- 完成后更新对应INDEX
```

**垃圾文件清理标准**

以下类型的文件应定期清理：

| 类型 | 示例 | 清理标准 | 处理方式 |
|------|------|---------|---------|
| **完结项目的临时文件** | `temp_*.py`、`debug_*.log` | 项目已结项且无后续计划 | 删除 |
| **重复的PDF/DOCX** | `文档_v1.pdf`、`文档_v2.pdf` 内容相同 | 保留最新版本 | 删除旧版本 |

| **过期的临时脚本** | `一次性的转换脚本.py` | 任务完成后 | 删除 |

**不宜清理**：
- 有历史参考价值的旧版本文档
- 未确认是否重复的文件（先确认再处理）
- 仍在活跃使用的参考文档

## 五、项目迭代方法论

> 以下以某限量商品抢购工具的迭代过程为例，展示"想法→验证→优化"的迭代方法论。V1→V2→V3 的技术细节详见百科「迭代详解：V1/V2/V3技术细节」。

### 迭代逻辑

```
V1 验证可行性 → V2 识别瓶颈局部替换 → V3 全链路优化
```

每次迭代的驱动力是明确的性能瓶颈，不是预设的完美方案。优化方向来自开发者对业务的理解，AI 负责验证可行性并给出实现路径。

**核心原则**：

| 版本 | 目标 | 驱动 | 关键决策 |
|------|------|------|---------|
| **V1** | 验证可行性 | 先跑起来 | CDP连接+线程分离 |
| **V2** | 识别瓶颈 | 瓶颈驱动 | 半API化（付款走API） |
| **V3** | 全链路优化 | 架构升级 | 全API化+伏击/拦截模式 |

**迭代演进**：

- V1 验证端到端流程可行性，跑通全DOM操作
- V2 识别付款环节等待DOM渲染是瓶颈，改为半API化
- V3 进一步思考：既然付款可以走API，加购为什么不行？实现全API化

V1→V2→V3 的完整技术细节（架构图、关键技术决策、性能数据），详见百科「迭代详解：V1/V2/V3技术细节」。

### 各阶段问题与解决

以下以某项目V1→V2→V3的实际迭代为例，展示协作各阶段（想法→需求→方案→落地→迭代）中遇到的问题和解决方式。问题本身是项目专属的，但协作模式（如何提问、AI如何分析、如何落地）是通用的。

#### V1 阶段问题

| 问题 | 如何提问 | AI 如何改良 | 落地结果 |
|------|---------|------------|---------|
| 平台检测自动化工具 | "用自动化框架打开网站被反爬拦截了，有反爬机制" | 提出 CDP 方案：连接本地浏览器而非创建新浏览器 | 绕过检测 |
| GUI 界面卡死 | "程序启动后 GUI 窗口卡住不动了" | 识别事件循环冲突，自动化框架移到独立线程 | 界面正常响应 |
| 自动化框架 timeout 单位错误 | "设置 timeout=5 但实际等了很久" | 指出 timeout 单位是毫秒不是秒 | timeout 正确生效 |
| JS click 不触发框架事件 | "JS 点击了按钮但页面没反应" | 改用原生 click 而非 evaluate | 事件正确触发 |
| CDP 连接用 localhost 失败 | "CDP 连接 localhost 超时" | 改用 127.0.0.1，避免 IPv6 解析问题 | 连接成功 |

#### V2 阶段问题

| 问题 | 如何提问 | AI 如何改良 | 落地结果 |
|------|---------|------------|---------|
| 付款太慢 | "付款环节是瓶颈，本质是 HTTP 请求，能直接调 API 吗？" | 抓包分析确认 API 可达，输出半 API 化方案 | 付款毫秒级完成 |
| 找不到商品 ID | "HTML 里没有商品 ID，怎么获取 SKU？" | 发现 SKU 在 GraphQL 响应中，写网络拦截器提取 | 获取到 SKU |
| API 请求被拒 | "付款 API 返回 Token 无效" | 识别 CSRF Token 每次刷新，改为实时提取 | 请求通过 |
| GraphQL 请求格式不对 | "API 返回 400 错误" | 分析请求格式，发现需批量 JSON 数组格式 | 请求格式正确 |
| 限定商品和普通商品是不同 SKU | "搜索到的商品加了购但付款金额不对" | 区分限定商品 SKU 和普通商品 SKU | 选对 SKU |

#### V3 阶段问题

| 问题 | 如何提问 | AI 如何改良 | 落地结果 |
|------|---------|------------|---------|
| 加购也依赖 DOM | "付款走 API 了，加购能不能也走？" | 分析加购接口，设计全 API 流程 | 加购不依赖页面 |
| 提前不知道 SKU | "能不能开售前拿到 SKU，到点直接请求？" | 设计伏击模式：预取 SKU + T-0 直接请求 | 零延迟启动 |
| 时间不精确 | "本地和服务器可能有偏差" | API 请求测时间差，自动校准 + 手动微调 | 偏移量可控 |
| 改源码不生效 | "改了代码但程序跑的还是旧逻辑" | 定位到编译缓存优先级高于源码文件 | 开发阶段不使用缓存 |
| 双重偏移叠加 | "设了 -0.1s 但实际跑 -0.3s" | 发现 config 和源码各有一个偏移量，叠加生效 | 统一为单一入口 |
| VPN导致闪退 | "开了VPN就崩，关了正常" | 4 次方案均失败，识别为系统级网络劫持 | 回滚，标注环境限制 |
| 打包后缺模块 | "打包后报 No module" | 打包工具扫不到懒加载模块 | 显式声明隐藏模块 |

## 六、开发基础设施：云电脑与代码管理

Agent 不只是文本输出器——它有云电脑可以执行命令、有 GitHub 操作能力、有浏览器可以访问网站。理解这些能力怎么用、边界在哪，是从"跟 AI 聊天"到"跟 AI 协作开发"的关键跳跃。

### 6.1 云电脑：Agent 的执行环境

Agent 有一台远程 Linux 云电脑，通过 bash 工具直接执行命令。大多数 Agent 平台都提供云电脑或远程执行环境（如 Coze/扣子 的高阶订阅、Dify 的云端部署等），你也可以选择自部署开源框架（如 OpenClaw）。

没有云电脑的 Agent 只能在对话中输出文本/代码，有了云电脑才能实际执行。

> **⚠️ 安全提醒**：不要用主力电脑部署 Agent。使用云电脑或独立机器隔离，确保 Agent 只能操作项目目录，不影响你的操作系统和个人数据。

**选择执行方式的判断**：

| 你想做什么 | 让 Agent 怎么做 |
|-----------|---------------|
| 改代码/写代码 | 直接说，Agent 在云电脑改 |
| 推仓库/建仓库 | 直接说，Agent 用 bash 操作 Git |
| 访问海外网站抓数据 | 说"用浏览器访问 xxx"，Agent 自动用 agent-browser |
| 打包成 EXE | 你自己在本地跑 build 脚本 |
| 测试 GUI 界面 | 你自己运行，有问题截图给 Agent |
| 安装 Python 包 | Agent 在云电脑装，装了就一直在 |

### 6.2 代码管理：记不住命令不是问题

这是 Agent 协作开发最实际的场景之一——你不需要记住 Git 命令。

**场景：我改了东西但不知道怎么推上去**

你只需要说：

```
帮我把改动推到 v3.0.4 分支
```

Agent 自动执行：

```bash
git add .
git commit -m "fix: 具体修改内容"
git push origin v3.0.4
```

你不需要知道 `git add`、`git commit`、`git push` 的顺序和参数。Agent 知道。

**场景：我想建一个新仓库**

```
帮我建个新仓库叫 my-project-lite，私有仓库
```

Agent 自动：GitHub API 创建仓库 → clone 到云电脑 → 初始化项目结构。

**场景：我改了仓库名/想改名**

```
把仓库名改成 my-project-lite
```

Agent 通过 GitHub API 修改，不用你去找设置页面。

**场景：看看最近改了什么**

```
帮我看看最近的提交记录
```

Agent 执行 `git log`，整理成你能看懂的列表返回。

**场景：回退到某个版本**

```
我觉得最近这次提交有问题，回退到上一个版本
```

Agent 执行 `git reset` 或 `git revert`，选择合适的方式回退。

**关键原则**：你描述意图，Agent 选命令。不需要记语法，不需要记参数，只需要说清楚你想做什么。

### 6.3 多分支管理

**场景：从主分支创建开发分支**

```
从 main 创建一个新分支叫 v2.0，推到远程
```

Agent 执行：

```bash
git checkout main
git pull origin main
git checkout -b v2.0
git push origin v2.0
```

**场景：在不同分支间切换修改**

```
切换到 v3.0.4 分支，把 browser.py 里的超时时间改成 3000
```

Agent 执行 `git checkout v3.0.4` → 修改文件 → commit → push。

**场景：合并分支**

```
把 v2.0 分支合并到 main
```

Agent 执行：

```bash
git checkout main
git merge v2.0
git push origin main
```

**场景：Force Push 后本地同步**

这是项目里踩过的坑——Agent force push 后，用户本地的 `git pull` 会报错（历史不一致）。

正确做法（Agent 会提醒你）：

```powershell
# PowerShell
git fetch origin
git reset --hard origin/分支名
```

这个命令的意思是：不管本地有什么，强制和远程保持一致。Force push 后必须这样做，否则本地和远程永远不同步。

**项目实例**：V3 开发过程中，一次 commit 需要回退重做，Agent force push 后提醒用户执行 `git fetch + git reset`。这条规则后来写入了 TOOLS.md，确保以后每次 force push 都提醒。

### 6.4 远程调试工作流

Agent 在云端改代码，你在本地跑程序。这是最常见的协作模式。

**完整流程**：

```
你在本地跑程序发现问题
→ 截图或复制报错信息发给 Agent
→ Agent 分析问题、定位根因、修改代码
→ Agent push 到远程仓库
→ 你在本地 git fetch + git reset 同步代码
→ 重新运行测试
→ 告诉 Agent 结果
```

**场景：打包后运行报错**

用户说：

```
build 完启动报错
[附截图：No module named 'logging.handlers']
```

Agent 定位：PyInstaller 静态扫描遗漏了懒加载模块。

修改 build 脚本，加 `--hidden-import=logging.handlers`，commit 并 push。

用户本地同步后重新打包，问题解决。

**场景：改了代码但运行结果没变**

用户说：

```
我改了 config.py 里的超时时间，但程序还是用的旧的
```

Agent 定位：编译缓存（.pyd 文件）优先级高于 .py 源码。改了源码但没删 .pyd = 没改。

方案：开发阶段删掉所有 .pyd 文件，只在发布时编译。

**场景：多文件关联修改**

用户说：

```
把 API 模式删掉，只保留 DOM 点击模式
```

Agent 不仅删 `api_client.py`，还检查：
- `main.py` 里的 API 分支代码
- `config.py` 里的 API 相关常量
- `gui_config.py` 里的输入模式选择
- `build` 脚本里的 Cython 编译步骤
- 入口文件的命名

改一个模块忘改关联模块，是最常见的返工原因。Agent 的优势在于能全局搜索、一次性改完。

### 6.5 "帮我看看"类场景

有些操作你自己能做但懒得做，或者不确定怎么做，让 Agent 帮你看。

**场景：检查代码里的某个模式**

```
帮我看看代码里有没有其他地方还在用 SKU_ID 这个变量
```

Agent 执行 `grep -rn "SKU_ID" .`，返回所有引用位置。

**场景：分析依赖关系**

```
browser.py 里 import 了哪些东西，哪些可以删掉
```

Agent 读取文件，分析每个 import 的实际使用情况。

**场景：对比两个版本的差异**

```
帮我看看 v3.0.3 和 v3.0.4 之间改了什么
```

Agent 执行 `git diff v3.0.3..v3.0.4`，整理成人类可读的变更列表。

**场景：检查仓库状态**

```
当前仓库有几个分支，最新提交是什么
```

Agent 执行 `git branch -a` + `git log`，返回当前状态。

### 6.6 环境隔离：理解 Agent 和你不在同一台电脑上

这是新手最容易混淆的点。

```
┌─────────────────┐          ┌─────────────────┐
│   你的电脑        │          │   Agent 云电脑    │
│                 │          │                 │
│  运行打包程序     │   代码    │  修改源代码       │
│  GUI 测试       │ ←同步→   │  Git 操作        │
│  截图/日志       │  通过     │  文件生成        │
│  实际使用        │ GitHub   │  搜索/分析       │
└─────────────────┘          └─────────────────┘
```

**关键理解**：

1. **代码通过 GitHub 同步**——这是唯一可靠的桥梁。Agent 改了代码推到 GitHub，你拉下来测试。你改了代码推到 GitHub，Agent 拉下来继续工作。

2. **文件通过对话传递**——你上传截图/日志给 Agent，Agent 生成文档/图片通过对话发给你。

3. **Agent 无法看到你的屏幕**——GUI 问题请截图，报错请复制文字。说"按钮点不动"Agent 只能猜，说"点击后报错 AttributeError: 'NoneType'"Agent 秒定位。

4. **Agent 的文件系统是自己的**——你本地的文件不会出现在云电脑上。需要 Agent 处理的文件，要么推到仓库，要么通过对话上传。

### 6.7 避雷清单

| ❌ 禁止 | ✅ 正确 |
|--------|--------|
| 说"帮我在我电脑上跑一下" | 说"帮我在云电脑上跑 xxx 命令" |
| 说"程序崩了"不给信息 | 截图+报错文字+触发条件一起给 |
| force push 后 git pull | `git fetch + git reset --hard` |
| Token 写在 MEMORY/TOOLS | Token 存 SECRET.md |
| 让 Agent 改一点测一点 | 攒齐需求→确认→一次性改完→push→测 |
| 改了代码忘删编译缓存 | 开发阶段删掉 .pyd 和 __pycache__ |
| 不告诉 Agent 当前在哪个分支 | Agent 通过仓库状态判断，但不确定时主动说 |

### 6.8 成本管理

每一次 Agent 操作都有成本。理解成本结构，才能做出高效的人机协作决策。

**token 消耗意识**

每次 Agent 读写文件、调用工具、进行搜索，都在消耗 token。

| 操作类型 | token 消耗量 | 优化空间 |
|---------|------------|---------|
| 读取文件 | 与文件大小正相关 | 大文件分批读取 |
| 写入文件 | 与内容长度正相关 | 精简表达 |
| 联网搜索 | 中等，取决于结果量 | 搜过的结果写入文件 |
| 工具调用 | 每次调用都有开销 | 合并多次操作为一次 |
| 对话上下文 | 与历史消息累积相关 | 及时归档，减少冗余 |

**高效交互策略**

用更少的 token 做更多的事。

| 策略 | 做法 | 效果 |
|------|------|------|
| 精简记忆文件 | 活跃文件 ≤15KB | 减少读取开销 |
| 分批读取大文件 | 用 offset+limit | 按需加载，不全量读取 |
| 避免全量加载 | 通过 INDEX 定位 | 只读需要的 |
| 搜索结果写入文件 | 避免重复搜索 | 下次直接读 |
| 精简对话表达 | 一句能说清不用两句 | 降低上下文累积速度 |

**搜索成本**

联网搜索每次都消耗 token。合理使用搜索：

| 场景 | 是否搜索 | 说明 |
|------|---------|------|
| 已有文件中有答案 | 不搜索 | 直接读取文件 |
| 刚搜过的信息 | 不搜索 | 上次结果还在 |
| 确实不知道的信息 | 搜索 | 搜完写入文件备用 |
| 快速事实确认 | 搜索 | 但只搜需要的部分 |

**搜索后必做**：把搜索结果的关键信息写入文件，下次直接读而不是再搜一遍。

**子 agent 成本**

每个 sub-agent 独立消耗 token。无谓的重复 spawn 是浪费：

| 问题 | 后果 | 正确做法 |
|------|------|---------|
| 任务拆得太细 | 多个 sub-agent 通信开销大 | 合并小任务为一个子任务 |
| 重复派发相同任务 | 浪费 token | 先检查是否已有结果 |
| sub-agent 间无共享 | 每个都独立加载上下文 | 在任务描述中明确共享信息 |
| 派发后不管结果 | 超时/卡死浪费资源 | 监控状态，及时 abort |

**性价比思维**

不同任务用不同方式，匹配性价比：

| 任务类型 | 推荐方式 | 原因 |
|---------|---------|------|
| 简单问答/确认 | 直接回复 | 不需要工具调用 |
| 文档生成 | 直接生成 | 不需要 sub-agent |
| 复杂数据分析 | sub-agent | 并行处理效率高 |
| 定时重复任务 | Calendar | 一次创建多次执行 |
| 定期检查 | Heartbeat | 轻量轮询不占资源 |

**性价比原则**：简单任务直接做，复杂任务才启动复杂流程。不要用大炮打蚊子，也不要用蚊子拍打大象。

**成本监控意识**

培养成本意识，但不追求绝对最小化。合理的目标是：

- 避免无谓浪费（重复搜索、重复加载）
- 保持活跃文件精简
- 搜索结果及时归档
- 选择合适的执行方式

**本节与其他章节的关联**

- 2.5 上下文管理：token 消耗的核心就是上下文占用
- 4.2 优化思路：精简记忆文件是成本管理的核心
- 6.7 避雷清单：某些避雷项同时也是节省成本的措施

成本管理不是抠门，是高效协作的体现。花在低价值操作上的每一分 token，都是从高价值推理中偷来的预算。

### 6.9 隔离环境原则

**Agent 拥有 bash 执行权限，可以读写删除文件、安装卸载软件、修改系统配置。** 这意味着它既能高效完成任务，也可能造成严重破坏——误删文件、改乱配置、安装不安全的软件。一个有执行能力的 Agent 不应该运行在你的主力电脑上。

**隔离是安全底线**

不要把 Agent 部署在日常使用的电脑或服务器上。使用云电脑或虚拟机隔离，确保 Agent 只能操作项目目录，不影响你的操作系统和个人数据。

| 场景 | 风险 | 隔离措施 |
|------|------|---------|
| 主力电脑直接运行 Agent | 误删文件、改坏配置、感染恶意软件 | 强制使用云电脑或虚拟机 |
| Agent 误执行危险命令 | `rm -rf`、格式化磁盘 | 隔离环境只影响虚拟磁盘 |
| Agent 安装不安全的软件 | 后门、挖矿程序 | 隔离环境可随时重建 |

**为什么隔离是必要的**

即使 Agent 有工作规则约束，也无法完全避免以下情况：

- Agent 误解指令，执行了错误的文件操作
- Agent 的子任务（sub-agent）没有正确继承安全约束
- 项目代码本身有问题（如依赖恶意包），导致连锁反应

隔离环境确保这些风险被控制在有限范围内，不会影响你的核心工作环境。

**典型案例**

某开发者在主力电脑上运行 Agent，Agent 执行了一个清理脚本想删除项目中的 node_modules，结果命令写错变成 `rm -rf ./*`，整个项目目录被清空。虽然代码有 Git 备份，但未提交的修改全部丢失。如果在隔离的云电脑中操作，这类事故的损失可以完全避免。

**实施建议**

1. 购买或申请云电脑服务，设置独立的项目工作目录
2. 将项目代码与云电脑绑定，所有 Agent 操作在云端完成
3. 定期将代码同步到本地，使用 Git 管理版本

## 七、踩坑速查

| 坑 | 根因 | 一句话方案 |
|----|------|-----------|
| 改源码不生效 | 编译缓存优先级高于源码文件 | 开发阶段不使用缓存，发布时才编译 |
| 配置值与预期不符 | 多处定义同一参数，叠加生效 | 核心参数单一入口 |
| VPN导致崩溃 | 系统级网络劫持，应用层无法绕过 | 标注环境限制，非代码问题 |
| GUI 框架行为异常 | 老旧框架 API 与直觉不符 | 逐项验证，不假设一致 |
| 打包后缺模块 | 打包工具扫不到懒加载模块 | 显式声明隐藏模块 |
| AI 未经确认就改代码 | 缺少确认机制 | 结构性改动必须审批 |
| sub-agent 幻觉导致代码改错 | 上下文缺失 + 不验证就采信 | 技术结论必须交叉验证，"没找到" ≠ "不存在"，详见百科3.10 |
| "理解后重写"导致选择器错误 | 没搞清参考代码实际走哪条路径就搬 | 先追踪代码路径，从实际执行的函数里直接复制，详见百科3.10 |
| AI 静态分析推断选择器废弃 | CSS/JS bundle 中没找到 ≠ 不存在 | 必须用动态 DOM 验证，参考 3.19.2 反向验证法 |
| AI 跳步执行未经确认 | 把"解释"等同于"修复"，省略确认环节 | 停→回退→补环节→确认→继续，参考 3.20 |
| AI 反复犯同一错误 | harness 写了但执行前不自检 | 记忆锚点+自检清单+违反强制回滚，参考 3.20 |
| 对话太长 AI 遗忘关键信息 | 上下文窗口有限，早期信息被截断 | 关键信息及时写入文件，参考 2.5 上下文管理 |
| sub-agent 结果质量不达标 | 上下文缺失或任务描述不完整 | 验收后用 sessions_send 补充要求，不亲自上手，见 3.10.1 |
| 技能加载了但没理解就执行 | 只读了技能名没读 SKILL.md | skill_load 后必须读 SKILL.md 理解能力，见 3.9.2 |
| Agent 重复搜索同一信息 | 没把搜索结果写入记忆 | 搜索结果写入文件，下次直接读取 |
| 长期项目换对话后状态丢失 | 只在对话中讨论没写文件 | 关键状态写入 MEMORY.md，参考 4.6 |
| token 消耗过快 | 记忆文件过大+频繁全量读取 | 精简文件+分批读取+按需搜索，参考 6.8 |
| force push 后本地不同步 | git pull 会报错历史不一致 | `git fetch + git reset --hard` |
| 改一个模块忘改关联模块 | 缺少全局搜索 | 让 Agent grep 全局检查引用 |
| Agent 无法访问目标网站 | 云电脑网络限制 | 改用 agent-browser |

项目专属踩坑的完整排查过程（如编译缓存、VPN崩溃等），详见百科「踩坑详解：项目专属踩坑排查过程」。

## 总结

| 阶段 | AI 协作重点 | 人的角色 |
|------|------------|---------|
| 想法→需求 | 梳理：拆解痛点为结构化需求 | 定义业务边界和约束 |
| 需求→方案 | 分析：可行性验证、方案对比 | 判断方向、做取舍 |

核心原则：**人提方向，AI 出方案；人做测试，AI 做分析；人掌舵，AI 执行。**

## 如何开始

看完这份文档后，你可能觉得方法论很完整，但不知道从哪下手。以下是一个从零开始的行动指南。

### 第一步：选择一个你想做的项目

哪怕很模糊也没关系。比如：
- "我想做一个自动抢票工具"
- "我想做个个人网站"
- "我想把某个重复工作自动化"

**重要的是先有一个起点**，而不是一开始就追求完美的方案。

### 第二步：按"想法→需求"阶段模式描述你的场景

用这套提问结构告诉 Agent：

```
我遇到一个业务场景：
- 什么类型的网站/系统
- 用户操作流程是什么
- 核心痛点是什么
- 有哪些已知约束（时间、环境、技术栈）

请帮我分析：
1. 技术可行性
2. 主要风险点
3. 建议的技术方向
```

让 Agent 帮你梳理需求，而不是你自己先想清楚所有细节。

### 第三步：建立基础文件结构

```
你的项目/
├── SOUL.md        # Agent 的性格和行为规则
├── USER.md        # 你的偏好和背景
├── MEMORY.md      # 项目当前状态和上下文
├── TOOLS.md       # 工作规则和避坑经验
└── SECRET.md      # 敏感信息（token、密码等）
```

一开始不用写得很完整，核心原则是：**Agent 告诉你的重要结论，写进文件。**

### 第四步：跑起来再说，先 V1 后 V2

不要追求一开始就设计完美的架构：
- V1 的目标是验证可行性，跑通流程
- V2 识别瓶颈，针对性优化
- V3 全面改进

**每一步都有价值，先跑起来比什么都重要。**

### 第五步：当记忆文件超过 15KB 时，开始建知识库

如果 TOOLS.md 或 MEMORY.md 越来越臃肿：
1. 创建 `知识库/` 目录，按主题分类
2. 把历史内容、详细技术文档迁移到知识库
3. 创建 INDEX.md 作为导航入口
4. 活跃文件只保留当前有效的信息

提示词：
```
请帮我整理 TOOLS.md 和 MEMORY.md：
1. 已过时的 harness 和已解决的问题移到知识库归档
2. 只保留当前有效的 harness 和 workflow
3. 创建知识库目录结构，建立 INDEX.md 导航
4. 确保活跃文件合计不超过 15KB
```

### 第六步：当协作出现问题时，回看 workflow 章节

常见问题对应的解决方案：

| 问题 | 回看章节 |
|------|---------|
| Agent 改了不该改的 | 3.15 确认机制 |
| Agent 在死胡同里死磕 | 3.16 止损机制 |
| Agent 不知道文件在哪 | 4.5 管理原则 + INDEX |
| Agent 给出过时方案 | 4.2 知识库优化 |
| 同一个问题反复出现 | TOOLS.md 写入 harness |

### 常见误区

| ❌ 误区 | ✅ 正确 |
|--------|--------|
| 一开始就追求完美架构 | 先跑起来，架构在迭代中演进 |
| 跳过"确认"环节 | 结构性改动必须先说明方案 |
| 让 Agent 在死胡同里死磕 | 设定试错上限，4 次失败就止损 |
| 重要信息只存在对话中 | 文件即记忆，写进去才能保持 |
| 不更新记忆文件 | 每次对话后补充新结论到 MEMORY |
| 全部放一个文件 | 按主题分类，保持文件精简 |

### 记住核心原则

> **人提方向，AI 出方案；人做测试，AI 做分析；人掌舵，AI 执行。**

Agent 是你的协作伙伴，不是你的替代品。它的优势是速度和信息处理，你的优势是判断和验收。分工明确，效率翻倍。

## FAQ

### Agent 遗忘了之前讨论的内容怎么办？

关键信息必须写入文件（MEMORY.md / TOOLS.md），对话上下文有限，不写入文件的内容约等于没告诉 Agent。如果已经遗忘，让 Agent 读取对应文件恢复上下文。

### sub-agent 执行结果不理想怎么办？

不要自己重做。用 sessions_send 向 sub-agent 补充要求和上下文，让它继续完善。详见 3.10.1 子agent异常处理。

### 记忆文件越来越大怎么办？

执行记忆文件优化流程（见 4.2 优化思路）：过时内容归档知识库，活跃文件 ≤15KB，建 INDEX 导航。

### Agent 总是跳过确认环节怎么办？

在工作规则文件中写入硬规则："所有写/改/删文件操作必须等用户确认"，并设置记忆锚点。详见 3.15 确认机制和 3.20 Workflow 遗忘防范。

### 怎么判断该用 sub-agent、Calendar 还是 Heartbeat？

看 3.13 三种执行方式选择指南。简单说：实时任务用 sub-agent，定时任务用 Calendar，轻量周期检查用 Heartbeat。

### Agent 给出的技术方案靠谱吗？

技术结论必须交叉验证。"没找到"≠"不存在"，静态分析结论必须动态验证。详见 3.19 幻觉识别与防范体系。

### 多个技能可以同时使用吗？

可以。复杂任务可同时加载多个技能，按需组合。但要注意技能间的依赖关系。详见 3.9 技能全生命周期管理。

### 怎么控制 token 消耗？

精简记忆文件、避免全量加载、分批读取大文件、按需搜索而非重复搜索。详见 6.8 成本管理。

### 长期项目换对话后状态丢失怎么办？

每次对话后把项目状态写入 MEMORY.md，换对话时 Agent 读取 MEMORY 恢复上下文。详见 4.6 长期项目的跨 session 追踪。

### 如何快速定位某个问题该看哪一节？

参考"如何开始"第六步的问题-章节对照表，或者直接搜索关键词。方法论的设计是模块化的，每个问题都有对应的解决方案。

## 附录：TOOLS.md参考范例

以下为某项目的实际TOOLS.md参考范例，经脱敏处理。项目特定名称已替换为通用标识（如"项目A/B"），核心结构和规则保留完整。

```markdown
# 工具使用技巧与工作流

## 一、铁律区（每次操作必过，按顺序检查）

### 1. 合规自检
□ 我给出方案了吗？
□ 我自查方案了吗？（遗漏？矛盾？副作用？）
□ 用户确认了吗？
□ 我是在确认之后才动手的吗？
□ 用户问的是"解释/原因"吗？如果是，我不应该动手改
□ 我是否在"顺便"做用户没要求的事？如果是，停下来

### 2. 方案先行
- **用户给方向≠授权直接动手**。"方向明确"只是需求清楚了，不代表跳过确认环节
- 即使是"小改动"，也要先给方案（改哪、怎么改、措辞是什么）→ 确认 → 再动手
- 跳过确认的口子一开就会越改越大

### 3. 验收+交付
- 验收时双向检查：正向（该做的做了没）+反向（不该出的问题出了没），详细清单见第七区
- **每一次** edit_file/write_file 成功后，下一个动作就是用 computer:// 协议引用给用户看，无例外
- 不要等用户问，改完就引

### 4. 副本+逐级验收
- **所有修改走副本**：先cp原文件→副本，在副本上改，逐级验收后再替换原文件
- 流程：cp原文件→副本 → 改副本 → 主agent验收 → 用户验收 → 确认后替换原文件
- 子agent产出 → 主agent验收 → 不通过则返工，通过则交给用户
- 主agent产出 → 用户验收 → 不通过则修改，通过则上线
- 例外：简单局部修改（<5行）且精确指定了old_string/new_string的，可直接改

### 5. 不要自我创造
- 已有能跑的代码直接搬，不要"理解后重写"——重写=引入新bug
- 严格参考现有代码，不能凭想象编造

## 二、子agent规范（派发时必读）

### 防幻觉+派发规范
1. **保留清单**：明确列出哪些内容必须保留在原位，不能删、不能迁移
   - ❌ "精简TOOLS.md" → 子agent不知道什么该留，编路径替换
   - ✅ "精简TOOLS.md，BUG修复/方案先行/交付规则这三节必须原样保留"

2. **迁移约束**：涉及内容迁移时，必须写明目标文件路径、迁移后源文件处理方式，必须先确认目标文件创建成功才能删源内容

3. **验收检查项**：任务描述末尾列出主agent验收时会检查什么，让子agent知道会被检查

4. **禁止自创文件路径**：子agent不能自己编路径然后引用，所有引用路径必须在任务描述中指定或已确认存在

5. **反面教训**：子agent曾把TOOLS.md内容替换成8个不存在的.md文件路径，上下文不足+没校验=幻觉

6. **串行原则**：B的输入依赖A的产出时，必须等A完成再派B，不能并行

7. **指定源文件**：派发时明确写"基于哪个文件改"，不能假设原文件就是最新版（可能有未覆盖的副本）

8. **递进链路**：多级产出有递进关系时（如配置→展开→精简），在任务描述中写明链路，子agent才知道自己的产出会被谁消费

## 三、场景工作流（用到才查）

### BUG修复
1. 用户反馈问题 → 只记录，不改
2. 所有问题凑齐 → 总结+给方案 → **自查方案** → 交给用户确认
3. 确认 → 一次性全改 → 自检代码 → push
4. 用户测试 → 有问题回到第1步
- 严禁边记边改、确认一个改一个push一个
- 不要给pull步骤，用户fetch+reset即可
- force push后必须提醒 `git fetch origin && git reset --hard origin/分支名`
- **⚠️ 所有写/改/删文件操作都必须等用户确认后才能执行**
- **⚠️ 用户问"为什么"时只解释，不要动手改代码！**

### 新功能/需求开发
1. 用户提需求 → 复述理解，确认没偏
2. 给实现方案 → **自查方案** → 交给用户确认
3. 确认 → 建项目目录+INDEX.md → 写代码 → 自检代码 → push
4. 用户测试

### 需求分析与调研
1. 用户提模糊需求/想法 → 先梳理，不要急于给方案
2. 明确三件事：目标是什么、边界在哪、约束条件
3. 给可行性分析 → 交给用户确认方向
4. 方向确认后 → 拆解为可执行任务 → 用户确认拆解 → 派sub-agent执行

**关键**：模糊需求不要直接动手，先问清楚再干。

### 信息整理与输出
1. 明确搜索/采集目标和产出格式
2. 执行搜索/采集 → 整理为结构化内容
3. 给用户看初步结果 → 确认方向
4. 补充完善 → 最终交付 → 重要结论写入知识库

**关键**：先给中间结果确认方向，不要闷头整理完才发现偏了。

### 文件生成类操作
1. 确认内容和格式要求
2. 给方案 → 确认 → 生成 → 验收质量 → 交付
3. 不满意 → 调整 → 重新交付
- 技术细节见 `./知识库/文档/文档处理技巧.md`

### 代码重构/优化
1. 用户提优化需求 → 明确优化目标
2. 给方案，**必须包含风险评估**
3. 用户确认 → 实施 → 自检代码 → 回归测试 → push
4. 用户测试

**关键**：重构最怕改A坏B，方案里必须说明影响范围和测试策略。

### 配置文件修改（TOOLS/MEMORY等自身调整）
1. 明确改什么、为什么改
2. 给方案 → **必须包含权重检查**：改的内容应该放在哪个区？放的位置是否影响其他条目的优先级？
3. 用户确认 → 走副本修改 → 验收 → 覆盖原文件
4. 同步更新方法论（完整版→入门手册）的对应章节和范例

**关键**：同样内容放在铁律区和放在附录区，对行为的影响完全不同。改内容不改位置=权重失调。

## 七、验收标准（交付前必查）

### 自查方案（给用户确认之前）
1. 需求每条都覆盖了吗？
2. 改了A，B要不要跟着改？
3. 有没有副作用？
4. 隐含假设什么时候不成立？

### 自检代码（push之前）
1. 语法/import/功能是否正确？
2. 代码和方案一致吗？
3. 崩溃点在哪？异常路径怎么办？
4. 数据流追踪：变量从哪来？变了没？

### 验收产出（检查任何交付物）
**正向验证（需求→产出）：沿着需求方向走，确认"该做的做了没"**
1. **覆盖完整性** — 逐条对照原始需求，每条是否有对应产出？
2. **意图准确性** — 产出是否真正满足了需求意图，还是字面糊弄？（用户说"加个说明"，加了一句废话=字面覆盖但意图没满足）
3. **产出可用性** — 产出能不能实际工作？文档能不能读？代码能不能跑？不只是"看起来有了"

**反向挑刺（产出→问题）：逆着产出方向挑，找"不该出的问题出了没"**
4. **逻辑审计** — 因果是否成立？概念是否矛盾？上下文是否衔接？
5. **边界完整性** — 序号连续？引用有效？命名一致？标题变了内容有没有跟着变？
6. **副作用扫描** — 改A是否意外影响B？删一段是否连带信息丢失？
7. **合适性判断** — 这么做是否合适？有没有更好的方式？是否过度设计或不足设计？
8. **交叉验证** — 用grep/脚本扫描，不靠肉眼

**交付引用** — 验收通过后必须立即用 computer:// 协议引用给用户，无例外
```
