# 双轨审查工作流 · 最终版

> 流程要轻到能记住，严到不出大问题。

---

## 风险分级

| 等级 | 判断标准 | 处理方式 |
|------|---------|---------|
| 🟢 低 | 日常对话、简单查询、改文案 | 主Agent直接做 |
| 🟡 中 | 改几行代码、短文档(<500行) | 主Agent做 + 自己grep抽检 |
| 🔴 高 | 长文档(>500行)、教程、核心代码、零基础受众 | 走完整流程 |

拿不准就按🔴高走。

---

## 3个硬约束（不做就别往下推）

### 硬约束1：约束源必须主Agent验证

素材库或设计文档是整条链的根基，根基错了后面全白跑。

- **主Agent抽检验证，不能委托给sub-agent**
- 抽检按风险优先级：标注行号的必查、引用罕见文件名的必查、看起来"太完美"的必查
- 发现错误时先诊断：系统性问题（某个文件完全没读到）→ 废弃重建；个别错误 → 修补可用
- 失败率>20%且属系统性问题 → 废弃重建

### 硬约束2：写产物必须基于约束源

- 改代码：只从素材库取代码，素材库没有的就不许用
- 从0开发：只按设计文档实现，文档没写的不许加
- 写文档：事实性内容只从素材库取，标注来源

### 硬约束3：P0未清零不能交付

问题分级：
- **P0**：编造/错误/偏离需求 → 不修不能交付
- **P1**：可读性/完整性/体验问题 → 建议修
- **P2**：锦上添花 → 可选

判定规则：
- 有未修复P0 → ❌ 打回
- P0清零且有3+个P1 → ⚠️ 有条件通过，告知用户P1清单由用户决定
- P0清零且P1≤2 → ✅ 通过

## 2个建议（做了更好）

### 建议1：并行跑审查

- 代码：Auditor（审计）+ Runner（运行验证）
- 文档：Auditor（事实审计）+ Reader（目标读者视角）

### 建议2：3轮不过就停下来

3轮还过不了 → 停下来，给用户诊断报告：
- 根因判断（约束源问题 / 需求不清 / 任务太复杂）
- 建议（重建素材库 / 重新对齐需求 / 拆分子任务）

## 完整流程

### 代码工作流

```
改代码模式（已有项目）：

① Agent建素材库 → 主Agent风险优先级抽检 → 通过/重建
② Proposer编码（只从素材库取）
③ 并行：Auditor（真实性+正确性）+ Runner（运行验证）
④ Grader：合并问题清单 → P0/P1/P2分级 → 给总分（参考）
   → Grader只仲裁"同一问题的矛盾结论"，不同维度的不同结论直接合并
⑤ 判定（P0清零？）
⑥ 主Agent关键路径验证 + grep抽检
⑦ 轮次硬限制3轮

从0开发模式（新项目）：

① 主Agent写设计文档
   → 含：功能点清单 + 明确不做的事 + 关键决策点
   → 主Agent用自己的话复述核心需求，让用户发现理解偏差
② Proposer按设计文档编码
③~⑦ 同上
```

### 文档工作流

```
① Agent建素材库 → 主Agent风险优先级抽检 → 通过/重建
② Proposer写文档（事实性内容只从素材库取）
③ 并行：
   - Auditor：抽样10-15个事实回源验证 + 结构完整性
   - Reader：完整阅读，输出卡点清单（非分数）
     格式："第X章第Y段：[原文] → 卡在[具体概念/术语/跳跃]"
④ Grader：合并问题 → P0/P1/P2分级 → 给总分（参考）
⑤ 判定（P0清零？）
⑥ 主Agent grep抽检关键事实
⑦ 轮次硬限制3轮
```

## 角色职责

| 角色 | 职责 | 关键原则 |
|------|------|---------|
| **主Agent** | 验证约束源、分配任务、最终抽检、交付 | 约束源验证不能委托 |
| **Proposer_A** | 写产物 | 只从约束源取材 |
| **Auditor_B** | 审查真实性和正确性 | 不接触Proposer思维链 |
| **Reader_D** | 模拟目标读者完整阅读 | 输出卡点清单，不输出分数 |
| **Runner_E** | 运行代码验证 | 跑不起来=自动P0 |
| **Grader_C** | 问题分级+辅助评分 | 只仲裁同一问题的矛盾结论 |

## 评分维度（Grader参考用，不是通过判据）

### 代码

| 维度 | 权重 |
|------|------|
| 真实性/设计符合度 | 30% |
| 逻辑正确性 | 25% |
| 可运行性 | 25% |
| 代码质量 | 20% |

### 文档（按类型调整）

| 维度 | 教程 | 方案 | 报告 |
|------|------|------|------|
| 事实准确性 | 25% | 35% | 40% |
| 可读性 | 30% | 15% | 15% |
| 结构完整性 | 15% | 30% | 30% |
| 概念准确性 | 20% | 10% | 10% |
| 可执行性 | 10% | 10% | 5% |

评分辅助判定但不单独决定通过与否，通过判据是P0清零。

---

## 修订规则

- Grader问题分级按**章节标注**，Proposer只改有问题的章节，不碰没问题的
- 第2轮起只修P0，P1留给用户决定
- 每轮修订后主Agent做grep抽检确认P0确实修了

---

## 什么时候不用跑流程

- 你明确知道答案的简单问题
- 改颜色/文案等低误伤操作
- 用户等不了的情况（先交付再补查）
- 探索性对话，没有明确产物
