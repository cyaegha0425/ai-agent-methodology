# 执行方式详细指南

## 4.1 sub-agent（子代理）

### 定义
主agent派出的独立执行单元，拥有相同工具、文件、技能，异步执行，不阻塞主对话。

### 用途
即时执行一次性任务，结果秒级返回。

### 完整链路
主agent编写任务描述 → 派发 → 子agent执行 → 结果返回 → 验收

### 任务描述5要素
1. **背景**：为什么做这件事
2. **当前进展**：项目/任务现在到什么阶段了
3. **关键文件位置**：涉及哪些文件，在哪
4. **产出要求**：要交付什么，格式要求
5. **结果目标**：验收标准是什么

### 验收管理
主agent先验收，再交给用户，不符合则追加要求。

## 4.2 Calendar（日程/定时任务）

### Calendar 是什么
Agent的定时任务工单系统。到设定时间，系统唤起独立session读取工单并执行。

### Calendar 能做什么
- 一次性提醒："明天下午1点半开会，提前10分钟提醒"
- 定期执行："每天下午查一下黄金价格"
- 重复追踪："持续追踪AI行业新闻"
- 分步推进长期项目：拆解目标，分步创建日程

### 适合什么场景
- 定时提醒（提前10分钟提醒）
- 定期执行（有明确产出，如"查价格并整理发送"）
- 时间敏感任务（设具体时间）
- 需要重复执行的检查/报告任务

### 不适合什么
- 需要秒级响应的检查（频率不够）
- 需要生成大文件的复杂任务（用sub-agent更合适）
- 不严格、随手一看的检查（用Heartbeat）

### 创建Calendar时的注意事项
| 字段 | 要求 | 示例 |
|------|------|------|
| summary | 用清晰动作短语，主语是Agent自己 | "同步黄金价格" ✅<br>"提醒我黄金价格" ❌ |
| description | 写给执行session看的工单详情，必须包含完整上下文（目标、步骤、文件位置） | 因为执行session不了解对话历史 |
| time_range | 弹性调度窗口，设置后系统选择空闲时段执行，避开整点高峰 | 除时间严格敏感的任务外都应设置 |
| rrule | 重复规则。最稳妥写法只填freq/interval/count/until，让dtstart推导时间 | 不要额外填byhour/byminute/byday/bymonthday，除非用户明确要求 |

### rrule避雷
不要填与dtstart冲突的字段，否则会导致创建失败。

### 修改重复规则
直接update即可，系统自动保留历史实例、终止旧日程、创建新的未来日程。

### Calendar与sub-agent的区别
| 对比维度 | sub-agent | Calendar |
|----------|-----------|----------|
| 执行时机 | "现在就做" | "到时候做" |
| 结果返回 | 秒级返回 | 需要等触发时间 |
| 适用类型 | 一次性任务 | 定期重复任务 |

### 提示词示例
```
帮我每天下午3点查一下XX价格，整理后发给我
```
```
明天早上9点提醒我开会
```

## 4.3 Heartbeat（心跳/周期轮询）

### Heartbeat 是什么
每30分钟触发一次的轻量检查机制。

### 工作方式
每次触发读取heartbeat.md中的检查清单，逐项快速检查，有事通知无事沉默。

### 适合什么场景
- 检查后台任务/sub-agent是否完成或卡住
- 回复其他渠道消息（如群聊消息、文档评论）
- 时间要求不严格的提醒（久坐提醒、晚间休息）
- 监控类检查（股价触及阈值时提醒）

### 不适合什么
- 需要生成文件或报告的任务（用Calendar）
- 有明确执行时间要求的任务（用Calendar）
- 耗时较长的数据分析（用Calendar）
- 需要高于30分钟频率的检查（用Calendar创建重复日程）

### Heartbeat与Calendar的区别
| 对比维度 | Heartbeat | Calendar |
|----------|-----------|----------|
| 定位 | "顺手看一眼" | "按时干活" |
| 耗时 | 秒级完成 | 有明确产出，可能耗时 |
| 产出 | 无事沉默，不产文件 | 必须有产出，可能耗时 |
| 频率 | 固定30分钟 | 可自定义 |

### heartbeat.md格式要求
```
检查清单
- 检查爬虫任务是否正常运行，卡住则通知用户
- 看一眼飞书群是否有需要回复的消息
- 用户久坐超过2小时，提醒活动一下
- XX股价低于300时提醒用户
```
首行必须是"检查清单"纯文本标题，之后用`- `列表。

### 使用工具
`heartbeat_md_read` / `heartbeat_md_update` / `heartbeat_md_delete`

> 严禁直接用文件工具操作heartbeat.md

## 4.4 三种执行方式选择指南

| 场景 | 选什么 | 原因 |
|------|--------|------|
| 用户要求现在就做一件事 | sub-agent | 即时执行，结果秒回 |
| 每天固定时间做某事 | Calendar | 定时触发，重复执行 |
| 定期追踪某个话题 | Calendar | 有明确产出，用topic_tracking技能 |
| 30分钟看一眼状态 | Heartbeat | 轻量检查，无事不扰 |
| 生成文件/报告 | Calendar或sub-agent | Heartbeat不适合 |
| 抢购类精确时间任务 | Calendar | 时间敏感，设具体时间 |
| 久坐提醒/休息提醒 | Heartbeat | 不严格，30分钟粒度够 |

## 避雷清单

| ❌ 禁止 | ✅ 正确 |
|--------|--------|
| 用Heartbeat做需要生成文件的任务 | 用Calendar或sub-agent |
| 用Calendar做需要秒级频率的检查 | 考虑其他方案 |
| 用sub-agent做需要等到特定时间的任务 | 用Calendar |
| Heartbeat放太多检查项 | 每项都要30分钟内跑完，保持轻量 |
| Calendar的description写太简略 | 执行session不了解对话历史，必须写完整 |
| Calendar的rrule填与dtstart冲突的字段 | 只填freq/interval/count/until |
