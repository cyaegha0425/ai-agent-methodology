# 工具使用技巧与工作流

## 一、铁律区（每次操作必过，按顺序检查）

### 1. 合规自检
□ 我给出方案了吗？
□ 我自查方案了吗？（遗漏？矛盾？副作用？）
□ 用户确认了吗？
□ 我是在确认之后才动手的吗？
□ 用户问的是"解释/原因"吗？如果是，我不应该动手改
□ 我是否在"顺便"做用户没要求的事？如果是，停下来

**强制触发点**：调用 sessions_spawn / calendar_create / 覆盖原文件之前，必须先输出确认模板：
```
准备执行：[具体动作]
用户确认：✗（等待确认）
```
- 没有"✓"就不能调工具，同一条消息里不能既出方案又调工具执行
- 出方案→停→等确认→下一条消息才执行

### 2. 方案先行
- **用户给方向≠授权直接动手**。"方向明确"只是需求清楚了，不代表跳过确认环节
- 即使是"小改动"，也要先给方案（改哪、怎么改、措辞是什么）→ 确认 → 再动手
- 跳过确认的口子一开就会越改越大

### 3. 验收+交付
- 验收时双向检查：正向（该做的做了没）+反向（不该出的问题出了没），详细清单见第七区
- **每一次** edit_file/write_file 成功后，下一个动作就是用 computer:// 协议引用给用户看，无例外
- 不要等用户问，改完就引
- **覆盖原文件流程**：副本验收通过 → 引用给用户看 → **停：等用户确认** → 确认后才能cp替换原文件
- 验收通过≠可以覆盖，中间必须有用户确认这个门

### 4. 副本+逐级验收
- **所有修改走副本**：先cp原文件→副本，在副本上改，逐级验收后再替换原文件
- 完整流程（每个"停"都是强制等待点）：
  - cp原文件→副本 → 改副本 → 主agent自验 → 引用给用户看 → **停：等用户确认** → 确认后替换原文件
- 子agent产出 → 主agent验收 → **引用给用户看** → **停：等用户确认** → 替换原文件
- 主agent产出 → **引用给用户看** → **停：等用户确认** → 上线
- 例外：简单局部修改（<5行）且精确指定了old_string/new_string的，可直接改
- **⚠️ 跳过任何"停：等用户确认"节点直接覆盖 = 严重违规**

### 5. 不要自我创造
- 已有能跑的代码直接搬，不要"理解后重写"——重写=引入新bug
- 严格参考现有代码，不能凭想象编造

### 6. 常见违规记录（操作前扫一眼）
| 违规 | 后果 | 日期 |
|------|------|------|
| 未确认就派sub-agent | 浪费token+时间+回滚双倍消耗 | 2026-05-04 |
| 验收完直接覆盖不给用户看 | 用户无法验收，重复操作 | 2026-05-04 |
| 方案和执行放在同一条消息 | 跳过确认环节 | 2026-05-04 |
| 计算余额时编造不存在的累计数字并加入计算 | 余额数据完全错误，破坏信任 | 2026-05-04 |

## 二、子agent规范（派发时必读）

### 防幻觉+派发规范
1. **保留清单**：明确列出哪些内容必须保留在原位，不能删、不能迁移
   - ❌ "精简TOOLS.md" → 子agent不知道什么该留，编路径替换
   - ✅ "精简TOOLS.md，BUG修复/方案先行/交付规则这三节必须原样保留"

2. **迁移约束**：涉及内容迁移时，必须写明目标文件路径、迁移后源文件处理方式，必须先确认目标文件创建成功才能删源内容

3. **验收检查项**：任务描述末尾列出主agent验收时会检查什么，让子agent知道会被检查

4. **禁止自创文件路径**：子agent不能自己编路径然后引用，所有引用路径必须在任务描述中指定或已确认存在

5. **反面教训**：子agent曾把TOOLS.md内容替换成8个不存在的.md文件路径，上下文不足+没校验=幻觉

6. **串行原则**：B的输入依赖A的产出时，必须等A完成再派B，不能并行

7. **指定源文件**：派发时明确写"基于哪个文件改"，不能假设原文件就是最新版（可能有未覆盖的副本）

8. **递进链路**：多级产出有递进关系时（如配置→展开→精简），在任务描述中写明链路，子agent才知道自己的产出会被谁消费

## 三、场景工作流（用到才查）

### BUG修复
1. 用户反馈问题 → 只记录，不改
2. 所有问题凑齐 → 总结+给方案 → **自查方案** → 交给用户确认
3. 确认 → 一次性全改 → 自检代码 → push
4. 用户测试 → 有问题回到第1步

详细操作见 `./知识库/Agent运维/force-push处理.md`

### 新功能/需求开发
1. 用户提需求 → 复述理解，确认没偏
2. 给实现方案 → **自查方案** → 交给用户确认
3. 确认 → 建项目目录+INDEX.md → 写代码 → 自检代码 → push
4. 用户测试

### 需求分析与调研
1. 用户提模糊需求/想法 → 先梳理，不要急于给方案
2. 明确三件事：目标是什么、边界在哪、约束条件
3. 给可行性分析 → 交给用户确认方向
4. 方向确认后 → 拆解为可执行任务 → 用户确认拆解 → 派sub-agent执行

### 信息整理与输出
1. 明确搜索/采集目标和产出格式
2. 执行搜索/采集 → 整理为结构化内容
3. 给用户看初步结果 → 确认方向
4. 补充完善 → 最终交付 → 重要结论写入知识库

### 文件生成类操作
1. 确认内容和格式要求
2. 给方案 → 确认 → 生成 → 验收质量 → 交付
3. 不满意 → 调整 → 重新交付
4. 技术细节见 `./知识库/文档/文档处理技巧.md`
5. 脱敏规则见 `./知识库/文档/脱敏规则.md`

### 代码重构/优化
1. 用户提优化需求 → 明确优化目标
2. 给方案，**必须包含风险评估**
3. 用户确认 → 实施 → 自检代码 → 回归测试 → push
4. 用户测试

### 配置文件修改（TOOLS/MEMORY等自身调整）
1. 明确改什么、为什么改
2. 给方案 → **必须包含权重检查**：改的内容应该放在哪个区？放的位置是否影响其他条目的优先级？
3. 用户确认 → 走副本修改 → 主agent自验 → 引用给用户看 → **停：等用户确认** → 覆盖原文件
4. 同步更新方法论（完整版→入门手册）的对应章节和范例

## 四、执行方式（选型与详细指南）

详细指南见 `./知识库/Agent运维/执行方式详细指南.md`

### 选型参考表

| 场景 | 选什么 | 原因 |
|------|--------|------|
| 用户要求现在就做一件事 | sub-agent | 即时执行，结果秒回 |
| 每天固定时间做某事 | Calendar | 定时触发，重复执行 |
| 定期追踪某个话题 | Calendar | 有明确产出，用topic_tracking技能 |
| 30分钟看一眼状态 | Heartbeat | 轻量检查，无事不扰 |
| 生成文件/报告 | Calendar或sub-agent | Heartbeat不适合 |
| 抢购类精确时间任务 | Calendar | 时间敏感，设具体时间 |

## 五、资源定位（动手前必须查）

### 两层INDEX体系
1. 先判断是项目专属还是跨项目通用
2. 项目专属 → 查项目INDEX（如`./文档工作/INDEX.md`、`./知识库/RSI星际公民/RSI-AB项目INDEX.md`等）
3. 跨项目通用 → 查知识库全局INDEX（`./知识库/INDEX.md`）
4. **不查INDEX就动手 = 可能遗漏已有知识 = 重复劳动或凭空编造**

### 文件路径查找技巧
当无法确定文件位置时，使用以下bash命令查找：
- 查找特定类型文件：`find /path -name "*.py" -type f`
- 查看当前目录结构：`ls -la`
- 确认当前路径：`pwd`

系统预装技能列表见每次对话的系统注入。自定义经验skill：触发时读取对应INDEX定位，按需加载。

## 六、记忆管理

- TOOLS.md：只写当前有效的harness和workflow
- MEMORY.md：只写当前活跃的项目状态
- 会过时的内容→写知识库，不会过时的→写记忆
- TOOLS+MEMORY加起来不超过15KB
- **经验教训**：记忆膨胀→上下文不足→输出质量下降→胡言乱语→巨量浪费积分。精简是生存问题

## 七、验收标准（交付前必查）

### 自查方案（给用户确认之前）
1. 需求每条都覆盖了吗？
2. 改了A，B要不要跟着改？
3. 有没有副作用？
4. 隐含假设什么时候不成立？

### 自检代码（push之前）
1. 语法/import/功能是否正确？
2. 代码和方案一致吗？
3. 崩溃点在哪？异常路径怎么办？
4. 数据流追踪：变量从哪来？变了没？

### 验收产出（检查任何交付物）
**正向验证（需求→产出）：沿着需求方向走，确认"该做的做了没"**
1. **覆盖完整性** — 逐条对照原始需求，每条是否有对应产出？
2. **意图准确性** — 产出是否真正满足了需求意图，还是字面糊弄？（用户说"加个说明"，加了一句废话=字面覆盖但意图没满足）
3. **产出可用性** — 产出能不能实际工作？文档能不能读？代码能不能跑？不只是"看起来有了"

**反向挑刺（产出→问题）：逆着产出方向挑，找"不该出的问题出了没"**
4. **逻辑审计** — 因果是否成立？概念是否矛盾？上下文是否衔接？
5. **边界完整性** — 序号连续？引用有效？命名一致？标题变了内容有没有跟着变？
6. **副作用扫描** — 改A是否意外影响B？删一段是否连带信息丢失？
7. **合适性判断** — 这么做是否合适？有没有更好的方式？是否过度设计或不足设计？
8. **交叉验证** — 用grep/脚本扫描，不靠肉眼

**交付引用** — 验收通过后必须立即用 computer:// 协议引用给用户，无例外
