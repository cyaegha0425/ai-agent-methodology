## 1. 核心概念

先搞懂3件事：Agent是什么、记忆怎么管、什么时候用什么执行方式。

### 1.1 Agent vs 普通AI：3个本质区别

普通AI（比如ChatGPT）是你问一句它答一句，这次聊完下次就忘了。Agent不一样，它是你长期的"项目搭档"。

| 区别 | 普通AI | Agent |
|------|--------|-------|
| **记忆** | 对话结束就忘 | 写在文件里，跨对话记得 |
| **主动性** | 等你问 | 能自己规划任务 |
| **持续性** | 每次从零开始 | 跨对话保持上下文 |

还有一个关键点：普通AI只能输出文字，Agent有云电脑能实际执行命令、读写文件、操作Git。

### 1.2 文件即记忆

跟Agent协作最重要的原则：**不写进文件 = 没说过**。

对话里的结论关了窗口就没了，写进文件才叫"记住了"。这个原则贯穿所有协作场景。

举个例子：Agent帮你分析了一个技术方案，你点头说"可以，就这么办"。结果下次换了个对话窗口，Agent完全不记得这件事——因为它没有写进MEMORY.md。

### 1.3 记忆文件职责

| 文件 | 干什么的 | 典型内容 |
|------|---------|---------|
| **SOUL.md** | Agent的性格设定 | 它该怎么称呼你、说话什么风格 |
| **USER.md** | 你的偏好 | 暗色主题、喜欢简洁、深夜不打扰 |
| **MEMORY.md** | 项目当前状态 | 做到哪了、在做什么、下一步计划 |
| **TOOLS.md** | 避坑经验和工作流程 | 之前踩过的坑、workflow规则 |
| **SECRET.md** | 敏感信息 | token、密码、API密钥 |
| **知识库/** | 历史归档 | MEMORY太满时把旧内容挪过来 |
| **INDEX.md** | 导航入口 | 知识库里有哪些文件、路径是什么 |

### 1.4 三种执行方式

| 方式 | 什么时候用 | 怎么触发 | 适合什么 |
|------|-----------|---------|---------|
| **sub-agent** | 现在就要做 | 主Agent直接派出去 | 一次性任务、并行处理 |
| **Calendar** | 定时/重复做 | 设置时间自动触发 | 每天查价格、每周生成报告 |
| **Heartbeat** | 定期看一眼 | 每30分钟自动检查 | 看后台任务状态、回消息 |

**选择原则**：
- 要**现在**做 → sub-agent
- 要**到点**做 → Calendar
- 要**顺手看一眼** → Heartbeat

### 1.5 Agent收到消息后怎么工作的

Agent收到你的消息后，会按这个顺序处理：

```
收到消息 
→ 读取SOUL.md（确认性格和行为规则）
→ 读取USER.md（了解你的偏好）
→ 读取MEMORY.md（恢复项目上下文）
→ 读取TOOLS.md（获取工作规则）
→ 判断任务类型
→ 决定自己执行还是派sub-agent
→ 执行
→ 结果验收
→ 更新记忆文件
```

所以如果你的MEMORY.md是空的，Agent就不知道项目进展到哪了；如果TOOLS.md没写规则，Agent就按自己想法来，容易踩坑。

## 2. 快速开始

5步上手，看完就能开始干。

### 第一步：选一个项目

哪怕很模糊也行：
- "我想做个自动抢票工具"
- "我想做个个人网站"
- "我想把日报自动生成"

有个起点比什么都重要。先跑起来再说完美方案。

### 第二步：描述你的场景

告诉Agent这个格式：

```
我遇到一个业务场景：
- 什么类型的系统/网站
- 用户操作流程是什么
- 核心痛点是什么
- 有哪些已知约束（时间、环境、技术栈）

请帮我分析：
1. 技术可行性
2. 主要风险点
3. 建议的技术方向
```

让Agent帮你把模糊想法变成清晰需求，而不是你自己先把所有细节想清楚。

### 第三步：建立基础文件结构

至少创建这5个文件：

```
你的项目/
├── SOUL.md        # Agent的性格和行为规则
├── USER.md        # 你的偏好和背景
├── MEMORY.md      # 项目当前状态和上下文
├── TOOLS.md       # 工作规则和避坑经验
└── SECRET.md      # 敏感信息（token、密码等）
```

一开始不用写得很完整。核心原则：**Agent告诉你的重要结论，写进文件**。

### 第四步：先V1后V2，不要追求一步到位

不要一上来就设计完美架构。迭代逻辑是：

- **V1**：验证可行性，跑通流程
- **V2**：识别瓶颈，针对性优化
- **V3**：全面改进

每一步都有价值。先跑起来比什么都重要。

### 第五步：MEMORY超过15KB就建知识库

如果TOOLS.md或MEMORY.md越来越臃肿，执行优化：

1. 创建 `知识库/` 目录，按主题分类
2. 把历史内容、详细技术文档迁移到知识库
3. 创建 INDEX.md 作为导航入口
4. 活跃文件只保留当前有效的信息

**优化目标**：TOOLS.md ≤5KB，MEMORY.md ≤3.5KB，活跃文件合计 ≤15KB。

## 3. 核心工作流

6个常用场景的精简流程。**每步都要确认再执行**，这是最核心的规则。

### 3.1 需求分析与调研

适用于：想法很模糊、不确定能不能做、需要先调研的场景。

```
1. 描述模糊想法
2. → Agent梳理需求：目标是什么、边界在哪、约束条件
3. → 你确认目标和边界
4. → Agent做可行性分析
5. → 你确认方向
6. → Agent拆解为可执行任务
7. → 你确认拆解
8. → 按任务执行
```

**关键**：模糊需求不要直接动手，先问清楚再干。

### 3.2 新功能/需求开发

适用于：需求已经明确，需要从零开发新功能。

```
1. Agent复述理解 → 你确认没偏
2. → Agent给实现方案（架构、模块、技术选型）
3. → Agent自查方案
4. → 你确认方案
5. → Agent编码实现
6. → Agent自检代码
7. → 提交
8. → 你测试
```

**关键**：理解偏差1度，执行偏差180度。方案确认环节不能省。

### 3.3 信息整理与输出

适用于：搜索/采集信息、整理资料、生成报告。

```
1. 明确采集目标和产出格式
2. → Agent执行搜索/采集
3. → Agent整理为结构化内容
4. → 你看初步结果确认方向
5. → Agent补充完善
6. → 交付
7. → 归档到知识库
```

**关键**：闷头整理完才发现偏了=白干。先给中间结果确认方向。

### 3.4 文件生成

适用于：生成文档、报告、PPT等文件产出。

```
1. 你说明内容、格式、风格要求
2. → Agent输出文档结构和大纲
3. → 你确认方向和结构
4. → Agent生成完整文件
5. → Agent自检验收
6. → 交付
7. → 不满意则调整
```

**关键**：大纲确认5分钟，全文返工30分钟。

### 3.5 BUG修复

适用于：修复已知的错误和问题。

```
1. 完整描述现象、触发条件、与预期的差异
2. → Agent分析根因，给出修复方案
3. → Agent自查方案
4. → 你确认方案
5. → Agent执行修改
6. → Agent自检代码
7. → 提交
8. → 你测试验证
```

**BUG修复的沟通技巧**：
- ❌ "程序崩了"
- ✅ "开启VPN后闪退，关闭VPN正常运行"
- ❌ "按钮点不动"
- ✅ "点击后无响应，但日志显示click事件已触发"

信息越精确，定位越快。

### 3.6 代码重构/优化

适用于：性能优化、架构调整、代码清理。

```
1. 明确优化目标（性能？可维护性？架构？）
2. → Agent给方案（含风险评估：改什么、怎么改、可能改坏什么）
3. → 你确认方案
4. → Agent实施修改
5. → Agent回归测试
6. → 提交
7. → 你测试
```

**关键**：重构最怕改A坏B。方案里必须有风险评估，没有风险评估的方案=不完整。

## 4. 记忆管理

### 4.1 文件职责速查

| 文件 | 写什么 | 不写什么 |
|------|--------|---------|
| SOUL.md | Agent性格、称呼、行为规则 | 项目信息 |
| USER.md | 你的偏好、背景、习惯 | 项目信息 |
| MEMORY.md | 活跃项目状态、待办、关键决策 | 历史归档、详细技术文档 |
| TOOLS.md | 当前有效的harness、workflow | 已解决的问题、过期信息 |
| 知识库/ | 历史归档、参考文档 | 活跃项目状态 |
| INDEX.md | 资源路径和摘要 | 具体内容 |

### 4.2 3条核心管理原则

**原则1：文件即记忆**
不写进文件 = 没告诉Agent。对话结束后Agent还能记得的，只有文件里的内容。

**原则2：活跃文件≤15KB**
大了就归档到知识库。超出则上下文溢出风险大，Agent读取时被截断，做出不完整决策。

**原则3：修改后同步INDEX**
INDEX过时 = Agent被误导。Agent通过INDEX定位资源，INDEX和实际文件不一致，Agent就找不到或找到错的。

### 4.3 什么放记忆、什么放知识库？

**放MEMORY/TOOLS（活跃文件）**：
- 当前要用的
- 经常变的
- 下次对话需要知道的

**放知识库（归档）**：
- 历史记录
- 详细技术文档
- 已解决的踩坑记录

**判断标准**：
- 当前活跃项目状态 → MEMORY.md
- 当前有效的避坑经验 → TOOLS.md
- 历史详细内容 → 知识库/

### 4.4 MEMORY.md的标准结构（长期项目）

如果项目要持续追踪，MEMORY.md应该包含：

```
## 项目状态

### 当前阶段
V3 开发中，第二轮优化

### 已完成
- ✅ 功能A
- ✅ 功能B

### 进行中
- 错误处理逻辑优化

### 待办
- [ ] 日志系统完善
- [ ] GUI优化

### 阻塞项
- VPN环境限制，暂不解决

### 关键决策
- 选择器用.itemCard，不走fallback函数
- API模式为默认，DOM模式为fallback

### 最近进展
- 2024-01-15: 完成某某模块
```

### 4.5 交接清单

换对话或换Agent时必须传递的信息：
- 项目阶段（当前在哪个阶段）
- 未完成工作（正在做什么、待做什么）
- 关键决策（重要结论和理由）
- 踩坑记录（已知问题和规避方案）
- 相关文件（所有资源的路径）

## 5. 避坑 Top 10

这是最常见的10个错误，提前知道能省很多返工时间。

| # | 错误描述 | 正确做法 |
|---|---------|---------|
| 1 | 改完代码发现没生效 | 检查是否有编译缓存，删掉.pyc和__pycache__ |
| 2 | Agent改了不该改的 | 结构性改动必须先给方案等你确认再执行 |
| 3 | 同一个问题反复出现 | 把教训写进TOOLS.md的harness，下次Agent会记得 |
| 4 | 换了对话Agent就失忆 | 每次对话后更新MEMORY.md |
| 5 | AI说"没找到"就以为不存在 | "没找到"≠"不存在"，必须实际跑一下验证 |
| 6 | Agent跳步执行未经确认 | 直接说"停"，让它补上确认环节 |
| 7 | 凭想象重写参考代码 | 先追踪实际走哪个函数，从那个函数直接复制，不要"理解后重写" |
| 8 | 改了A忘了B | 让Agent grep全局检查所有引用 |
| 9 | sub-agent结果不达标 | 用sessions_send补充要求，别自己重做 |
| 10 | token消耗太快 | 精简记忆文件，避免重复搜索同一信息 |

### 补充5个常见坑

| # | 错误描述 | 正确做法 |
|---|---------|---------|
| 11 | force push后git pull报错 | 用 `git fetch + git reset --hard` |
| 12 | 配置值与预期不符 | 检查是否有多处定义，叠加生效了 |
| 13 | 长期项目换对话后状态丢失 | MEMORY里写清楚项目状态 |
| 14 | 技能加载了但效果不好 | skill_load后必须读SKILL.md |
| 15 | AI在死胡同里死磕 | 设定试错上限，4次失败就换方向 |

## 6. FAQ

### Q1: Agent遗忘了之前讨论的内容怎么办？

**A:** 不写进文件 = 没说过。以后让Agent做重要结论时，主动说"把这个写进MEMORY.md"。如果已经忘了，让Agent读取对应文件恢复上下文。MEMORY.md是你的项目"存档点"。

---

### Q2: sub-agent执行结果不理想怎么办？

**A:** 不要自己重做。用sessions_send向sub-agent补充要求，让它继续改。亲自上手会破坏协作闭环——sub-agent不知道哪里错了，下次还会犯。记住：**让它改，不自己做**。

---

### Q3: 记忆文件越来越大怎么办？

**A:** 当TOOLS.md或MEMORY.md超过15KB时，执行优化：
1. 把过时内容迁移到知识库，按主题分类
2. 建INDEX.md做导航
3. 活跃文件只留当前有效的

核心目标：TOOLS.md ≤5KB，MEMORY.md ≤3.5KB。

---

### Q4: Agent总是跳过确认环节怎么办？

**A:** 两个办法：
1. 在TOOLS.md写硬规则："所有写/改/删文件操作必须等用户确认"
2. 发现跳步时直接说"停"，让它补上确认环节

长期解决方案是把这条规则写进TOOLS.md，Agent每次唤醒都会读到。

---

### Q5: 怎么判断该用sub-agent、Calendar还是Heartbeat？

**A:** 简单判断：
- **sub-agent**：现在就要做，用"马上干"
- **Calendar**：到点/重复做，用"到点干"
- **Heartbeat**：顺手看一眼，用"定期看一眼"

复杂任务用sub-agent，定时报告用Calendar，检查状态用Heartbeat。

---

### Q6: AI给出的技术方案靠谱吗？

**A:** 需要验证。几个容易出问题的模式：
- "在源码里没找到X" → 不等于X不存在，可能是编译产物
- "V1就是这么写的" → 要确认V1实际走的是哪个函数
- "这个选择器已废弃" → 用旧选择器跑一下，能用就没废

原则：**技术结论必须能用已有代码或可验证证据交叉验证**。

---

### Q7: 怎么控制token消耗？

**A:** 几个实用技巧：
1. 活跃文件≤15KB，精简MEMORY和TOOLS
2. 大文件分批读取，用offset+limit
3. 搜索结果写进文件，避免重复搜索
4. 同一信息只在一个地方定义，避免冲突

---

## 核心原则速记

> **人提方向，AI出方案；人做测试，AI做分析；人掌舵，AI执行。**

Agent是你的搭档，不是你的替代品。它的优势是速度和执行力，你的优势是判断和验收。分工明确，效率翻倍。
