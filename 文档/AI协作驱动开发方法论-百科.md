## 3.10 详解：sub-agent派发与幻觉案例
（对应方法论 3.10）

### 派发规范详解

**串行原则**

B的输入依赖A的产出时，必须等A完成再派B，不能并行。

```
✅ 正确：派A分析页面结构 → 等A完成 → 派B基于A的分析结果写代码
❌ 错误：同时派A和B → A还没分析完B就开始写 → B基于错误假设工作
```

**指定源文件**

派发时明确写"基于哪个文件改"，不能假设原文件就是最新版。

```
✅ 正确：基于 ./项目A/src/browser.py 的最新代码改，不要基于备份副本
❌ 错误：基于"当前代码"改 → 可能有未覆盖的副本导致版本混乱
```

**递进链路**

多级产出有递进关系时（如配置→展开→精简），在任务描述中写明链路。

```
场景：配置TOOLS.md → 展开成方法论完整版 → 精简成入门手册
任务描述应该写明：
- 你的产出会被谁消费（下一个环节是谁）
- 产出格式要求（配置→完整版→精简版的格式差异）
- 链路位置（你是第几个环节）
```

### 复合幻觉完整案例

**案例背景**：主Agent派子Agent分析目标平台的页面结构，为项目A的精简版确定DOM选择器。这次任务产生了两次严重幻觉，最终导致代码被错误修改。

**幻觉一：选择器"已废弃"推断**

| 环节 | 事实 |
|------|------|
| 起因 | 子Agent被派去分析目标平台的编译后CSS/JS文件，提取页面元素的class名 |
| 幻觉推论 | 子Agent在编译产物中"没找到旧选择器`.product-card`"，直接得出"已废弃，新选择器是`.ProductWidget / .ItemCard__cta`"的结论 |
| 真实情况 | "没找到旧class" ≠ "旧class已废弃"——前端框架可以同时给元素多个class、CSS可以分chunk加载、编译产物不一定包含所有class定义 |
| 证伪 | 项目A用`.product-card`选择器实际运行正常，成功完成流程 |
| 根本原因 | **静态源码分析 ≠ 动态DOM分析**，子Agent用错误的方法论得出了看似合理的结论 |

**幻觉二：流程"很简单"推断**

| 环节 | 事实 |
|------|------|
| 起因 | 子Agent读取项目A版本代码，了解结账流程 |
| 幻觉推论 | 子Agent只读了备用简化函数（几步就完成），没读主流程函数（包含地址确认、协议勾选等多步骤），就得出"结账就这么简单"的结论 |
| 真实情况 | 项目A有两套结账逻辑：主流程函数（分步结账）和备用简化函数（直接提交，仅fallback） |
| 证伪 | 用户用项目A成功付款，流程包含地址确认和协议勾选；精简版按简化逻辑改后，页面不推进导致流程失败 |
| 根本原因 | **读参考代码时只看了一个版本就下了结论**，没有确认哪个是主模式 |

**复合错误**：两个幻觉互相"印证"——先误以为平台改版（所以旧选择器不能用），又误以为结账很简单（所以不需要分步），两个错误叠加导致精简版完全偏离正确方向。

**这个案例印证了什么**：

1. **上下文管理是生死问题**：子Agent没有对话历史，任务描述是它唯一的上下文
2. **文件即记忆**：幻觉发生时，中间产物留在了文件系统中，必须追查来源
3. **验证比结论重要**：当现有代码用相同选择器能正常工作时，"改版"结论一定是假的
4. **读代码必须读全**：有多个模式时必须搞清哪个是主模式

### "理解后重写"陷阱完整案例

**案例背景**：在上面的复合幻觉事故之后，开发者明确要求"按项目A的实际代码来写"。主Agent读取了项目A代码，确认了项目A用`.product-card`选择器找卡片。然而精简版再次运行时，依然找不到卡片。

**调试过程**：开发者在页面上按F12检查，发现实际卡片class是`.itemCard`，而`.product-card`命中为0。但项目A**现在仍能正常购买**——同一页面，项目A能找到，精简版找不到。

**根因**：主Agent看到项目A代码里有`.product-card`，就以为"项目A用这个选择器"。但项目A默认走API模式，实际调用的是另一个函数`add_to_cart_via_page()`，那个函数用的是`.itemCard`。项目A代码里确实有`.product-card`——但那是fallback函数里的，项目A几乎不会走到那条路径。

**错误模式**：

| 步骤 | 主Agent做了什么 | 应该做什么 |
|------|---------------|-----------|
| 1 | 看到项目A代码里有`.product-card` | 先搞清项目A实际走哪个函数 |
| 2 | 以为"项目A就是用这个选择器" | 追踪项目A的实际代码路径 |
| 3 | 把`.product-card`搬进精简版 | 从项目A实际走的函数里搬选择器`.itemCard` |

**本质**："理解后重写" ≠ "复制后适配"。看懂了代码的逻辑，然后用自己的方式重新实现，看似省事，实则引入偏差。

**Harness规则**：

1. **已有能跑的代码直接搬，不要"理解后重写"**——重写=引入新bug
2. **先确认代码路径再搬**：有多个模式时，必须搞清实际走的是哪个函数
3. **选择器必须有出处**：任何DOM选择器，必须能在参考代码的对应函数中找到原文
4. **调试优先**：改完必须加调试日志，第一次运行就能定位问题

- 教训 → 写入TOOLS.md的harness
- 经验 → 固化为workflow
- 踩坑记录 → 归档知识库，供后续项目参考

## 3.21 详解：邮件分身配置
（对应方法论 3.21）

### 完整配置示例

邮件分身的处理规则通过 `EMAIL_RULES.md` 定义。一个典型的规则文件结构：

```markdown
## 规则优先级（数字越小优先级越高）

### 1. 钓鱼邮件拦截
- 触发条件：[根据你的场景定义，如：发件人在黑名单 OR 邮件内容包含可疑关键词]
- 操作：[定义你的处理逻辑，如：移动到垃圾箱]
- 优先级：10（最高）

### 2. 验证码自动处理
- 触发条件：[根据你的场景定义，如：邮件主题包含验证码关键词]
- 操作：[定义你的处理逻辑，如：提取验证码→追加到文件]
- 优先级：50

### 3. 金额/余额追踪
- 触发条件：[根据你的场景定义，如：邮件主题包含确认关键词且包含金额]
- 操作：[定义你的处理逻辑，如：提取金额→累加到追踪文件]
- 优先级：100

### 4. 订单状态追踪
- 触发条件：[根据你的场景定义]
- 操作：[定义你的处理逻辑]
- 优先级：100
```

**注意事项**：
- 邮件分身是常驻能力，一旦配置就会持续运行，除非手动关闭或删除规则
- 规则按优先级顺序匹配，第一个匹配的规则会执行
- 建议先测试规则，确保匹配条件和操作符合预期
- 敏感操作（如删除邮件）建议设置为高优先级，确保优先执行

### 规则定义方法

**通用规则模板**：

```markdown
### [规则名称]
- 触发条件：
  - [条件1] OR
  - [条件2]
- 操作：
  1. [操作步骤1]
  2. [操作步骤2]
- 优先级：[数字越小优先级越高]
```

**占位符说明**：
- `[根据你的场景定义]` → 根据实际项目需求填写具体条件
- `[定义你的处理逻辑]` → 根据实际业务逻辑实现

## 3.22 详解：文件修改安全案例
（对应方法论 3.22）

### TOOLS被替换成8个路径的完整案例

**背景**：主Agent派sub-agent"精简TOOLS.md"，没有给出保留清单。

**发生了什么**：

sub-agent的上下文窗口有限，看到任务描述是"精简"，以为要把内容拆成独立文件。

于是TOOLS.md的内容被替换成了8个不存在的`.md`文件路径：

```
./知识库/子Agent规范.md
./知识库/产出交付.md
./知识库/验收标准.md
./知识库/BUG修复流程.md
./知识库/需求开发流程.md
./知识库/文件操作规范.md
./知识库/沟通确认规则.md
./知识库/避坑经验.md
```

而这些文件根本不存在，原有内容也丢失了。

**根因分析**：

| 问题 | 说明 |
|------|------|
| 任务描述不完整 | 只说了"精简"，没说"保留什么" |
| 上下文不足 | sub-agent看不到原文件的完整结构，只能猜测 |
| 没校验 | 没有在sub-agent完成后检查引用路径是否存在 |

**教训**：告诉sub-agent什么不能动 > 告诉它该做什么。保留清单是安全底线。

**正确做法**：

```
❌ 错误："精简TOOLS.md"
✅ 正确："精简TOOLS.md，铁律区/方案先行/产出交付三节必须原样保留，其余过时内容移到知识库"

❌ 错误："整理知识库"
✅ 正确："整理知识库，项目A的harness移到`知识库/项目A/`目录，禁止删除任何harness"
```

## 4.3 详解：INDEX体系模板
（对应方法论 4.3）

### INDEX体系通用设计思路

INDEX是知识库的导航入口，每个项目/领域都应该有自己的INDEX。

**INDEX核心原则**：
1. **唯一入口**：根目录INDEX.md记录所有项目入口
2. **项目独立INDEX**：每个项目目录有自己的INDEX.md
3. **文件可追溯**：INDEX中引用的每个文件都必须存在
4. **层次清晰**：按主题/项目/时间等维度组织

### 通用INDEX模板骨架

```markdown
# [项目/领域名] INDEX

## 项目概述
- 项目名称：[你的项目名]
- 核心目标：[简述]
- 时间跨度：[日期范围]

---

## 核心文件
- [核心文件1] — [简要说明]
- [核心文件2] — [简要说明]

## 参考文件
- [参考文件1] — [简要说明]
- [参考文件2] — [简要说明]

---

## 关键架构决策
1. [决策点1]
2. [决策点2]

## Harness（必须保留的修复点）
1. [Harness规则1]
2. [Harness规则2]

---

## 相关文件索引
| 文件 | 内容摘要 |
|------|----------|
| [文件1] | [描述] |
| [文件2] | [描述] |
```

### 知识库目录结构设计原则

| 原则 | 说明 |
|------|------|
| 全局INDEX | 根目录必须有INDEX.md，记录所有项目入口 |
| 项目独立INDEX | 每个项目目录有自己的INDEX.md，记录项目内所有文件 |
| 通用分类 | 跨项目的通用内容放在"通用/"下 |
| 类型分组 | 同类型文件放同一目录，便于管理 |

**通用目录结构示例**：

```
知识库/
├── INDEX.md                    # 唯一导航入口
├── [项目A]/
│   ├── INDEX.md               # 项目A完整索引
│   ├── harness.md             # 开发流程与避坑
│   ├── api.md                 # 接口文档
│   └── [其他项目文件]
├── [项目B]/
│   ├── INDEX.md               # 项目B完整索引
│   └── [项目B文件]
├── 通用/
│   ├── 工具技巧/              # 工具使用技巧
│   ├── 文档处理/              # 文档生成规范
│   └── AI协作/                # 人机协作经验
└── skills/
    └── 参考文档/
        ├── harness文件.md
        └── 模板文件.md
```

> **项目专属INDEX详情已迁移到知识库**：
> - 具体文件名列表 → 知识库/项目A/INDEX.md
> - 项目目录结构 → 同上

## 迭代详解：V1/V2/V3技术细节
（对应方法论第五章）

### V1：全DOM操作

**架构**：程序通过浏览器自动化框架控制浏览器，模拟完整用户操作链路——页面导航→元素定位→点击加购→购物车跳转→余额选择→付款确认。

**关键技术决策**：

- **CDP连接真实浏览器**：目标平台有反自动化检测，自动化框架默认浏览器实例带自动化标记，直接被拦截。通过CDP连接本地浏览器，平台看到的是真实环境。
- **线程分离**：GUI主循环和自动化框架同步API都要求占用主线程，同线程死锁。自动化框架放独立线程，通过消息队列与GUI通信。

**端到端耗时**：数秒。瓶颈在于每一步DOM操作都需要等待页面渲染。

### V2：半API化

**瓶颈识别**：付款环节等待DOM渲染是纯浪费——付款本质是HTTP请求，等页面渲染没有意义。

**架构**：加购走DOM + 付款走HTTP API。

**端到端耗时**：1-2秒。SKU获取仍依赖页面渲染，是残留瓶颈。

### V3：全API化

**架构**：GraphQL查询→API加购→API付款，浏览器仅用于Cookie保持。

**新增两种运行模式**：

- **伏击模式**：开售前预取SKU，T-0直接发请求，无页面交互
- **拦截模式**：页面刷新触发GraphQL响应，拦截器实时提取SKU

**时间校准**：通过API请求测量本地与服务器时间差，自动校准+手动微调。

**端到端耗时**：亚秒级，性能提升10x+。

### 技术细节总结

| 版本 | 核心架构 | 关键技术 | 端到端耗时 |
|------|---------|---------|-----------|
| V1 | 全DOM操作 | CDP连接+线程分离 | 数秒 |
| V2 | 半API化 | API付款 | 1-2秒 |
| V3 | 全API化 | 伏击+拦截+时间校准 | 亚秒级 |

## 6.1 详解：云电脑与自部署
（对应方法论 6.1）

### OpenClaw自部署完整指南

**OpenClaw是什么**

OpenClaw是一个开源的AI Agent框架，可以部署到你的独立主机上，实现与平台高阶订阅相同的Agent能力——文件记忆、工具调用、bash执行、多渠道通信。

> **⚠️ 重要原则：不要用主力电脑部署。** Agent拥有执行权限，可以读写文件、运行命令、访问网络。把它部署在日常使用的主力电脑上，等于给你的工作环境开了一个有权限的后门。请使用一台独立的机器（旧笔记本、NUC、树莓派、云服务器均可），将Agent环境与你的日常工作环境隔离。

**部署方式对比**

| 方式 | 适合谁 | 难度 | 成本 |
|------|--------|------|------|
| 云服务器一键部署 | 想快速上手、7×24在线 | ⭐ | 云服务器月费 |
| Docker部署 | 有运维基础、想控制环境 | ⭐⭐ | 云服务器月费 |
| 独立本地机器安装 | 有闲置旧电脑/零月租成本需求 | ⭐ | 硬件成本 |

**方式一：云服务器一键部署**

腾讯云Lighthouse、DigitalOcean等平台提供OpenClaw预装镜像，选择镜像→购买→SSH连接→运行onboard向导，10分钟内完成。

推荐配置：2 vCPU / 4GB RAM / 40GB SSD / Ubuntu 22.04

云服务器的优势是7×24在线、独立环境、不影响本地工作。

**方式二：Docker部署**

```bash
# 拉取镜像
docker pull openclaw/openclaw:latest

# 启动容器（替换YOUR_API_KEY为你的LLM API Key）
docker run -d \
  --name openclaw \
  -p 8080:8080 \
  -e ANTHROPIC_API_KEY=YOUR_API_KEY \
  -v /opt/openclaw/data:/app/data \
  --restart unless-stopped \
  openclaw/openclaw:latest
```

Docker提供额外隔离层，即使Agent执行了异常命令也不会影响宿主机。

**方式三：独立本地机器安装**

```bash
# 安装
npm install -g openclaw@latest

# 运行初始化向导（配置API Key、渠道等）
openclaw onboard

# 启动网关
openclaw gateway start
```

macOS / Linux / Windows (WSL2)均支持。闲置的旧笔记本、Mac Mini、NUC都是不错的选择。

**部署后的文件结构**

```
~/.openclaw/
├── openclaw.json         # 全局配置（模型、渠道）
└── workspace/            # Agent工作空间
    ├── SOUL.md           # Agent人设
    ├── MEMORY.md         # 记忆文件
    ├── TOOLS.md          # 工作规则
    ├── knowledge/        # 知识库
    └── skills/           # 技能目录
```

**能做什么**

- 执行任何终端命令：`pip install`、`python xxx.py`、`git push`
- 读写文件、修改代码
- 安装的依赖持久保存
- 通过agent-browser技能控制浏览器访问网站

**不能做什么**

| 限制 | 说明 | 实例 |
|------|------|------|
| 无法访问部分海外网站 | 云电脑网络限制 | 目标平台需用agent-browser |
| 无法操作用户本地环境 | 云电脑与用户电脑完全隔离 | 打包EXE必须用户自己做 |
| 无法看到用户屏幕 | 无图形界面 | GUI截图验证做不到 |

## 踩坑详解：项目专属踩坑排查过程
（对应方法论第八章踩坑速查）

> **项目专属踩坑详情已迁移到知识库**：
> - 编译缓存/VPN/GUI框架/打包/多重配置/Force Push等 → 知识库/项目A/开发笔记.md
> - 通用踩坑速查见下方

### 通用踩坑速查

以下是跨项目适用的通用踩坑：

| 踩坑类型 | 典型表现 | 排查方向 |
|---------|---------|---------|
| 缓存问题 | 改了代码但不生效 | 检查编译缓存、CDN缓存、浏览器缓存 |
| 环境差异 | 本地正常/线上异常 | 检查Python版本、环境变量、依赖版本 |
| 打包问题 | 打包后缺模块/报错 | 检查hidden-import、打包工具配置 |
| 配置覆盖 | 设了配置但不生效 | 检查多处配置是否叠加生效 |
| 版本冲突 | git pull/push报错 | 检查本地远程历史是否一致 |
| 网络问题 | 开了VPN代理就异常 | 检查系统级网络劫持 |

**通用排查原则**：
1. **环境隔离**：开发/生产环境分离，避免互相影响
2. **单一入口**：核心参数只在一处定义，避免多处覆盖
3. **充分验证**：修改后必须测试，不要假设
4. **止损判断**：识别无解问题后及时止损，避免无效投入

## 附录：通用知识库维护脚本

以下是可以通用的知识库维护脚本：

```bash
#!/bin/bash
echo "=== 知识库完整性检查 ==="

echo "[1/3] 检查INDEX引用的文件是否存在..."
ERRORS=0
grep -rh '\.md' 知识库/INDEX.md 知识库/*/INDEX.md 2>/dev/null | \
  grep -oE '[\./]*\S+\.md' | sort -u | while read f; do
  if [ ! -f "$f" ]; then
    echo "❌ INDEX引用了不存在的文件: $f"
    ERRORS=$((ERRORS+1))
  fi
done

echo "[2/3] 检查实际文件是否都在INDEX中被引用..."
find 知识库 -name "*.md" -type f | while read f; do
  if ! grep -rq "$f" 知识库/INDEX.md 知识库/*/INDEX.md 2>/dev/null; then
    echo "⚠️ 文件未被INDEX引用: $f"
  fi
done

echo "[3/3] 检查项目INDEX是否覆盖项目目录下所有文件..."
for dir in 知识库/*/; do
  if [ -f "${dir}INDEX.md" ]; then
    echo "检查 ${dir}..."
    find "$dir" -maxdepth 1 -name "*.md" -type f | while read f; do
      if ! grep -q "$f" "${dir}INDEX.md"; then
        echo "⚠️ ${dir}目录下文件未被项目INDEX引用: $f"
      fi
    done
  fi
done

echo "=== 检查完成 ==="
if [ $ERRORS -eq 0 ]; then
  echo "✅ 未发现严重问题"
else
  echo "❌ 发现 $ERRORS 个严重问题，请修复"
fi
```
