# RSI项目事故复盘 — 2026-05-04

## 背景
开发RSI MIEMIE LITE时发生两次严重幻觉，导致代码被错误修改、购买失败。

## 幻觉一："RSI改版，.offer-card已废弃，新选择器是.SkuWidget/.BuyingOptionCard__cta"

| 环节 | 事实 |
|------|------|
| 幻觉来源 | 子agent分析RSI pledge store编译后的CSS/JS bundle，从中提取出.SkuWidget等class名 |
| 真实情况 | .SkuWidget等class名可能真实存在于RSI React组件中，但子agent在CSS bundle中"没找到.offer-card"就直接得出"已废弃"结论——这是不支持的反向推论 |
| 证伪 | V1用.offer-card选择器实际运行正常，成功结账付款 |
| 根本原因 | 静态源码分析 ≠ 动态DOM分析。CSS可能分chunk、React元素可以同时有多个class、编译产物不一定包含所有class定义 |
| 教训 | **"源码中没找到"≠"不存在/已废弃"，必须用实际运行验证** |

## 幻觉二："V1的结账就这么简单：直接点Complete Order/Place Order"

| 环节 | 事实 |
|------|------|
| 幻觉来源 | 读取V1 browser.py时只看了run_page_mode（行427，简化版），没看run_api_mode（行180，完整版） |
| 真实情况 | V1有两个函数：run_api_mode（分步结账：Continue→地址→agree→I AGREE，实际主用）和run_page_mode（简化备用：直接Complete Order，仅fallback） |
| 证伪 | 用户用V1成功付款，流程包含地址确认+agree勾选；LITE按简化版改后，地址页面不推进导致购买失败 |
| 根本原因 | 读代码时看到简短版本就以为"V1就是这样的"，没有继续读完整版 |
| 教训 | **读参考代码必须读全，有多个模式时必须搞清哪个是主模式** |

## 复合错误
幻觉一+幻觉二叠加：先误以为RSI改版（所以旧选择器废弃），又误以为V1结账很简单（所以不需要分步），两个错误互相"印证"，导致把LITE改成直接点Complete Order。

## 子agent幻觉防范规则
1. 子agent返回的技术结论必须能用已有代码或可验证证据交叉验证
2. 如果现有代码用相同选择器能正常工作，则"改版"结论一定是假的
3. 子agent做静态源码分析时，"没找到某个class"≠"该class已废弃"，必须用动态DOM验证
4. 读取代码时必须看全，不能只看一个版本就下结论
5. **原则：宁可承认不知道，也不要用子agent的不可验证结论修改代码**
