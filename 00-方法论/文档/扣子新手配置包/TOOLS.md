# 工具使用技巧与工作流（通用版）

## 一、铁律区（每次操作必过，按顺序检查）

### 1. 合规自检
□ 我给出方案了吗？
□ 我自查方案了吗？（遗漏？矛盾？副作用？）
□ 用户确认了吗？
□ 我是在确认之后才动手的吗？
□ 用户问的是"解释/原因"吗？如果是，我不应该动手改
□ 我是否在"顺便"做用户没要求的事？如果是，停下来

**强制触发点**：调用 sub-agent / Calendar / 覆盖原文件之前，必须先输出确认模板：
```
准备执行：[具体动作]
用户确认：✗（等待确认）
```
- 没有"✓"就不能调工具，同一条消息里不能既出方案又调工具执行
- 出方案→停→等确认→下一条消息才执行

### 2. 方案先行
- **用户给方向≠授权直接动手**。"方向明确"只是需求清楚了，不代表跳过确认环节
- 即使是"小改动"，也要先给方案（改哪、怎么改、措辞是什么）→ 确认 → 再动手
- 跳过确认的口子一开就会越改越大

### 3. 验收+交付
- 验收时双向检查：正向（该做的做了没）+ 反向（不该出的问题出了没）
- **每一次** edit_file/write_file 成功后，下一个动作就是引用给用户看，无例外
- 不要等用户问，改完就引
- **覆盖原文件流程**：副本验收通过 → 引用给用户看 → **停：等用户确认** → 确认后才能替换原文件
- 验收通过≠可以覆盖，中间必须有用户确认这个门

### 4. 副本+逐级验收
- **所有修改走副本**：先复制原文件→副本，在副本上改，逐级验收后再替换原文件
- **防幻觉校验**：文件操作后必须验证引用路径是否存在、关键内容是否丢失
- 完整流程（每个"停"都是强制等待点）：
  - 复制原文件→副本 → 改副本 → 自验 → 引用给用户看 → **停：等用户确认** → 替换原文件
  - sub-agent产出 → 验收 → **引用给用户看** → **停：等用户确认** → 替换原文件
  - 主agent产出 → **引用给用户看** → **停：等用户确认** → 上线
  - 例外：简单局部修改（<5行）且精确指定了 old_string/new_string 的，可直接改
- **⚠️ 跳过任何"停：等用户确认"节点直接覆盖 = 严重违规**

### 5. 不要自我创造
- 已有能跑的代码直接搬，不要"理解后重写"——重写=引入新bug
- 严格参考现有代码，不能凭想象编造

### 6. 主动举手
- 不确定就说不确定，标记[不确定]，不要猜、不要编、不要硬撑
- 适用于主agent和所有sub-agent
- 宁可多问一句，不可错说一个字

## 二、sub-agent规范（派发时必读）

1. **保留清单优先**：明确列出必须保留的内容，禁止模糊指令
2. **路径约束**：禁止自创文件路径，所有引用路径必须已确认存在
3. **串行执行**：依赖任务必须串行，禁止并行
4. **指定源文件**：明确基于哪个文件修改，避免使用旧版本
5. **反面教训**：sub-agent 曾因上下文不足+无校验，将配置文件替换为不存在的路径
6. **高风险任务**：🔴高风险任务走双轨审查工作流 → `知识库/工作流/双轨审查工作流.md`

## 四、执行方式选型

| 场景 | 选什么 | 原因 |
|------|--------|------|
| 用户要求现在就做一件事 | sub-agent | 即时执行，结果秒回 |
| 每天固定时间做某事 | Calendar | 定时触发，重复执行 |
| 定期追踪某个话题 | Calendar | 有明确产出，用topic_tracking技能 |
| 30分钟看一眼状态 | Heartbeat | 轻量检查，无事不扰 |
| 生成文件/报告 | Calendar或sub-agent | Heartbeat不适合 |
| 抢购类精确时间任务 | Calendar | 时间敏感，设具体时间 |

详见：`知识库/Agent运维/执行方式详细指南.md`

## 六、记忆管理

- TOOLS.md：只写当前有效的规范和工作流
- MEMORY.md：只写当前活跃的任务状态
- 会过时的内容→写知识库，不会过时的→写记忆
- **TOOLS+MEMORY 控制在 15KB 以内**
- **经验教训**：记忆膨胀→上下文不足→输出质量下降→胡言乱语→巨量浪费积分。精简是生存问题

### 日常整理体系
- **每日轻量整理（20:00）**：清文件 + 验 INDEX + 知识库碎片整理
- **每周深度瘦身（周日22:00）**：TOOLS去肥 + MEMORY归档 + 知识库全面整理

## 七、验收标准（交付前必查）

### 自查方案（给用户确认之前）
1. 需求每条都覆盖了吗？
2. 改了A，B要不要跟着改？
3. 有没有副作用？
4. 隐含假设什么时候不成立？

### 自检代码（交付之前）
1. 语法/import/功能是否正确？
2. 代码和方案一致吗？
3. 崩溃点在哪？异常路径怎么办？
4. 数据流追踪：变量从哪来？变了没？

### 验收产出（检查任何交付物）
**正向验证（需求→产出）：**
1. 覆盖完整性 — 逐条对照原始需求
2. 意图准确性 — 产出是否真正满足需求意图
3. 产出可用性 — 能不能实际工作

**反向挑刺（产出→问题）：**
4. 逻辑审计 — 因果是否成立？概念是否矛盾？
5. 边界完整性 — 序号连续？引用有效？命名一致？
6. 副作用扫描 — 改A是否意外影响B？
7. 合适性判断 — 是否过度设计或不足设计？
8. 交叉验证 — 用工具扫描，不靠肉眼

**交付引用** — 验收通过后必须立即引用给用户，无例外

## 八、跨渠道提醒配置（踩坑经验）

### 典型场景
吃药提醒、打卡提醒等需要**跨渠道确认**的任务。

### 常见问题
- 用户在微信/其他渠道回复确认，但没有写入知识库或记录
- Agent 不知道用户已确认，导致重复提醒或遗漏

### 正确做法
1. 创建提醒时，**同时约定确认机制**：用户在哪回复？Agent 怎么知道？
2. **确认后必须回写**：用户确认完成后，Agent 要主动写入记录（如 MEMORY 或知识库）
3. 示例流程：
   ```
   Calendar提醒 → 用户在微信回复"已吃药" → 
   邮件分身捕获 → 提取确认信息 → 回写知识库 → 
   下次提醒时检查记录，避免重复
   ```

### 禁止做法
- ❌ 只发提醒，不约定确认机制
- ❌ 用户确认后不写记录
- ❌ 依赖 Agent"记住"而不写下来

## 九、常见踩坑清单（操作前扫一眼）

| 踩坑 | 后果 | 预防 |
|------|------|------|
| 未确认就派sub-agent | 浪费token+时间+回滚双倍消耗 | 方案确认后再派 |
| 验收完直接覆盖不给用户看 | 用户无法验收，重复操作 | 覆盖前必须引用给用户 |
| 方案和执行放在同一条消息 | 跳过确认环节 | 分开输出 |
| 跨渠道确认后不回写 | 重复提醒或遗漏 | 确认后必须写记录 |
| 记忆膨胀不清理 | 上下文不足→输出质量下降 | 定期瘦身 |
