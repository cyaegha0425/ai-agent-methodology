# 场景工作流详解

> 用到才查，不占日常上下文。

## BUG修复
1. 用户反馈问题 → 只记录，不改
2. 所有问题凑齐 → 总结+给方案 → **自查方案** → 交给用户确认
3. 确认 → 一次性全改 → 自检代码 → 交付
4. 用户测试 → 有问题回到第1步

## 新功能/需求开发
1. 用户提需求 → 复述理解，确认没偏
2. 给实现方案 → **自查方案** → 交给用户确认
3. 确认 → 建项目目录+INDEX.md → 写代码 → 自检代码 → 交付
4. 用户测试

## 需求分析与调研
1. 用户提模糊需求/想法 → 先梳理，不要急于给方案
2. 明确三件事：目标是什么、边界在哪、约束条件
3. 给可行性分析 → 交给用户确认方向
4. 方向确认后 → 拆解为可执行任务 → 用户确认拆解 → 派sub-agent执行

## 信息整理与输出
明确目标和格式 → 执行搜索采集 → 整理为结构化内容 → 先给用户初步结果确认方向 → 完善后交付并写入知识库

## 文件生成类操作
1. 确认内容和格式要求
2. 给方案 → 确认 → 生成 → 验收质量 → 交付
3. 不满意 → 调整 → 重新交付

## 代码重构/优化
1. 用户提优化需求 → 明确优化目标
2. 给方案，**必须包含风险评估**
3. 用户确认 → 实施 → 自检代码 → 回归测试 → 交付
4. 用户测试

## 配置文件修改（TOOLS/MEMORY等自身调整）
1. 明确改什么、为什么改
2. 给方案 → **必须包含权重检查**：改的内容应该放在哪个区？放的位置是否影响其他条目的优先级？
3. 用户确认 → 走副本修改 → 自验 → 引用给用户看 → **停：等用户确认** → 覆盖原文件

## 知识库整理
1. **触发**：发现缺INDEX、分类不全、内容错位、垃圾文件等问题
2. **诊断**：INDEX vs 实际文件交叉比对，用脚本检查问题
3. **执行**：新增缺失分类 → 清理垃圾文件 → 更新INDEX
4. **验证**：检查INDEX引用的文件是否都存在、实际文件是否都在INDEX中被引用

## 知识库维护（定期）
1. **定期检查**：每月执行一次INDEX完整性检查
2. **权重调整**：铁律区→流程区→参考区，顺序影响优先级
3. **垃圾文件清理标准**：
   - 临时文件（`temp_*.py`、`debug_*.log`）
   - 重复的文档（保留最新版本）
   - 过期的临时脚本
