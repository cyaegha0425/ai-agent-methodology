# 信用点追踪规则

## 追踪方式
- 基准余额：由主人确认账户实际显示余额后更新
- 日常变动：邮件分身报的融船/消费在基准上累加/扣除
- 校准：下次主人确认实际余额时校准

## 注意
- 邮件分身逐笔累加数可能与实际不符（之前消费/部分退款等原因）
- 基准数字必须以主人说的实际余额为准，不能用累加数替代基准
- 每次主人告知实际余额后，更新MEMORY.md中的基准

## 邮件分身配置补充（来源：百科3.21详解）

### 信用点追踪规则（百科3.21完整配置示例）

当需要追踪某类金额变动时，配置示例：

```markdown
### X项目余额追踪
- 触发条件：邮件主题包含"X项目确认"且包含余额数字
- 操作：
  1. 用正则提取余额数字（如"余额：1234.56"）
  2. 读取 ./余额追踪.md 获取当前累计值
  3. 累加新值，更新文件
  4. 记录本次变动（日期/来源/变动额/余额）
- 优先级：100
```

**余额追踪文件格式**：

```markdown
# X项目余额追踪

## 累计记录
| 日期 | 来源 | 变动额 | 余额 |
|------|------|--------|------|
| 2024-01-15 | 邮件1 | +100 | 100 |
| 2024-01-16 | 邮件2 | +50 | 150 |

## 当前余额：150
```

### 融船确认触发条件（RSI项目实例）

- **触发条件**：邮件主题包含"Confirmation"或"确认"关键词
- **识别特征**：
  - 邮件内容包含商品名称（如"Kraken"、"890 Jump"）
  - 包含金额信息（格式如"$xxx.00 USD"）
  - 包含余额变动描述（"credits have been refunded"或"charged"）
- **操作**：
  1. 正则提取余额数字：`\$?([\d,]+\.?\d*)\s*(?:credit|CC|USD)`
  2. 判断是增加（退款/融船）还是减少（消费）
  3. 追加记录到余额追踪文件
  4. 更新当前余额

## 中间值追踪机制

维护三个中间值，每封邮件来了只更新对应项：

| 中间值 | 含义 | 更新时机 |
|--------|------|---------|
| 每日消费累计 | 当天花了多少 | 收到消费/扣款邮件时累加 |
| 每日回收累计 | 当天质押回收了多少 | 收到质押回收邮件时累加 |
| 账户总余额 | 上次确认余额 + 回收累计 - 消费累计 | 自动计算 |

**计算逻辑**：
```
账户总余额 = 用户确认的基准余额 + 每日回收累计 - 每日消费累计
```

**规则**：
- 每封邮件只取单笔金额，累加到对应中间值
- 不使用邮件分身自己统计的累计数字（那些是分身自己的记账，不是RSI账户数据）
- 中间值的逻辑必须跟用户确认后才能使用
- 用户确认实际余额时，校准基准余额，清零当日累计

## 核心理解

- 信用点 = 充值真金白银，是用户的钱
- Pledge（质押）= 信用点换来的回馈，钱以另一种形态存在
- 质押回收（melt）= Pledge退回为信用点，是**资金形态转换**，不是收入
- Credit ↔ Pledge 大部分情况可互转：Credit买Pledge，Pledge melt回Credit
- **24小时CD**：新Pledge有24小时冷却期，过CD后才能melt回Credit
- **Buy Back Pack**：melt后的Pledge不会消失，进入Buy Back Pack，之后可以用Credit或新的支付方式买回
- **Buy Back限制**：用Credit从Buy Back Pack买回，每季度只有2次机会，季度刷新不累加；超过2次只能用现金（新支付方式）回购
- **商店常驻Pledge**：常驻商店的船不需要走Buy Back流程，melt后随时可以重新从商店购买，不受季度2次限制
- **限时出售Pledge**：melt后只能从Buy Back Pack买回，消耗季度2次Credit额度；但这类船通常不限量且经常返场，所以影响不大
- 所以melt常驻船=完全可逆无限制；melt限时船=走Buy Back但风险可控（经常返场）
- **限时+限量Pledge（最稀缺）**：重大节日发售，北京时间0点起每4小时整点放量；分两种版本：
  - **Warbond版**：限现金购买，带折扣，不能用信用点
  - **Standalone版**：可用信用点购买，无折扣
- 限量船融了很难买回，melt前必须慎重
- 所以melt虽可逆，但Buy Back有季度次数限制，大规模melt前要考虑当季买回额度是否够用
- 真正的"消费"只有：信用点→Pledge（且大部分可逆，需等CD）
- 不要用"收入""赚了""理财产品"等表述描述质押回收，那只是内部循环
